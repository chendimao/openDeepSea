# 文件管理实现与数据流梳理

## 目标与边界

本次仅梳理现有文件/资源管理实现，不变更运行时代码。结论用于后续细化文件管理功能时确定最小影响范围，重点区分当前页面实际使用的上传文件能力与后端已经存在但前端尚未接入的统一资源资产能力。

## 页面入口与主要组件

- 全局文件库入口：`packages/frontend/src/main.tsx` 注册 `/files`，侧边栏 `packages/frontend/src/components/AppShell.tsx` 通过 `SidebarLink to="/files"` 进入。
- 项目文件库入口：`packages/frontend/src/main.tsx` 注册 `/projects/:projectId/files`，复用 `packages/frontend/src/pages/FilesPage.tsx`。
- 房间文件入口：`packages/frontend/src/pages/RoomPage.tsx` 的 `RoomFeatureTabs` 包含 `files` tab，激活后渲染 `packages/frontend/src/components/RoomFilesPanel.tsx`。
- 文件列表复用视图：`packages/frontend/src/components/ProjectFileView.tsx`，负责列表/卡片展示、缩略图和 action 渲染。
- 当前没有独立资源详情页路由；详情/预览以弹窗形式存在于 `FilesPage.tsx` 和 `RoomFilesPanel.tsx`。

## 现有前端数据流

`FilesPage.tsx` 当前调用 `api.listFiles({ projectId, roomId })` 读取 `/api/files`，支持前端本地搜索、项目筛选、房间筛选、上传、预览、下载和删除。

`RoomFilesPanel.tsx` 当前调用 `api.listFiles({ projectId, roomId })` 读取房间引用过的上传文件，支持本地搜索、预览、下载、定位最后引用消息和删除。

`packages/frontend/src/lib/api.ts` 目前只封装了上传文件接口：

- `listFiles(filters)` -> `GET /api/files?projectId=&roomId=`
- `listProjectFiles(projectId)` -> `GET /api/projects/:projectId/files`
- `uploadProjectFiles(projectId, files)` -> `POST /api/projects/:projectId/files`
- `deleteProjectFile(fileId)` -> `DELETE /api/files/:fileId`

前端 `packages/frontend/src/lib/types.ts` 目前只有 `ProjectFile` 类型，尚未定义 `ResourceAsset` / `ResourceAssetListItem` 类型，也尚未封装或消费 `/resource-assets` 系列接口。

## 后端上传文件接口与结构

上传文件的主表是 `files`，引用关系表是 `message_file_refs`。

`files` 字段：

- `id`
- `project_id`
- `original_name`
- `stored_name`
- `mime_type`
- `size`
- `url`
- `storage_path`
- `uploaded_by_id`
- `uploaded_by_name`
- `created_at`
- `deleted_at`

`message_file_refs` 字段：

- `id`
- `project_id`
- `room_id`
- `message_id`
- `file_id`
- `created_at`

`fileRepo.list()` 返回 `ProjectFileWithRefs`，在 `files` 基础上附加：

- `reference_count`
- `last_referenced_at`
- `last_referenced_message_id`
- `last_referenced_room_id`
- `last_referenced_room_name`

上传文件接口：

- `GET /api/files`：可选 `projectId`、`roomId` 查询参数；会校验项目存在、房间存在、房间归属项目；返回 `ProjectFileWithRefs[]`。
- `GET /api/projects/:projectId/files`：返回项目下 `ProjectFileWithRefs[]`。
- `POST /api/projects/:projectId/files`：multipart 表单字段 `files`，可选 `uploaded_by_id`、`uploaded_by_name`；返回 `ProjectFile[]`。
- `DELETE /api/files/:fileId`：软删除 `files` 记录，标记消息附件删除，并尝试删除上传目录内物理文件。

下载/预览没有单独 API。上传文件通过 `url` 字段指向静态文件路径 `/uploads/files/:projectId/:storedName`；后端在 `server.ts` 中用 `express.static` 暴露 `projectFileUploadRoute`，并设置 `Content-Disposition: attachment` 和 `X-Content-Type-Options: nosniff`。前端预览目前直接使用该 `url` 作为 `img` 或 `iframe` 的 `src`，下载使用 `<a href={url} download={original_name}>`。

## 后端统一资源资产接口与结构

后端已经存在 `resource_assets` 表、`resourceAssetRepo` 和路由，当前支持两类资源：

- `uploaded_file`
- `agent_document`

分组类型：

- `uploaded_files`
- `agent_documents`

`ResourceAsset` 字段：

- `id`
- `project_id`
- `asset_type`
- `group_key`
- `title`
- `content`
- `mime_type`
- `size`
- `url`
- `file_id`
- `source_message_id`
- `source_room_id`
- `source_agent_id`
- `source_task_id`
- `metadata`
- `created_at`
- `updated_at`
- `deleted_at`

统一资源接口：

- `GET /api/projects/:projectId/resource-assets`：可选 `assetType`、`groupKey`；返回 `ResourceAssetListItem[]`，列表项不包含 `content`。
- `POST /api/projects/:projectId/resource-assets`：创建资源资产，当前输入校验字段包括 `asset_type`、`group_key`、`title`、`content`、`mime_type`、`size`、`url`、`file_id`、`source_message_id`、`source_room_id`、`source_agent_id`、`source_task_id`、`metadata`。
- `GET /api/resource-assets/:assetId`：返回资源详情，包含 `content`。
- `DELETE /api/resource-assets/:assetId`：只允许删除非 `uploaded_file` 资源；`uploaded_file` 走 `/api/files/:fileId`。

`resourceAssetRepo.list()` 会把真实 `resource_assets` 表中的智能体文档与 `files` 表中的上传文件合并排序。上传文件不会复制写入 `resource_assets` 表，而是以虚拟资源映射返回：

- `id = "file:" || files.id`
- `asset_type = "uploaded_file"`
- `group_key = "uploaded_files"`
- `title = files.original_name`
- `content = null`
- `mime_type = files.mime_type`
- `size = files.size`
- `url = files.url`
- `file_id = files.id`
- `source_message_id = null`
- `source_room_id = null`
- `source_agent_id = files.uploaded_by_id`
- `source_task_id = null`
- `metadata = null`

注意：当前虚拟上传文件资源把 `uploaded_by_id` 映射到 `source_agent_id`，但没有提供 `uploaded_by_name` 或明确的 `source_type` 字段；后续 UI 若要展示“用户上传/智能体生成”的语义，应避免把 `source_agent_id` 直接解释为智能体来源。

## 字段能力结论

- 已有资源类型字段：后端有 `asset_type`，取值 `uploaded_file | agent_document`；前端上传文件页面尚未消费。
- 已有资源分组字段：后端有 `group_key`，取值 `uploaded_files | agent_documents`；前端尚未消费。
- 已有来源字段：后端 `resource_assets` 有 `source_message_id`、`source_room_id`、`source_agent_id`、`source_task_id`、`metadata`；上传文件表有 `uploaded_by_id`、`uploaded_by_name`；但统一资源列表对上传文件没有 `uploaded_by_name`。
- 已有创建者/上传者字段：上传文件有 `uploaded_by_id`、`uploaded_by_name`；智能体文档目前只通过 `source_agent_id` 表达来源，缺少独立 `created_by_name`。
- 已有内容字段：`agent_document` 可在 `content` 中存 Markdown；资源列表不返回 `content`，详情接口返回；上传文件虚拟资源 `content = null`。
- 已有文件 URL 字段：上传文件和资源资产均有 `url`；智能体文档可以为空。
- 已有详情能力：`GET /api/resource-assets/:assetId` 可取统一资源详情；上传文件旧页面没有单独详情接口，直接用列表数据弹窗预览。
- 已有预览能力：上传文件前端根据 `mime_type` 直接预览图片、PDF、文本；智能体文档尚无前端 Markdown 详情/预览入口。
- 已有下载能力：上传文件通过静态 `url` 下载；`agent_document` 当前无导出/下载实现。
- 基础搜索现状：文件页面只有前端本地过滤；统一资源接口没有 `q` 参数或后端搜索。

## 回归风险

- 当前 `/files` 页面仍只走旧文件接口。如果直接替换为 `/resource-assets`，需要处理 `title` 与 `original_name`、`asset_type` 与文件 MIME 操作的差异。
- `uploaded_file` 删除语义不能走 `DELETE /resource-assets/:assetId`，否则会返回 404；仍需调用 `/files/:fileId`。
- 静态上传文件响应默认 `Content-Disposition: attachment`，浏览器内嵌预览能力可能受影响；后续若强化预览，需要专门评估安全头和预览路由。
- `agent_document.content` 是 Markdown 内容，前端渲染必须继续使用受控 Markdown 渲染路径，避免引入不受信 HTML。
- 统一资源列表当前没有后端搜索，前端若只做本地过滤，会漏掉分页/大量数据场景；虽然当前接口未分页，但后续扩展需统一约定。

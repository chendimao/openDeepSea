# 智能体文档自动归档产品规则说明

## 目标

自动归档只识别智能体产出的明显 Markdown 文档资产，并将其作为 `agent_document` 进入项目资源库。它不是聊天记录归档、消息备份或日志收集机制。

核心原则是保守自动归档：宁可少归档，也不要把短回复、普通聊天、日志、错误堆栈、纯代码或临时进度塞进文件管理。规则无法稳定判断的内容不自动入库，只提供用户手动保存入口。

## 范围

默认纳入自动归档候选：

- 方案文档、设计文档、需求文档、PRD。
- 实施计划、任务拆解、验收报告、复盘总结。
- 结构化 Markdown 文档，且具备明确标题、章节、列表、表格、任务清单或文档意图。

默认不纳入自动归档：

- 用户消息、系统消息、工具日志、运行状态事件。
- 短回复、普通聊天说明、进度播报。
- 纯日志、命令输出、错误堆栈。
- 纯代码块或代码占比过高的回复。
- 普通问答、解释、翻译。
- 用户上传文件。
- 已经归档过的同源消息。

用户上传文件继续作为 `uploaded_file` 管理；智能体生成文档作为 `agent_document` 管理。两者可以在资源库合并展示，但资源类型、分组、来源字段和删除语义必须保持区分。

## 触发条件

自动归档评估必须同时满足以下条件：

1. 消息已完整结束。
   - 只在智能体消息生成完成后评估。
   - 不在流式输出中途、partial delta、typing 状态、运行进度事件中归档。
2. 来源必须是智能体消息。
   - 只处理 `sender_type=agent`。
   - 不处理用户消息、系统消息、工具日志或状态事件。
3. 上下文必须完整。
   - 必须具备 `project_id`、`room_id`、`message_id`、`agent_id`。
   - 任一缺失时，不进入自动归档。
4. 内容必须通过文档资产判定。
   - 触发条件只决定是否允许评估。
   - 是否真正归档仍由硬排除和评分规则决定。
5. 同源消息不得重复归档。
   - 同一个 `source_message_id` 已存在有效 `agent_document` 时，不再新增。

## 硬排除

硬排除先于评分执行。命中任一条时，直接判定不归档，不再计算分数。

- 内容长度小于 500 字符：不归档。
- 代码块内容占比超过 70%：不归档。
- 日志、终端输出、错误堆栈特征明显：不归档。
- 没有 Markdown 结构且没有文档关键词：不归档。
- 短回复、普通聊天说明、运行状态、进度播报：不归档。
- 普通问答、解释、翻译：默认不自动归档。
- 来源不是智能体消息：不归档。
- 消息未完整结束：不归档。
- 缺少 `project_id`、`room_id`、`message_id` 或 `agent_id`：不归档。
- 同源消息已归档：不重复归档。

典型日志和堆栈特征包括 `npm ERR!`、`Traceback`、`Exception`、`at Function...`、`Process exited`、`stdout`、`stderr` 或大量时间戳日志。

## 评分规则

只有通过硬排除后，才进入评分。评分只判断内容是否足够像可复用文档资产，不判断内容质量。

| 评分项 | 分值 | 判定说明 |
| --- | ---: | --- |
| 有明确标题 | +2 | 包含 `#` / `##` 标题，或首行明显是文档标题 |
| 章节数量充足 | +2 | 有 2 个及以上 Markdown 小节 |
| 包含列表、表格或任务清单 | +1 | 有有序列表、无序列表、表格、checkbox task list |
| 包含文档关键词 | +2 | 如方案、设计、计划、需求、PRD、总结、报告、复盘、验收、实施计划、规则、范围 |
| 内容长度超过 1200 字符 | +1 | 长度只能作为弱信号，不能单独触发归档 |
| 智能体角色匹配文档产出 | +1 | 如产品经理、Planner、架构师、Reviewer、总结类智能体 |
| 用户请求明确要求文档化 | +3 | 如生成文档、写方案、输出计划、整理成需求、形成总结、生成 PRD、归档 |

结果阈值：

- `score >= 5`：自动归档为 `agent_document`。
- `score 3-4`：默认不自动归档，只建议用户手动保存。
- `score < 3`：不归档，也不提示。

评分必须保持保守：单个弱信号不能触发自动归档，自动归档应由结构、意图和上下文多信号共同成立。

## 结果元数据

自动归档成功后生成资源库记录：

- `asset_type`: `agent_document`
- `group_key`: `agent_documents`
- `mime_type`: `text/markdown`
- `content`: 智能体消息完整 Markdown 内容
- `title`: 文档标题
- `source_message_id`: 来源消息 ID
- `source_room_id`: 来源聊天室 ID
- `source_agent_id`: 来源智能体 ID
- `source_task_id`: 如有关联任务则记录，否则为空

必须可追溯字段：

- `project_id`
- `room_id`，映射到 `source_room_id`
- `message_id`，映射到 `source_message_id`
- `agent_id`，映射到 `source_agent_id`

标题生成优先级：

1. 第一个 Markdown 一级标题。
2. 第一个 Markdown 二级标题。
3. 正文前 30 个有效字符摘要。
4. 兜底标题：`智能体文档`。

标题只用于资源库展示和检索，不改变原始消息内容。

## uploaded_file 与 agent_document 边界

`uploaded_file` 用于用户上传文件：

- `asset_type`: `uploaded_file`
- `group_key`: `uploaded_files`
- 依赖文件记录、`file_id`、`url`、`mime_type`、上传者字段。
- 不参与智能体文档自动归档评分。

`agent_document` 用于智能体生成文档：

- `asset_type`: `agent_document`
- `group_key`: `agent_documents`
- 依赖完整 Markdown 内容和来源消息字段。
- 可由自动归档或用户手动保存产生。

删除语义必须区分：

- 移除 `agent_document` 只移除归档文档资源，不删除原始聊天消息。
- 删除或移除 `uploaded_file` 不应受智能体文档取消归档影响。

## 手动保存与取消

### 手动保存

未自动归档但用户认为有价值的智能体消息，可以手动保存为 `agent_document`。

适用条件：

- 消息来源是智能体。
- 具备 `project_id`、`room_id`、`message_id`、`agent_id`。
- 尚未存在同源 `agent_document`。
- 内容不一定达到自动归档阈值，但用户主动选择保存。

保存后建议元数据：

- `metadata.manual_saved = true`
- `metadata.auto_archived = false`

`score 3-4` 的灰区内容默认只显示轻量保存入口，例如消息操作区的 `保存为文档`，不弹窗、不强提醒、不自动打开资源库。

### 取消归档

已归档的 `agent_document` 可以取消归档或从资源库移除。

取消语义：

- 删除或软删除 `agent_document` 资源记录。
- 不删除原始聊天消息。
- 不修改原始消息内容。
- 不影响用户上传文件。
- 资源库中不再展示该文档。

建议文案使用 `从资源库移除`，避免使用 `删除消息`。确认提示应明确：只会移除归档文档，不会删除原始聊天消息。

## 边界案例

| 类型 | 示例 | 判定 | 原因 |
| --- | --- | --- | --- |
| 正例 | 方案文档，含标题、多个章节、任务清单，用户要求写方案 | 自动归档 | 结构、关键词、用户意图多信号成立，分数通常 >= 5 |
| 正例 | 需求文档或 PRD，含背景、用户故事、验收点 | 自动归档 | 明确文档类型和结构，可复用 |
| 正例 | 总结报告，含结论、证据、后续动作 | 自动归档 | 明确总结/报告意图，具备章节 |
| 正例 | 实施计划，含任务拆分、风险、验证方式 | 自动归档 | 文档关键词和计划结构明确 |
| 反例 | `好的，我来处理。` | 不归档 | 内容长度小于 500 字符，属于短回复 |
| 反例 | 命令输出或 `npm ERR!` 日志 | 不归档 | 日志/终端输出不是文档资产 |
| 反例 | `Traceback`、`Exception` 错误堆栈 | 不归档 | 错误堆栈是过程证据，不进入资源库 |
| 反例 | 几乎全是 ```ts 代码块 | 不归档 | 代码块占比过高 |
| 反例 | 普通问答解释，没有结构也没有文档关键词 | 不归档 | 文档资产信号弱 |
| 模糊例 | 有标题和少量列表，但缺少明确文档关键词和用户文档化请求 | 建议手动保存 | score 3-4，默认不自动归档 |
| 模糊例 | 有规则/范围关键词但结构较弱 | 建议手动保存 | 可能有价值，但证据不足 |
| 边界 | 流式输出中途已出现完整标题 | 不归档 | 消息未完整结束 |
| 边界 | 用户发送一篇完整 PRD | 不自动归档 | 来源不是智能体消息 |
| 边界 | 同一消息已归档后再次评估 | 不重复归档 | 同源 `agent_document` 已存在 |

## 验收标准

后续实现和测试至少覆盖：

- 只在智能体消息完整结束后触发，不在流式输出中途归档。
- 只处理智能体消息，不对用户消息自动归档。
- 缺少 `project_id`、`room_id`、`message_id`、`agent_id` 任一字段时不归档。
- 硬排除先于评分执行。
- 短回复、日志输出、错误堆栈、纯代码块、普通问答不自动归档。
- `score >= 5` 自动归档。
- `score 3-4` 只建议手动保存，不自动写库。
- `score < 3` 不归档。
- 同一 `source_message_id` 不重复生成 `agent_document`。
- 用户可手动保存未自动归档的智能体消息为 `agent_document`。
- 用户可取消归档，取消只移除资源记录，不删除原始聊天消息。
- `uploaded_file/uploaded_files` 与 `agent_document/agent_documents` 类型和分组保持区分。
- 用户上传文件不参与 `agent_document` 自动归档判定。

## 后续实现输入

后续开发任务可按以下顺序拆分：

1. 在消息完整结束事件后调用文档资产判定。
2. 查询同源 `agent_document`，避免重复归档。
3. 通过硬排除和评分后，创建 `agent_document` 资源。
4. 为 `score 3-4` 暴露手动保存入口。
5. 提供取消归档能力，并保证不删除原始消息和用户上传文件。
6. 在资源库展示中区分用户上传文件和智能体文档。
7. 补充后端单元测试、路由测试、前端交互测试和端到端冒烟验证。

本规则说明是产品规则收口文档，不表示自动归档功能已经完整实现。

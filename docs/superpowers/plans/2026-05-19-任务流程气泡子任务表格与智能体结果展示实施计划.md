# 任务流程气泡子任务表格与智能体结果展示实施计划

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 将任务启动后的 Markdown 计划归一为固定 JSON，并在群聊气泡中展示可动态更新的子任务表格、进度和按智能体分组的执行结果。

**Architecture:** Markdown 计划只做人可读稿，点击开始 workflow 后由后端生成唯一的 `workflow_plan_json` 作为执行源。workflow runtime 根据 JSON 拆分子任务、计算并行/串行、分配智能体并回写状态；前端气泡只订阅状态变化，渲染任务表格、进度条和按智能体分组的 Tabs 结果区。这样可以把“计划文档”“调度逻辑”“展示层”彻底分离。

**Tech Stack:** Node.js、TypeScript、SQLite/better-sqlite3、Express、React 18、Vite、node:test、现有 workflow runtime、现有聊天气泡组件。

## 核心数据结构

### WorkflowPlanJson

```ts
interface WorkflowPlanJson {
  workflow_name: string;
  source_message_id: string;
  goal: string;
  summary: string;
  tasks: WorkflowPlanTaskJson[];
}

interface WorkflowPlanTaskJson {
  id: string;
  title: string;
  description: string;
  role: 'planner' | 'executor' | 'reviewer' | 'acceptor';
  agent_id: string | null;
  mode: 'parallel' | 'serial';
  depends_on: string[];
  status: 'pending' | 'running' | 'completed' | 'blocked' | 'failed';
  progress: number;
  result_refs: string[];
}
```

### 推导规则

1. Markdown 计划中的自然语言任务只作为来源材料。
2. JSON 里的 `depends_on` 决定串行约束。
3. 没有依赖且同阶段可执行的任务标记为 `parallel`。
4. 有依赖的任务标记为 `serial`。
5. `agent_id` 由协调器结合角色、权限、房间内智能体能力决定。
6. `result_refs` 只保存可追踪引用，不保存整段冗余大文本。

2. JSON 里能稳定表达子任务、依赖、并行 / 串行和智能体分配。

4. 气泡下方能按智能体切换执行结果。
5. workflow 的执行顺序由 JSON 和 state 驱动，不再依赖自然语言猜测。
6. 相关回归测试和 `npm run build` 通过。

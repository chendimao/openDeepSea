# 群聊协作与正式工作流分流 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 让 planner 在执行型请求后先让用户选择“轻量群聊协作”或“正式 workflow”，并让轻量群聊协作按 executor -> review/test -> acceptance 顺序运行。

**Architecture:** 保留现有 LangGraph workflow 作为正式交付流；新增 collaboration decision 与 lightweight collaboration runner 作为普通群聊和正式 workflow 之间的轻量编排层。`fallback_route` 不再直接把 planner 回复中的 @ 名单并发启动，而是先生成决策卡片，由用户选择后进入 staged collaboration 或 promote-to-workflow。

**Tech Stack:** TypeScript, Express, SQLite, React 18, TanStack Query, WebSocket, node:test。

## 风险与回滚

- 如果 decision JSON 不稳定，先保留 planner 原始文本并只展示“升级为任务/继续群聊”两个按钮。
- 如果 collaboration 表改动过大，可第一阶段只使用 message metadata + agent_run 关联字段。
- 如果前端时间线复杂，先只展示决策卡片和系统消息，阶段 UI 延后。
- 如果正式 workflow promote 影响现有任务创建，保留 slash command 原路径并把新接口设为可选入口。

## 最终验收

第一阶段验收：

- [ ] 普通问答不误触发协作或 workflow。
- [ ] 执行型请求先出现模式选择。
- [ ] 轻量协作不再让 executor/reviewer/tester/acceptor 同时开始。
- [ ] 正式 workflow 选择后能完整创建 task 和 workflow_run。
- [ ] 原 `/task`、`/start-task`、显式 @ 行为不回归。
- [ ] `npm run build` 通过。

第二阶段验收：

- [ ] collaboration run/step 可持久化查询。
- [ ] 前端能展示轻量协作阶段时间线。
- [ ] 支持阶段级失败、重试和取消。
- [ ] planner summary 能汇总轻量协作结果。
- [ ] 轻量协作记忆不会冒充正式 workflow 完成记忆。

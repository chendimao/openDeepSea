# 智能体运行边界一等化 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 将当前以内置提示词为主的智能体差异升级为可落库、可调度、可执行约束的 Agent Runtime Profile。

**Architecture:** 保留现有 `Agent` 作为全局模板、`RoomAgent` 作为聊天室 profile，不引入独立 OpenClaw workspace。新增运行边界字段与 `agent-runtime` 解析层，让模型、工具权限、工作区、记忆范围和 ACP 权限成为一等配置；后续 dispatcher、workflow resolver 和前端配置面板都只消费统一解析结果。

**Tech Stack:** TypeScript、Node.js、Express、better-sqlite3、React 18、Vite、现有 ACP adapters（Codex / Claude Code / OpenCode）、node:test。

## 设计约束

- 不引入新外部依赖。
- 不删除 `default_runtime: 'openclaw'` 兼容值，但新逻辑不恢复 OpenClaw Gateway。
- 第一阶段不做独立文件 workspace（`AGENTS.md` / `SOUL.md` 目录），只做数据库 profile 与 prompt/runtime 注入。
- 默认保持现有可用性：既有用户智能体没有配置时继续按当前 ACP 配置运行。
- 内置智能体默认边界必须变硬：
  - Planner：不写代码，默认 `read-only`。
  - Backend Executor：默认可写 `packages/backend`。
  - Frontend Executor：默认可写 `packages/frontend`。
  - Reviewer：默认 `read-only`。
  - Acceptor：默认 `read-only`。

  if (!value) return fallback;
  try {
    const parsed = JSON.parse(value) as unknown;
    return parsed && typeof parsed === 'object' ? parsed as T : fallback;
  } catch {
    return fallback;

- [x] **Step 3: 实现解析模块**

创建 `packages/backend/src/agent-runtime.ts`，导出：

```ts
export function resolveAgentRuntimeProfile(input: {
  agent: RoomAgent;
  projectPath: string;
  imagePaths?: string[];
}): ResolvedAgentRuntimeProfile

export function buildAgentRuntimeContextPrompt(profile: ResolvedAgentRuntimeProfile): string
```

关键规则：

- `runtimeBackend` 默认从 `agent.runtime_backend ?? agent.default_runtime` 映射，旧值 `openclaw` 视为 `none` 或 ACP 不可用。
- `acpBackend` 必须来自 `agent.acp_backend`。
- `workspace_policy.write` 解析为项目内绝对路径。
- 空 write 时 `acpPermissionMode` 强制降级为 `read-only`。
- `imagePaths` 的目录只加入 readable，不加入 writable。
- 不允许 `..`、绝对路径逃逸项目根；发现非法路径时忽略并记录 warning 字符串。

## 并行执行建议

适合并行：

- Task 1 完成后，Task 2 和 Task 3 可以串行优先；Task 6 的前端 UI 可在后端 API schema 稳定后并行。
- Task 5 依赖 Task 3 的 runtime policy 类型，但不依赖前端 UI。

不适合并行：

- `types.ts`、`db.ts`、`repos/rooms.ts`、`routes.ts` 属于 shared contract，必须串行或统一收口。
- `dispatcher.ts` 的权限修复必须在 `agent-runtime.ts` 稳定后进行。

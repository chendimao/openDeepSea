# 工作流自由增删改查 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 将工作流从单一默认编排扩展为支持系统级、项目级、群聊级自由增删改查和默认设置的管理中心。

**Architecture:** 后端复用 `workflow_definitions` 与 `settings` 现有表，补齐 repo/API 的复制、派生草稿、归档、删除和可见性校验。前端把 `/workflow` 升级为管理中心，并把 `WorkflowBuilderDialog` 泛化为可保存草稿、发布、按 scope 设默认的 builder。运行记录继续依赖现有 definition snapshot，不改变 workflow runtime。

**Tech Stack:** Node.js、Express、better-sqlite3、node:test、React 18、TypeScript、Vite、TanStack Query、@xyflow/react、Tailwind、lucide-react。

- [x] **Step 3: 实现校验辅助**

在 `settings.ts` 增加私有函数：

```ts
function validateDefaultWorkflowDefinition(
  scope: SettingsScope,
  scopeId: string,
  workflowDefinitionId: string | null,
): string | null {
  const id = normalizedOptionalString(workflowDefinitionId);
  if (!id) return null;
  const definition = workflowDefinitionRepo.get(id);
  if (!definition) throw new Error('default workflow definition not found');
  if (definition.status !== 'published') throw new Error('default workflow definition must be published');
  if (definition.status === 'archived') throw new Error('default workflow definition is archived');

  if (scope === 'system' && !workflowDefinitionRepo.isVisibleForSystem(id)) {
    throw new Error('default workflow definition is not visible for system settings');
  }
  if (scope === 'project' && !workflowDefinitionRepo.isVisibleForProject(id, scopeId)) {
    throw new Error('default workflow definition is not visible for project settings');
  }
  if (scope === 'room' && !workflowDefinitionRepo.isVisibleForRoom(id, scopeId)) {
    throw new Error('default workflow definition is not visible for room settings');
  }
  return id;
}
```

注意：`definition.status !== 'published'` 已覆盖 archived，因为类型为 union；错误文案保持测试能匹配即可。

## 风险与注意事项

1. 当前前端没有项目/群聊全局选择器。管理中心第一版若要从空白新建 project/room workflow，需要读取项目/房间列表；否则应把 project/room 新建入口放在对应设置页，避免做半成品选择器。
2. `settingsRepo` 校验会改变现有行为，可能暴露旧数据中已保存非法 id 的问题。实现时只在 update 时硬拦，resolve 时应尽量兼容旧数据并显示当前 id。
3. `archived` 是现有类型但当前 repo 未充分使用。列表和可见性函数必须统一排除 archived，否则设置下拉会出现已归档项。
4. 不要修改 workflow runtime 的图执行语义；本任务只管理 definition 生命周期。
5. 不要提交当前工作区已有的上传相关未提交改动。

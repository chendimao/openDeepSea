# 任务画像与全局智能体路由实施计划

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** 让 workflow 分配先识别任务类型和能力需求，再从当前群聊与全局智能体池中选择合适智能体，必要时自动拉入群聊，避免前端/UI/PPT/文档等任务被错误分配给不匹配的 executor。

**Architecture:** 新增一个纯函数任务画像模块，负责把 `ParsedPlanTask` 归类为 domain、artifact、workflow template 和 required capabilities。Dispatch 阶段先用当前群聊智能体匹配；若无合适智能体，查询全局智能体池并把匹配者加入群聊；若已有群聊 executor 但 domain 明显不匹配，不能静默使用，应优先拉入全局匹配智能体或阻塞。Planner 提示词同步要求输出更准确的 scope 和 role，但最终正确性由后端规则门禁保证。

**Tech Stack:** Node.js、TypeScript、better-sqlite3、本地 `node --import tsx --test` 测试、现有 workflow graph runtime。

---

## 文件结构

- 新建 `packages/backend/src/workflows/task-profile.ts`
  - 纯函数模块，负责根据任务标题、描述、scope、验收点推断任务画像。
  - 输出 `taskType`、`domains`、`artifactTypes`、`workflowTemplate`、`requiredCapabilities`、`recommendedTemplateId`、`confidence`、`reasons`。
  - 不访问数据库，不产生副作用，方便单测覆盖。
- 新建 `packages/backend/src/workflows/task-profile.test.ts`
  - 覆盖 frontend UI、backend API、fullstack、PPT、文档、bugfix 等分类。
- 修改 `packages/backend/src/workflows/graph/tools.ts`
  - 给 `GraphTools` 增加读取全局智能体和拉入全局智能体的能力。
  - 复用 `agentRepo.list()` 与 `roomAgentRepo.addFromGlobalAgent()`。
- 修改 `packages/backend/src/workflows/graph/nodes.ts`
  - Dispatch 分配逻辑接入任务画像。
  - 当前群聊无合适智能体时，从全局智能体池查找匹配者并拉入群聊。
  - 当前群聊存在 executor 但与任务 domain 明显不匹配时，不允许静默分配。
  - 分配 artifact metadata 写入 routing profile 和 assignment reason，方便 UI/日志追踪。
- 修改 `packages/backend/src/workflows/graph/runtime.test.ts`
  - 增加 workflow 级回归测试，覆盖“全局前端智能体被拉入群聊并分配”“PPT 不分给前后端 executor”“无匹配全局智能体时阻塞或用内置模板”。
- 修改 `packages/backend/src/workflows/prompts.ts`
  - 规划提示词要求 planner 给出 scopeRead/scopeWrite、任务类型和验收阶段，不把复杂需求压成单个 executor。
  - 注意：提示词只提升 planner 质量，不作为唯一门禁。

## 设计约束

- 不把方案写死成“所有任务都是前后端全流程”。必须先分类，再选 workflow template。
- 全局智能体池优先使用用户已有全局智能体；只有找不到合适全局智能体时，才考虑内置模板补位。
- 当前群聊中已有合适智能体时不重复拉入。
- 只自动拉入可执行且能力匹配的全局智能体：`default_acp_backend` 存在，权限和 `default_workspace_policy.write` 能覆盖写范围。
- `scopeWrite=[]` 不再代表“任何 executor 都可做”。若任务文本能推断出 domain，应按 domain 和能力匹配；例如 UI/侧边栏/组件应匹配 frontend。
- reviewer/acceptor 仍可以 read-only，但需要 `workflow_role` 匹配。
- PPT、文档、数据分析类任务不能被 frontend/backend executor 静默接走；应匹配 PowerPoint/technical-writer/data-analyst 等能力，缺失时阻塞并说明缺什么。

## 验收标准

- “最近群聊指的是最近访问过的群聊”这类前端 UI 任务，即使 planner 输出 `scopeWrite=[]`，也不会被后端 executor 静默接走。
- 如果全局智能体池已有合适前端智能体，dispatch 会自动拉入群聊并分配。
- 如果任务是 PPT/文档/数据分析，不能被 frontend/backend executor 误接；缺少合适全局智能体时应阻塞并说明原因。
- 有明确 `scopeWrite` 的现有前后端分配测试继续通过。
- 分配 artifact 或 graph_state 能追踪任务画像、匹配原因和拉入来源。

## 风险与取舍

- 关键词分类不是完美语义理解，但作为后端门禁比完全依赖 planner 自然语言更稳。
- 第一版不新增数据库表，只用现有 agents/room_agents 数据；后续可以把 routing decision 写成独立 artifact 或 metric。
- 当前内置智能体没有专门 PPT 制作者；本计划只防止误派给开发智能体，并允许后续新增 `presentation-designer` 模板。
- 如果用户手动把某个后端智能体配置成 frontend capability 且 workspace_policy 覆盖 frontend，则它可以被选中；这是配置表达的能力，不应被名字硬拦。

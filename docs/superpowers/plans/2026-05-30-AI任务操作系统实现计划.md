# AI Task Operating System Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [x]`) syntax for tracking.

**Goal:** 按 `docs/superpowers/specs/2026-05-30-ai-task-os-总体架构-design.md` 把 openclaw-room 渐进演进为以 Task 为聚合根的 AI Task Operating System。

**Architecture:** 当前分支按 M1 -> M2 -> M3 -> M4 推进，先建立可回放的 TaskEventLog 和 MessageLayer，再让消息路由、前端三栏和 TaskExecutor 接入同一套 Task 中心模型。M5 仅保留为未来评估，不在本轮实现。

**Tech Stack:** SQLite、Express、WebSocket、TypeScript、React 18、Vite、现有 ACP/LangGraph runtime。

## 本轮范围说明

- 已完成 M1-M4 的端到端闭环，而非仅基础切片：群聊消息可以自动建任务并激活任务，消息归属会实时同步到前端，任务事件流可回放，三栏 UI 能完成任务查看与状态流转，task 级 agent/planner 派发会进入 TaskExecutor 隔离会话。
- M2 当前支持显式任务引用在已有激活任务不同时返回 `switch_task`，记录 `message_routed` 事件并广播 `task:activated`，使前端激活任务与路由结果保持一致。
- M2 当前会拒绝把群聊消息路由到 `done` / `failed` 终态任务，即使消息显式包含 `@task:<id>` 或 `#task:<id>`，也会降级为 `ask_user` 并给出 activity 层提示，避免已完成/失败任务被新消息和执行上下文污染。
- M2 当前对 `create_task` 路由结果只创建任务并写入任务事件，不再把尚未带 `task_id` 的原始用户消息继续调度给默认 planner，避免自动建任务消息绕过 M4 的 task 级执行上下文。
- M2 当前对无明确 @ 智能体的 `ask_user` 路由结果只落库并提示 activity，不再继续触发默认 planner；若用户显式 @ 智能体，仍保留人工指定的协作入口。
- M2/M4 当前补强 dispatcher 回归：planner fallback 回复在包含明确任务化信号或结构化 JSON 时会写入 `task_readiness`，支持后续从 planner 建议提升为正式 workflow；普通聊天回复不会误标记；`/task` 自动建任务意图不再触发 fallback ACP 派发，避免自动建任务消息绕过 task 级执行上下文。
- M1 当前进一步补强了事件回放能力：`task_created` / `task_updated` / `task_status_changed` / `task_deleted` 事件 payload 可重建任务当前态，后端提供 `replayTaskEvents()`，`GET /rooms/:roomId/task-events?taskId=<id>&replay=1` 会返回事件流和 replay 投影，前端 API 类型已同步。
- M3 当前提供三栏任务操作面、任务选择、任务时间线、Plan/Diff/Logs 分层视图、事件层过滤和任务属性编辑。
- M3 当前补强 Diff/Logs 读层：任务详情会把 `diff_detected` payload 投影为文件路径、增删行统计和 diff 片段，把 `runtime_event` payload 投影为命令/工具摘要和输出片段，避免运行层与变更层只显示通用事件标题。
- M3 当前补强 Router 可解释读层：任务详情会优先展示 `message_routed` / `message_route_uncertain` payload 中的 `route_reason` 与 `route_confidence`，避免只显示 `append_to_task` / `create_task` 等内部动作名。
- Chat 主列当前只投影 `chat` 层与旧数据缺省层消息；`activity`/`timeline`/`runtime`/`diff` 通过任务侧栏、活动汇总和任务详情面板查看，避免系统事件与运行日志污染群聊主流。
- M4 当前完成 `(task, agent)` ACP session 隔离，并在 task session 恢复时按 active task 构建 handoff 摘要，避免混入同房间其他 task 的历史。
- M4 当前也覆盖 planner dispatch：当 planner 决策来源消息已经归属 task 时，派发给执行智能体和后续 planner 汇报都会携带同一 `task_id`，写入 task-scoped agent run 与 TaskExecutor session，而不会污染 `room_agents.acp_session_id`。
- M4 当前补强执行会话可观测性：后端提供 `GET /tasks/:id/executors` 返回任务级 TaskExecutor session，前端任务详情面板展示执行智能体、状态、ACP backend、短 session id 与 handoff pending 状态，并在 task-scoped agent run 更新时刷新该面板。
- 浏览器闭环验证已覆盖：在真实群聊中发送 `新建任务：浏览器闭环测试任务`，任务自动创建并进入左侧任务列表，点击任务打开右侧详情，使用 UI 将状态从 `todo -> in_progress -> review -> done`，活动汇总与任务详情同步更新；后端事件流产生连续 `seq=1..5` 的 `task_created`、`message_routed` 和三条 `task_status_changed` 事件。
- 2026-05-30 追加浏览器验证覆盖：在独立临时后端 `7331` / 前端 `5181` 真实浏览器环境中发送 `新建任务：M1-M4 浏览器完整闭环...`，验证任务自动创建、右侧详情激活、通过详情属性编辑完成 UI 状态流转 `todo -> in_progress -> review -> done`、事件流连续 `seq=1..6`、回放状态 `done`，并验证 `GET /tasks/:id/executors` 与任务详情执行会话区显示 `规划师 / 运行中 / codex / session`。
- 2026-05-30 最终代码审查：发现并修复 planner 普通回复被过宽启发式标记为 `task_readiness` 的风险，补充普通聊天负例；浏览器测试还暴露多行新建任务标题解析边界，已补 `extractCreateTaskTitle()` 首行提取测试与实现。

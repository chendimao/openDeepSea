# 项目级文件库与群聊引用设计

## 背景

当前聊天附件通过 `POST /rooms/:roomId/messages` 直接上传到服务端目录，并把附件信息写入 `messages.metadata.attachments`。这种模型适合一次性消息附件，但不适合作为可管理、可复用的项目资产。

本次设计将文件提升为项目级资源：用户可以在文件页面主动上传文件，也可以在群聊消息编辑框中引用同项目文件库里的文件。

历史数据无需迁移，当前没有正式数据。

## 目标

1. 新增项目级文件库，文件主归属为项目。
2. 左侧“文件”菜单进入当前项目的文件页面。
3. 文件页面展示项目下所有未删除文件，支持上传、预览、下载、删除。
4. 群聊消息编辑框支持从项目文件库选择并引用文件。
5. 群聊内直接上传的文件也进入项目级文件库，并自动作为当前消息引用。
6. Agent 仍能收到引用文件的 `localPath`，保持现有附件上下文能力。

## 非目标

1. 不迁移旧的 `messages.metadata.attachments` 历史数据。
2. 不做跨项目文件共享。
3. 不做文件版本管理、标签、全文检索或权限系统。
4. 不改变现有单文件大小和单条消息附件数量限制，除非实现时发现必须调整。

## 数据模型

### `files`

项目级文件资产表。

字段建议：

- `id TEXT PRIMARY KEY`
- `project_id TEXT NOT NULL`
- `original_name TEXT NOT NULL`
- `stored_name TEXT NOT NULL`
- `mime_type TEXT NOT NULL`
- `size INTEGER NOT NULL`
- `url TEXT NOT NULL`
- `storage_path TEXT NOT NULL`
- `uploaded_by_id TEXT`
- `uploaded_by_name TEXT`
- `created_at INTEGER NOT NULL`
- `deleted_at INTEGER`

约束与索引：

- `project_id` 外键指向 `projects(id)`，项目删除时级联删除记录。
- `idx_files_project_created`：`project_id, created_at DESC`
- `idx_files_project_deleted`：`project_id, deleted_at`

### `message_file_refs`

消息与文件的引用关系表。文件归属于项目，引用关系记录聊天行为。

字段建议：

- `id TEXT PRIMARY KEY`
- `project_id TEXT NOT NULL`
- `room_id TEXT NOT NULL`
- `message_id TEXT NOT NULL`
- `file_id TEXT NOT NULL`
- `created_at INTEGER NOT NULL`

约束与索引：

- `project_id` 外键指向 `projects(id)`。
- `room_id` 外键指向 `rooms(id)`。
- `message_id` 外键指向 `messages(id)`。
- `file_id` 外键指向 `files(id)`。
- 同一消息内同一文件只引用一次：`UNIQUE(message_id, file_id)`。
- `idx_message_file_refs_file`：`file_id, created_at DESC`
- `idx_message_file_refs_room`：`room_id, created_at DESC`

## 存储策略

文件仍保存到服务端本地目录，但从消息附件目录调整为项目文件目录：

```text
packages/backend/data/uploads/files/<projectId>/<storedName>
```

对外访问 URL：

```text
/uploads/files/<projectId>/<storedName>
```

`storedName` 继续使用安全随机文件名，保留安全扩展名，不暴露用户本地路径。

## 后端 API

### 文件列表

```http
GET /api/projects/:projectId/files
```

返回当前项目未删除文件。返回项包含：

- 文件基础信息
- `reference_count`
- `last_referenced_at`
- `last_referenced_room_id`
- `last_referenced_room_name`

### 上传项目文件

```http
POST /api/projects/:projectId/files
Content-Type: multipart/form-data
```

表单字段：

- `files`：一个或多个文件
- `uploaded_by_id` 可选
- `uploaded_by_name` 可选

返回创建的文件记录数组。

### 删除文件

```http
DELETE /api/files/:fileId
```

行为：

1. 写入 `files.deleted_at`。
2. 尝试删除磁盘文件。
3. 不删除 `message_file_refs`。
4. 不删除历史消息。
5. 后续列表和文件选择器不再返回该文件。

如果磁盘文件已经不存在，接口仍应完成软删除。

### 发送消息扩展

```http
POST /api/rooms/:roomId/messages
```

扩展支持：

- multipart 上传本地文件时，先写入 `files` 表，再创建消息引用。
- JSON 或 multipart 均支持 `fileIds`，引用项目文件库中的已有文件。

服务端校验：

1. `roomId` 必须存在。
2. 引用的 `fileIds` 必须属于该 room 的 `project_id`。
3. 引用的文件必须未删除。
4. 同一消息内重复 `fileId` 去重。
5. 本地上传文件和文件库引用合计数量不得超过单条消息附件上限。

消息 metadata 继续包含附件快照，保证现有消息渲染和 agent dispatch 能复用：

```json
{
  "attachments": [
    {
      "id": "file_id",
      "fileId": "file_id",
      "name": "设计稿.png",
      "mimeType": "image/png",
      "size": 12345,
      "url": "/uploads/files/project_id/stored.png",
      "isImage": true,
      "deleted": false
    }
  ]
}
```

历史消息中若文件后续被删除，前端可根据文件详情或附件 URL 请求失败展示“文件已删除”状态。实现时也可以在删除时更新对应消息 metadata 的附件快照为 `deleted: true`，但不是首选路径。

## 前端页面

### 路由

新增：

```text
/projects/:projectId/files
```

左侧“文件”菜单：

- 当前已选项目时跳转 `/projects/:projectId/files`。
- 未选项目时保持不可用，或跳回首页并提示先选择项目。

### `FilesPage`

页面能力：

1. 顶部显示项目名、文件数量、总大小。
2. 上传按钮支持多文件上传。
3. 搜索框按文件名过滤。
4. 文件列表展示：
   - 文件名
   - MIME 类型或简化类型
   - 文件大小
   - 上传时间
   - 最近引用群聊
   - 引用次数
5. 行操作：
   - 预览
   - 下载
   - 删除

预览策略：

- 图片：弹窗内直接预览。
- PDF：弹窗 iframe 或新标签打开，取决于现有静态服务响应头限制。
- 文本：可在后续增强；首版可新标签打开或下载。
- 其他类型：显示基础信息和下载入口。

### 群聊编辑器引用文件

`RichMessageComposer` 增加文件库按钮，打开 `FilePickerDialog`。

待发送附件分为两类：

- 本地待上传文件：`File`
- 已有文件引用：`ProjectFile`

发送时：

- 本地文件走 multipart `files`。
- 已有文件引用传 `fileIds`。
- 附件栏统一展示文件名、大小、类型，并支持移除。

## Agent 调度

现有 dispatch 会从消息 metadata 的附件 URL 解析本地路径。新文件 URL 前缀变为 `/uploads/files/` 后，需要扩展解析逻辑：

1. 支持 `/uploads/messages/` 旧格式。
2. 支持 `/uploads/files/<projectId>/<storedName>` 新格式。
3. 路径解析必须限制在上传根目录内，防止路径穿越。

Agent prompt 中继续包含：

- 文件名
- MIME 类型
- 文件大小
- URL
- `localPath`

## 删除行为

采用软删除加物理删除：

1. `files.deleted_at` 写入当前时间。
2. 删除服务端磁盘文件。
3. 历史消息保留引用关系。
4. 历史消息 UI 显示“文件已删除”，不再提供下载和预览。
5. 新消息不能引用已删除文件。

## 验证策略

### 后端

- 上传文件写入 `files` 表，并返回项目级 URL。
- 文件列表只返回当前项目未删除文件。
- 文件引用必须校验项目归属。
- 删除文件后列表不再返回，磁盘文件被删除或缺失时接口仍成功。
- 发送消息时 `fileIds` 创建 `message_file_refs` 并写入 metadata 快照。
- Agent dispatch 能解析新 URL 的本地路径。

### 前端

- 文件菜单能进入当前项目文件页。
- 文件页面可以上传、展示、预览、下载、删除。
- 群聊编辑器可以从文件库选择文件并发送。
- 发送后消息中展示被引用文件。
- 删除后的文件不再出现在选择器中。

### 最低门禁

当前仓库以 `npm run build` 作为最低验证门禁。实现完成后至少运行：

```bash
npm run build
```

如新增测试 runner 或现有测试可用，应补充运行相关后端单元测试。

## 风险与权衡

1. `files` 表和消息 metadata 同时保存附件信息，会带来轻微冗余；好处是消息快照稳定，渲染和 agent dispatch 不依赖额外查询。
2. 物理删除后历史消息无法再打开文件；通过“文件已删除”状态保持历史可追溯。
3. 文件页面预览能力受静态服务响应头影响；首版可优先保证图片预览和通用下载，PDF/文本预览可根据实现复杂度调整。
4. 项目级文件库避免群聊间重复上传，但删除文件会影响同项目内所有引用，需要删除确认文案明确说明影响范围。

## 已确认决策

1. 采用方案 C：新增 `files` 表。
2. 不迁移历史数据。
3. 文件归属为项目级。
4. 文件页面支持主动上传。
5. 群聊消息编辑框通过文件库按钮选择项目文件。
6. 删除采用软删除加物理删除，历史消息保留删除状态。

# 工作流上下文条目优化设计

## 背景

当前多智能体任务流转中，Claude Code、OpenCode、Codex 等 ACP CLI 各自运行在独立 session。系统通过后端工作流、消息、运行记录、步骤结果和阶段产物把一个智能体的输出传给后续智能体。

现有问题是下游 prompt 的上下文边界不够清晰：原始输出、阶段产物、审查结果和验收内容可能被直接拼进后续 prompt。普通 agent 输出会完整保存到 `messages` 和 `agent_runs`；轻量群聊协作有 4000 字截断；记忆提取会异步沉淀短记忆；但正式 workflow 缺少一个专门面向“下游智能体消费”的压缩上下文层。

用户确认采用独立表方案：新增 `workflow_context_entries`，不兼容现有 artifact 作为下游上下文来源。也就是说，后续智能体的工作流上下文应从新表读取，而不是从 `task_artifacts` 拼接。

## 目标

1. 为正式 workflow 增加独立、结构化、可预算的上下文条目层。
2. 保留原始输出用于 UI、审计和 debug，但不默认进入后续智能体 prompt。
3. 后续智能体只消费 `workflow_context_entries` 中的摘要、交接、决策、验证和问题条目。
4. 支持 token/字符预算、版本化、来源追溯和后续检索扩展。
5. LangGraph workflow 首先接入；legacy orchestrator 可按同一接口接入，但不作为首要优化路径。

## 非目标

1. 不删除或压缩 `messages`、`agent_runs.stdout`、`workflow_steps.result`、`task_artifacts.content` 中的原始内容。
2. 不继续把 `task_artifacts` 作为下游 prompt 的兼容回退来源。
3. 不在第一版实现复杂语义检索、向量索引或跨 workflow 上下文复用。
4. 不要求 LLM 摘要成功才能推进工作流。
5. 不改变 ACP CLI session 的生命周期或不同 CLI 之间的 session 隔离。

## 推荐方案

新增 `workflow_context_entries` 表，作为正式 workflow 内部的唯一下游上下文来源。每个 workflow 阶段完成后，后端从该阶段的原始输出和结构化结果生成一组上下文条目。后续阶段构造 prompt 时读取这些条目，而不是读取 `task_artifacts`。

核心思想：

```text
Agent 原始输出
  -> 完整保存到 message / agent_run / workflow_step / artifact
  -> 生成 workflow_context_entries
  -> 后续 agent prompt 只读取 workflow_context_entries
  -> 需要追溯时通过 source refs 找原始输出
```

选择该方案的原因：

- 边界清晰：原始产物负责存档，上下文条目负责传递。
- 可测试：可以直接断言 prompt 不包含超长原始输出。
- 可扩展：后续可以加检索、token 统计、上下文版本、人工编辑。
- 可降级：摘要生成失败时可写确定性 fallback 条目，不阻塞 workflow。

## 数据模型

新增表：

```sql
CREATE TABLE IF NOT EXISTS workflow_context_entries (
  id TEXT PRIMARY KEY,
  workflow_run_id TEXT NOT NULL,
  workflow_step_id TEXT,
  task_id TEXT NOT NULL,
  room_agent_id TEXT,
  agent_run_id TEXT,
  source_type TEXT NOT NULL,
  source_id TEXT NOT NULL,
  entry_type TEXT NOT NULL,
  title TEXT NOT NULL,
  content TEXT NOT NULL,
  metadata TEXT,
  raw_char_count INTEGER NOT NULL DEFAULT 0,
  summary_char_count INTEGER NOT NULL DEFAULT 0,
  token_estimate INTEGER NOT NULL DEFAULT 0,
  version INTEGER NOT NULL DEFAULT 1,
  created_at INTEGER NOT NULL,
  FOREIGN KEY (workflow_run_id) REFERENCES workflow_runs(id) ON DELETE CASCADE,
  FOREIGN KEY (workflow_step_id) REFERENCES workflow_steps(id) ON DELETE CASCADE,
  FOREIGN KEY (task_id) REFERENCES tasks(id) ON DELETE CASCADE,
  FOREIGN KEY (room_agent_id) REFERENCES room_agents(id) ON DELETE SET NULL,
  FOREIGN KEY (agent_run_id) REFERENCES agent_runs(id) ON DELETE SET NULL
);
```

索引：

```sql
CREATE INDEX IF NOT EXISTS idx_workflow_context_run ON workflow_context_entries(workflow_run_id, created_at);
CREATE INDEX IF NOT EXISTS idx_workflow_context_step ON workflow_context_entries(workflow_step_id);
CREATE INDEX IF NOT EXISTS idx_workflow_context_task ON workflow_context_entries(task_id, created_at);
CREATE INDEX IF NOT EXISTS idx_workflow_context_type ON workflow_context_entries(workflow_run_id, entry_type, created_at);
CREATE UNIQUE INDEX IF NOT EXISTS idx_workflow_context_source_version
  ON workflow_context_entries(source_type, source_id, entry_type, version);
```

建议类型：

- `source_type`: `agent_run`、`workflow_step`、`artifact`、`verification`、`system`
- `entry_type`: `summary`、`handoff`、`decision`、`verification`、`file_change`、`issue`、`open_question`

`metadata` 用于保存可追溯引用，例如：

```json
{
  "workflow_stage": "implementation",
  "graph_node": "execute",
  "raw_refs": {
    "message_id": "msg_x",
    "agent_run_id": "run_x",
    "workflow_step_id": "step_x",
    "artifact_id": "artifact_x"
  },
  "summary_method": "deterministic|model|fallback",
  "truncated": false
}
```

## 写入流程

每个阶段完成时调用统一 helper，例如：

```ts
createWorkflowContextEntries({
  workflowRun,
  workflowStep,
  task,
  agentRun,
  message,
  artifact,
  stage,
  output,
  parsedResult,
});
```

生成策略：

1. **确定性摘要优先**
   - planning：从 parsed plan 生成 `summary`、`decision`、`verification`。
   - dispatch：生成任务分配 `summary`。
   - implementation：优先从 agent 输出中提取变更文件、验证结果、交接说明；无法解析时写 fallback `handoff`。
   - review：从 JSON verdict 生成 `issue`、`decision`、`summary`。
   - verify：从命令结果生成 `verification`。
   - acceptance：从 JSON verdict 生成最终 `summary`。

2. **模型摘要作为增强**
   - 当输出超过阈值或确定性提取信息不足时，可调用已配置模型生成 digest。
   - 模型失败不阻断 workflow，只记录 warning，并写 fallback 条目。

3. **fallback 条目**
   - 对无法结构化解析的输出，写入截断后的摘要。
   - fallback 必须包含 raw refs，便于追溯原文。

## 读取与 Prompt 组装

新增 formatter：

```ts
formatWorkflowContextEntries(entries, {
  maxEntryChars: 1500,
  maxTotalChars: 10000,
});
```

prompt 中的上下文结构：

```text
已有工作流上下文：
1. [summary] planning / 执行计划摘要
   ...
2. [handoff] implementation / 执行交接
   ...
3. [issue] code_review / 审查问题
   ...

原始输出引用：
- message_id=...
- agent_run_id=...
- workflow_step_id=...
```

排序和预算：

1. 当前 workflow 的条目优先。
2. 当前 task 或子 task 相关条目优先。
3. `handoff`、`issue`、`verification`、`summary` 优先。
4. 最近条目优先。
5. 单条默认最多 1500 字，总量默认最多 10000 字。
6. 超预算时截断并标记 `...已截断`。

重要约束：下游 prompt 不再调用 `formatArtifacts(context.artifacts)` 作为工作流上下文来源。`task_artifacts` 仍可用于 UI 和审计，但不是下游智能体的默认上下文。

## LangGraph 接入

第一版重点接入 LangGraph：

1. `readWorkflowContext()` 加载 `workflow_context_entries`。
2. `buildStagePrompt()` 接收 `workflowContext` 字符串，替代 `artifacts` 字段。
3. `planningNode` 完成后写 plan 摘要条目。
4. `dispatchNode` 完成后写任务分配摘要条目。
5. `executeNode` 完成后写 implementation handoff 条目。
6. `reviewNode` 完成后写 review verdict 和 issue 条目。
7. `verifyNode` 完成后写 verification 条目。
8. `acceptanceNode` 完成后写 acceptance summary 条目。
9. `memoryNode` 可以继续写长期记忆，但不承担短期工作流上下文职责。

legacy orchestrator 可以使用同一 repo/helper 后续接入。若 legacy 暂未接入，它的 prompt 也不应继续引入无预算 artifact；可以先限制为不包含 artifact 上下文或接入同一 formatter。

## API 与 UI

第一版 API 建议只读：

- `GET /workflows/:id/context`

返回：

```json
{
  "entries": [],
  "total_token_estimate": 1234,
  "total_summary_chars": 5678
}
```

UI 可以后续增强，在 workflow timeline 或任务详情中显示“上下文摘要”。第一版不要求实现复杂编辑能力。

## 错误处理

1. 写 context entry 失败不应直接导致 agent run 失败。
2. 如果关键阶段无法写入任何条目，应记录 system event 或 warning，便于排查。
3. 唯一约束冲突时视为幂等重放，优先跳过或更新同版本条目。
4. LLM 摘要失败时使用 fallback 摘要。
5. prompt 构造时如果没有任何 context entry，应明确写入 `已有工作流上下文：暂无。`，不要回退读取 artifact 原文。

## 测试与验证

后端测试：

1. `workflowContextRepo` 可创建、列表、按 workflow 查询、按 entry type 查询。
2. 同一 `source_type/source_id/entry_type/version` 重复写入具备幂等行为。
3. formatter 对单条和总量做预算截断。
4. LangGraph execute 完成后生成 `handoff` 条目。
5. review 阶段 prompt 包含 implementation handoff，不包含超长 implementation raw output。
6. acceptance 阶段 prompt 包含 review issue/verdict 和 verification summary。
7. context entry 写入失败时 workflow 不崩溃。
8. `GET /workflows/:id/context` 返回条目和统计信息。

最低验证：

```bash
PATH="$(dirname $(mise which node)):$PATH" npm run test -w @openclaw-room/backend
PATH="$(dirname $(mise which node)):$PATH" npm run build
```

建议增加一个长输出回归测试：

- 构造 50,000 字 implementation 输出。
- 确认 review prompt 不包含该 raw 输出。
- 确认 prompt 包含 `handoff` 和 raw refs。
- 确认 prompt 总字符数低于预算。

## 风险

1. 若摘要质量过低，后续智能体可能缺少必要细节。
2. 如果不保留 raw refs，排查问题会变困难。
3. 完全停止 artifact 作为下游上下文来源后，必须确保每个关键阶段都写入 context entry。
4. LLM 摘要会增加额外调用成本；需要默认可降级。
5. legacy orchestrator 若未同步改造，可能出现 LangGraph 与 legacy 行为差异。

## 验收标准

1. 数据库存在 `workflow_context_entries` 表和必要索引。
2. LangGraph 后续阶段 prompt 的工作流上下文来自 `workflow_context_entries`。
3. 下游 prompt 不再默认拼接 `task_artifacts.content`。
4. 长原始输出不会原样进入 review/acceptance prompt。
5. 每个 context entry 可追溯到原始 message、agent run、workflow step 或 artifact。
6. 摘要生成失败时 workflow 继续推进，并写 fallback 条目。
7. 有 API 可查看 workflow context entries。
8. 后端测试和构建通过。

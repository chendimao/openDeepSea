# 去除轻量群聊协作只保留普通群聊与 workflow 设计

## 背景

当前群聊页存在三层模式：

1. 普通群聊：用户直接发消息，显式 @ 智能体或 fallback agent/model 回复。
2. 轻量群聊协作：planner 生成 `collaboration_decision`，用户在“群聊协作”和“正式 workflow”之间选择；若选择群聊协作，则按 `execute -> review -> acceptance -> summary` 运行。
3. 正式 workflow：从消息创建 task，并启动 LangGraph workflow。

轻量群聊协作最初用于修复“执行型请求导致多个智能体同时并发”的问题，但现在它与 workflow 的边界重叠，增加了模型决策、前端选择、后端 runner、agent_run 字段和测试维护成本。产品目标收敛为：只保留普通群聊和 workflow。

## 目标

- 删除“轻量群聊协作”作为独立模式的用户入口和执行链路。
- 普通群聊继续保留即时交流能力，不承担阶段化执行、审查、验收或任务追踪。
- workflow 成为唯一正式执行模式，承接实现、修复、验收、提交、回归等交付型请求。
- 前端不再展示“群聊协作 / 正式 workflow”二选一卡片。
- 后端不再生成或消费 `collaboration_decision` 作为模式选择。

## 非目标

- 不删除普通群聊。
- 不删除显式 @ 智能体回复能力。
- 不删除 fallback agent/model 的普通回复能力。
- 不重写 LangGraph workflow。
- 不在本次设计中引入新的自动 workflow 启动策略；自动化程度继续遵循现有任务就绪和交互策略。

## 模式边界

### 普通群聊

普通群聊用于问答、解释、分析、讨论和临时判断。

保留行为：

- 用户显式 @ 智能体时，目标智能体可直接回复。
- 未 @ 时，可按房间设置走 fallback agent 或 model chat。
- 多个智能体可以各自回复，但这只是普通聊天，不形成阶段化协作运行。

删除行为：

- 不生成 `collaboration_decision`。
- 不展示轻量协作选择卡片。
- 不创建 `collaboration_run_id`。
- 不按 `execute/review/acceptance/summary` 阶段调度。

### Workflow

workflow 用于正式交付型请求。

适用场景：

- 实现功能。
- 修复 bug。
- 修改代码或配置。
- 执行测试、验收、回归。
- 提交 commit 或准备 PR。

保留行为：

- 从群聊消息创建 task。
- 使用现有 `promote-to-workflow` 路由启动 workflow。
- 继续使用 LangGraph workflow 的规划、执行、审查、验证和记忆流程。
- 前端继续展示 workflow 任务卡片、进度和事件。

## 后端设计

### Dispatcher

`dispatchUserMessage` 不再调用 `buildCollaborationDecisionPrompt` 或 `parseCollaborationDecision`。

目标行为：

- 显式 @：照常运行目标智能体。
- fallback reply：planner/fallback agent 作为普通回复运行，不产生模式选择系统消息。
- 无可运行目标：照常走 model chat 或静默。

需要删除：

- `decision_from_planner_fallback` 分支。
- `collaboration_decision` 系统消息创建逻辑。
- `collaboration_decision_failed` 系统消息逻辑。
- 与 collaboration decision prompt/parser 相关的 import。

### Routes

删除轻量协作启动接口：

```text
POST /rooms/:roomId/collaborations
```

保留 workflow 提升接口：

```text
POST /rooms/:roomId/messages/:messageId/promote-to-workflow
```

`promote-to-workflow` 可继续读取既有历史消息 metadata 中的 `task_readiness` 或旧 `collaboration_decision`，用于兼容历史数据生成 task title/description。但新消息不再写入 `collaboration_decision`。

### Runner 与类型

删除：

- `packages/backend/src/collaboration-runner.ts`
- `CollaborationRunResult`
- `CollaborationStepResult`
- `CollaborationRunStatus`
- `CollaborationStepStatus`
- `COLLABORATION_STAGES`
- `CollaborationStage`
- `agent_runs.collaboration_run_id`
- `agent_runs.collaboration_stage` 的新写入路径

数据库字段是否立即迁移删除需要谨慎处理。第一轮实现建议只停止读写并从类型/API 层移除，暂不做破坏性迁移，避免影响已有本地 SQLite。后续如需要清理 schema，可单独做数据迁移设计。

## 前端设计

删除：

- `CollaborationDecisionCard`
- `api.startCollaboration`
- `RoomPage` 中的 `startCollaboration` mutation。
- `RoomPage` 对 `metadata.collaboration_decision` 的特殊渲染。
- 样式中的 `.collaboration-decision-*` 规则。

保留：

- `api.promoteMessageToWorkflow`
- 任务就绪行动条或现有 workflow 启动入口。
- workflow 事件气泡和任务进度展示。

历史兼容：

- 如果历史消息仍带 `collaboration_decision`，前端可以按普通系统消息展示，不再渲染选择卡片。
- 如果仍需从历史决策消息启动 workflow，应通过消息本身或已有任务就绪入口处理，不保留“群聊协作”按钮。

## 测试策略

后端：

- 删除 `collaboration-runner.test.ts`。
- 删除 `/collaborations` 路由测试。
- 更新 dispatcher 测试，断言 fallback 不再生成 `collaboration_decision`。
- 保留并补充 `promote-to-workflow` 测试，确认 workflow 入口仍可用。

前端：

- 删除 `CollaborationDecisionCard` 相关测试或快照。
- 更新 `messageMetadata` 测试，移除 collaboration decision 解析预期。
- 更新 `RoomPage` 测试，确认系统消息不会渲染轻量协作选择卡片。
- 保留 workflow task card 去重和渲染测试。

最终验证：

```bash
npm run test -w @openclaw-room/backend
npm run build -w @openclaw-room/frontend
npm run build
```

如果测试脚本或环境不可用，至少运行受影响文件的定向测试和 TypeScript build，并明确报告未执行项。

## 风险与处理

- 风险：历史 SQLite 中已有 `collaboration_decision` 或 `agent_runs.collaboration_*` 字段。
  - 处理：第一轮不做破坏性迁移；停止新写入，历史数据按普通消息或旧字段保留。
- 风险：用户失去“分析后让多个智能体按阶段协作”的轻量路径。
  - 处理：阶段化执行统一进入 workflow；普通群聊只负责交流。
- 风险：删除类型牵涉面较大。
  - 处理：按后端调度、路由、类型、前端渲染、测试顺序逐步收敛，每步运行定向验证。

## 验收标准

- 群聊页不再出现“群聊协作”按钮或“协作模式选择”卡片。
- 后端不再创建 `event_type=collaboration_decision` 的新系统消息。
- `/rooms/:roomId/collaborations` 不再作为公开接口存在。
- 正式 workflow 入口仍能从群聊消息创建 task 并启动 workflow。
- 普通群聊、显式 @、fallback 回复不受影响。

# 开发闭环设计

## 背景

OpenClaw Room 当前已经具备项目、聊天室、Agent 成员、消息流、任务看板、ACP 配置和 Agent 执行记录。用户可以在聊天室中发消息，系统会把消息转发给指定 Agent 或项目兜底 Agent；用户也可以手动创建任务、指派 Agent、切换任务状态。

目标能力是：在指定项目的指定聊天室中，用户发布任务后，聊天室中的不同职责智能体可以围绕同一个任务完成分析、计划、分配、执行、代码审查和功能验收，形成可追踪、可重试、可人工干预的开发闭环。

## 设计结论

第一版采用混合推进模式：

- 分析和计划阶段由 Agent 自动执行。
- 计划完成后进入人工确认，用户确认后再继续。
- 执行、代码审查和功能验收阶段自动推进。
- 任一阶段失败时停止推进，保留失败原因，并允许重试、取消或改派。

OpenClaw 作为执行底座，不作为业务编排层：

- 复用 OpenClaw Gateway 的 Agent、session、消息发送、取消、事件订阅能力。
- 复用现有 ACP adapter 执行本地 CLI。
- OpenClaw TaskFlow 暂不作为第一版主存储，但数据模型预留 `openclaw_flow_id` 和 `openclaw_child_task_id`，后续可映射到 TaskFlow。
- 开发闭环的状态机、阶段产物、人工确认、子任务分配和 UI 由 OpenClaw Room 自己实现。

## 范围

### 本期包含

- 新增开发闭环数据模型。
- 支持从任务启动一个 workflow。
- 支持阶段：分析、计划、等待确认、分配、执行、代码审查、功能验收、完成、失败。
- 支持计划确认后自动生成子任务。
- 支持按 Agent 职责自动选择阶段执行者。
- 支持每个阶段绑定 Agent run。
- 支持前端展示 workflow timeline、阶段输出、错误、重试和取消。

### 本期不包含

- 不实现独立 PR 创建、merge、push。
- 不引入 MetaGPT、ChatDev、CrewAI 等外部编排框架。
- 不把 OpenClaw Room 改造成 OpenClaw plugin。
- 不做复杂 DAG 编排；第一版使用线性阶段加子任务批次。
- 不强制自动 commit。自动 commit 属于后续执行策略增强。

## 核心状态机

Workflow 状态：

- `draft`：已创建但尚未启动。
- `running`：正在执行阶段。
- `awaiting_approval`：计划完成，等待用户确认。
- `blocked`：阶段失败或缺少可用 Agent。
- `cancelled`：用户取消。
- `completed`：验收通过。
- `failed`：验收失败或用户标记失败。

阶段枚举：

- `analysis`：分析任务目标、上下文、风险和成功标准。
- `planning`：生成实施计划、子任务、验证方式和建议分配。
- `assignment`：根据计划创建子任务并分配 Agent。
- `implementation`：执行子任务。
- `code_review`：审查执行结果和代码变化。
- `acceptance`：按任务目标和验收标准做功能验收。

阶段状态：

- `pending`
- `running`
- `awaiting_approval`
- `completed`
- `failed`
- `cancelled`
- `skipped`

## 编排流程

### 启动 workflow

用户在任务详情中点击“启动闭环”。

后端执行：

1. 校验任务存在且属于当前聊天室。
2. 检查是否已有 active workflow。
3. 创建 `workflow_runs`。
4. 创建 `analysis` step。
5. 选择 `analyst`，若缺失则使用 `planner`，再缺失则进入 `blocked`。
6. 调用现有 `respondAsAgent` 能力启动 Agent run。

### 分析阶段

分析 Agent 输入包含：

- 项目名称和路径。
- 聊天室名称。
- 主任务标题和描述。
- 当前聊天室 Agent 列表与职责。
- 最近消息摘要。

输出要求：

- 任务目标。
- 成功标准。
- 风险。
- 需要确认的问题。
- 建议是否继续规划。

分析完成后自动进入 planning。

### 计划阶段

规划 Agent 输入包含分析结果。

输出要求包含一个结构化计划块：

```json
{
  "summary": "计划摘要",
  "tasks": [
    {
      "title": "子任务标题",
      "description": "子任务描述",
      "suggestedRole": "executor",
      "priority": "normal",
      "acceptance": ["验收点"]
    }
  ],
  "reviewFocus": ["审查重点"],
  "verification": ["验证方式"],
  "risks": ["风险"]
}
```

后端解析成功后保存 `plan` artifact，并把 workflow 置为 `awaiting_approval`。

如果解析失败：

- 保存原始计划。
- workflow 进入 `blocked`。
- UI 提供“手动确认并继续”或“重试计划”。

### 人工确认

用户在任务详情中查看计划。

可执行操作：

- 确认计划：进入 assignment。
- 重试计划：重新触发 planning。
- 取消闭环：workflow 进入 `cancelled`。

第一版不做在线编辑计划。需要修改时通过聊天室追加说明后重试 planning。

### 分配阶段

确认计划后，后端：

1. 根据计划创建子任务。
2. 根据 `suggestedRole` 和房间 Agent 的 `workflow_role` 分配执行者。
3. 无匹配 executor 时使用任意 `executor`，仍无则进入 `blocked`。
4. 创建 assignment artifact。
5. 进入 implementation。

### 执行阶段

第一版串行执行子任务，降低冲突和调试复杂度。

每个子任务执行：

1. 更新子任务为 `in_progress`。
2. 创建 implementation step。
3. 调用分配到的 executor Agent。
4. Agent run 成功则子任务进入 `review`。
5. Agent run 失败则 workflow 进入 `blocked`。

全部子任务执行完成后进入 code_review。

### 代码审查阶段

Reviewer 输入包含：

- 主任务和子任务。
- 分析与计划 artifact。
- 执行阶段输出摘要。
- 可选 git diff 摘要。
- 验证要求。

输出要求：

```json
{
  "verdict": "pass",
  "findings": [],
  "requiredFixes": [],
  "riskLevel": "low"
}
```

`verdict` 可取：

- `pass`
- `changes_requested`
- `failed`

处理规则：

- `pass`：进入 acceptance。
- `changes_requested`：生成修复子任务，回到 implementation。
- `failed`：workflow 进入 `blocked`。

### 功能验收阶段

Acceptor 输入包含：

- 原始任务。
- 计划。
- 子任务状态。
- review 结果。
- 执行输出。
- 构建或验证结果。

输出要求：

```json
{
  "verdict": "pass",
  "acceptedCriteria": [],
  "failedCriteria": [],
  "notes": "验收说明"
}
```

处理规则：

- `pass`：主任务置为 `done`，workflow 置为 `completed`。
- `failed`：主任务置为 `failed` 或 `in_progress`，workflow 置为 `failed`。

## Agent 选择策略

第一版使用确定性规则：

1. 优先选择 `workflow_role` 匹配阶段的 Agent。
2. 多个匹配时优先选择已启用 ACP 的 Agent。
3. 再按加入顺序选择。
4. 如果阶段没有匹配角色：
   - `analysis` 可退到 `planner`。
   - `planning` 可退到 `analyst`。
   - `assignment` 可由系统直接完成。
   - `implementation` 必须有 `executor`。
   - `code_review` 必须有 `reviewer`。
   - `acceptance` 可退到 `reviewer`，但 UI 标记为降级验收。

## 后端 API

新增接口：

- `POST /tasks/:id/workflows`：启动闭环。
- `GET /tasks/:id/workflows`：查看任务的 workflow 列表。
- `GET /workflows/:id`：查看 workflow 详情、steps、artifacts。
- `POST /workflows/:id/approve-plan`：确认计划并继续。
- `POST /workflows/:id/retry-step`：重试当前或指定阶段。
- `POST /workflows/:id/cancel`：取消 workflow。
- `PATCH /rooms/:roomId/agents/:agentId/workflow-role`：设置 Agent 职责。

新增 WebSocket 事件：

- `workflow:created`
- `workflow:updated`
- `workflow_step:created`
- `workflow_step:updated`
- `workflow_artifact:created`

## 错误处理

- 缺少角色 Agent：workflow 进入 `blocked`，提示缺少哪个角色。
- Agent run 失败：step 进入 `failed`，workflow 进入 `blocked`。
- 计划解析失败：保存原文，允许重试或手动继续。
- 用户取消：取消 active agent run，workflow 和 step 进入 `cancelled`。
- 后端重启：启动时把 active workflow 标为 `blocked`，提示需要手动恢复。

## 验证策略

最低验证：

- `npm run build`
- 手动冒烟：
  1. 创建项目和聊天室。
  2. 添加并配置 analyst、planner、executor、reviewer、acceptor。
  3. 创建任务。
  4. 启动闭环。
  5. 等待分析和计划完成。
  6. 确认计划。
  7. 验证执行、审查、验收自动推进。
  8. 验证失败时可重试或取消。

建议补充测试：

- workflow repo 单元测试。
- plan parser 单元测试。
- orchestration service 的阶段推进测试。
- 缺少角色、Agent run 失败、计划解析失败的错误路径测试。

## 验收标准

- 用户可以从任务详情启动开发闭环。
- 分析和计划阶段自动运行，并在计划完成后等待用户确认。
- 用户确认计划后，系统自动创建子任务并分配执行者。
- 执行、代码审查、功能验收能按顺序自动推进。
- 每个阶段都能看到执行 Agent、状态、输出和错误。
- 失败状态不会被静默吞掉，用户可以重试或取消。
- 主任务状态与 workflow 最终状态保持一致。
- 数据模型保留未来接入 OpenClaw TaskFlow 的映射字段。

# 真正 ACP 客户端服务端接入设计

## 背景

当前项目的 `packages/backend/src/acp/` 名义上是 ACP adapter，但实际运行路径仍是本地 CLI JSON 输出适配：

- Codex：`codex exec --json --skip-git-repo-check`
- Claude Code：`claude --print --output-format stream-json`
- OpenCode：`opencode run --format json`

这条路径能拿到部分结构化输出，但它不是 ACP client/server 连接。后端需要自己从 CLI JSON 中猜测 thinking、tool、command、diff 和 plan，导致以下问题：

- provider 输出格式变化时容易丢事件。
- `channel: event` 类 raw event 没有天然协议语义，容易在 dispatcher 层漏持久化。
- tool lifecycle、diff、plan、command output 无法稳定按 ACP session update 还原。
- Codex/Claude/OpenCode 的行为差异被混在单个 stdout normalizer 中，边界不清晰。

目标是把执行层升级为真正 ACP client/server：OpenClaw Room 作为 ACP client，连接 Codex、Claude Code、OpenCode 的 ACP server，把协议事件直接归一化为 `AgentTimelineEvent` 并写入房间消息。

## 目标

- 后端优先通过真正 ACP client/server 运行 Codex、Claude Code、OpenCode。
- `AgentTimelineEvent` 成为消息时间线的主数据源。
- thinking、tool call、tool result、command、command output、file diff、plan update、permission request、raw event 都能进入 `trace.events`。
- 现有 CLI JSON adapter 降级为 fallback，只在 ACP server 不可用或显式配置 legacy 模式时使用。
- 保留现有 room agent 配置、session resume、权限模式和 writable dirs 的产品语义。
- 前端不需要理解 provider 差异，只消费统一 timeline event。

## 非目标

- 不恢复项目内置任务或 workflow 自动创建。
- 不承诺显示 provider 没有明文返回的 thinking。
- 不尝试解密 Codex `encrypted_content`。
- 不在本阶段引入远程云端 agent 执行。
- 不一次性删除 legacy CLI adapter，避免 ACP server 安装或版本问题导致全部不可用。

## 方案选择

### 方案 A：一次性删除 legacy CLI adapter

直接删除 `codex exec --json`、Claude stream-json、OpenCode JSON 路径，只保留 ACP SDK。

优点是架构最干净。缺点是风险集中：不同用户机器上的 ACP server 命令、版本、包名和事件形态可能不同，一次替换会让聊天执行路径不可用。

### 方案 B：新增真正 ACP protocol adapter，legacy CLI 作为 fallback

新增 `AcpProtocolAdapter`，优先使用 ACP SDK 启动和连接 provider server。若 server 命令不存在、初始化失败或配置显式关闭 protocol 模式，再回退现有 CLI adapter。

这是推荐方案。它满足“真正 ACP client/server”目标，同时保留当前可用性；每个 provider 可以逐个切换和验证。

### 方案 C：继续增强 CLI JSON parser

继续修 `normalizeStdoutChunk`，补上 raw event、diff、plan、tool lifecycle 映射。

这只能缓解短期显示问题，不符合本次目标。它可以作为 fallback 维护，但不再作为主路径。

## 架构设计

### 1. Adapter 分层

保留现有 `SessionAdapter` 对上层 dispatcher 的接口，但内部拆为两层：

```text
respondAsAgent
  -> getAdapter(backend)
  -> Protocol-first adapter
      -> AcpProtocolClient.invoke()
      -> 失败时 LegacyCliAdapter.invoke()
```

新增文件建议：

- `packages/backend/src/acp/protocol-client.ts`
- `packages/backend/src/acp/protocol-registry.ts`
- `packages/backend/src/acp/protocol-events.ts`
- `packages/backend/src/acp/legacy-cli.ts`

`protocol-client.ts` 负责启动 ACP server 进程、建立 JSON-RPC/NDJSON 连接、初始化 session、发送 prompt、接收 update。

`protocol-events.ts` 负责把 ACP event 映射成统一 `AgentTimelineEvent`。

`legacy-cli.ts` 保留当前 `runStreaming`、`normalizeStdoutChunk` 等 CLI JSON 兼容逻辑，后续逐步收敛。

### 2. ACP Server Registry

新增默认 provider registry：

```ts
interface AcpServerConfig {
  backend: 'codex' | 'claudecode' | 'opencode';
  mode: 'protocol' | 'legacy' | 'auto';
  command: string;
  args: string[];
  env?: Record<string, string>;
  transport: 'stdio';
  enabled: boolean;
}
```

默认命令建议：

- Codex：`npx @zed-industries/codex-acp`
- Claude Code：`npx @agentclientprotocol/claude-agent-acp`
- OpenCode：`opencode acp`

如果实际安装命令不同，允许通过环境变量覆盖：

- `OPENCLAW_ACP_CODEX_COMMAND`
- `OPENCLAW_ACP_CLAUDECODE_COMMAND`
- `OPENCLAW_ACP_OPENCODE_COMMAND`
- `OPENCLAW_ACP_MODE=auto|protocol|legacy`

初期不新增复杂 UI，避免把 protocol 迁移和配置 UI 混在一起；后续可在系统设置中增加 provider server 配置。

### 3. ACP Session 生命周期

协议主路径应覆盖：

1. 启动 provider ACP server 子进程。
2. 创建 `ClientSideConnection`，通过 stdio NDJSON stream 连接。
3. `initialize` 获取 server capability。
4. 如果 room agent 有 `acp_session_id`，尝试 `loadSession` 或等价恢复；否则 `newSession`。
5. 发送 prompt、图片附件和运行边界上下文。
6. 监听 session update / event stream。
7. 将 answer delta 写入 message content。
8. 将结构化事件写入 `trace.events` 并 WebSocket 推送。
9. 完成后保存新 session id、关闭连接和子进程。
10. cancel/abort 时发送协议 cancel，再兜底 kill 子进程。

### 4. Timeline Event 映射

统一输出仍使用现有 `AgentTimelineEvent`：

```ts
type AgentTimelineEventType =
  | 'thinking'
  | 'assistant_message'
  | 'tool_call'
  | 'tool_result'
  | 'command'
  | 'command_output'
  | 'file_diff'
  | 'plan_update'
  | 'web_search'
  | 'permission_request'
  | 'error'
  | 'raw';
```

ACP event 映射原则：

- 所有可识别事件都保留 `raw`。
- 无法识别的协议事件降级为 `raw`，不得丢弃。
- 同一 tool lifecycle 如果有稳定 call id，应合并为同一个 event；否则按 seq 追加。
- command start、stdout/stderr delta、exit code 应能归到同一 command event 或相邻 command output events。
- file edit/diff 事件优先映射为 `file_diff`，payload 至少包含 `path`、`patch`、`additions`、`deletions`。
- plan/next_steps 映射为 `plan_update`，前端用中文字段展示。
- encrypted thinking 显示 provider 不可读占位，并在 payload 中标记 `encrypted: true`。

### 5. Dispatcher 改造

`respondAsAgent` 当前只处理：

```ts
activity | thinking | tool | command | answer
```

需要改为统一处理：

```ts
answer
activity
event
stderr
```

其中 `event` 携带 `AgentTimelineEvent`，直接进入：

- `messageRepo.mergeTrace(... { events: [event] })`
- `message:stream` WebSocket event
- agent run activity 或事件摘要

legacy CLI fallback 也必须产出同样的 event，而不是让 dispatcher 知道 legacy 细节。

### 6. 数据存储

本阶段继续使用消息 metadata：

```json
{
  "trace": {
    "events": []
  }
}
```

但实现时应避免把大型 stdout、diff 和 raw payload 无限写入同一字段。短期策略：

- 单条 event payload 限制最大字符数。
- raw payload 大于阈值时保留摘要和 `truncated: true`。
- 最终回复正文不再重复写入 raw event。

后续如果事件量明显增大，再迁移到 `agent_run_events` 表。

### 7. 前端影响

前端 `AgentTimeline` 已经以 `trace.events` 为权威来源。协议接入后不需要新增大结构，只需要确保以下显示完整：

- thinking 默认折叠。
- tool call/result 显示参数和结果。
- command 显示命令、stdout/stderr、exit code。
- file diff 使用 diff 样式。
- plan update 用中文标签展示。
- raw event 默认折叠，只在“显示原始事件”或开发信息区域展开。

如果协议事件量较大，前端需要继续按 event id 合并 delta，避免每个 token 生成一张卡片。

## 错误处理

- ACP server 命令不存在：记录 warning，自动 fallback legacy CLI。
- ACP initialize 失败：记录 error event，fallback legacy CLI。
- session resume 失败：尝试 new session，并把旧 session id 失效原因写入 metadata。
- provider event schema 未知：保留 raw event，不阻塞最终回复。
- prompt 发送后中断：标记 run cancelled/interrupted，WebSocket 推送最终状态。
- fallback 也失败：保持现有错误消息和 agent_run failed 语义。

## 测试策略

### 单元测试

- `protocol-registry`：环境变量覆盖、默认命令、mode 选择。
- `protocol-events`：ACP thinking/tool/command/diff/plan/raw 样例映射。
- `dispatcher`：`channel: event` 能持久化到 `trace.events` 并广播。
- legacy fallback：protocol 失败时仍能走旧 CLI adapter。

### 集成测试

- fake ACP server：用本地测试进程输出固定 ACP NDJSON 事件，验证后端完整接收。
- session resume：已有 session id 时调用恢复路径。
- cancel：中断时协议 cancel 和 run 状态正确。

### 手动验收

在真实浏览器中打开房间，分别让 Codex、Claude Code、OpenCode 执行：

- 读文件。
- 写一个小文件或修改测试文件。
- 运行命令。
- 输出 next_steps。

验收 UI 中能看到：

- thinking 或 encrypted thinking 占位。
- tool call/tool result。
- command 和输出。
- file diff。
- plan update。
- 最终 assistant 回复。

## 实施分期

### 阶段 1：事件通道补齐

- 让 dispatcher 接受统一 `event` channel。
- legacy CLI fallback 也产出 `AgentTimelineEvent`。
- 修复 raw patch/plan event 不持久化的问题。

### 阶段 2：ACP protocol client 骨架

- 引入 `@agentclientprotocol/sdk`。
- 实现 stdio transport client。
- 用 fake ACP server 完成端到端测试。
- 保留 legacy fallback。

### 阶段 3：Provider 接入

- Codex 接 `@zed-industries/codex-acp`。
- Claude Code 接 `@agentclientprotocol/claude-agent-acp`。
- OpenCode 接 `opencode acp`。
- 每个 provider 独立测试可用性和事件映射。

### 阶段 4：运行配置与观测

- 增加环境变量和日志，显示当前 provider 使用 protocol 还是 legacy。
- 在 UI 中展示运行来源：`ACP protocol` 或 `legacy CLI`。
- 记录 fallback 原因，方便排查用户机器差异。

## 验收标准

- 新运行的 agent 消息默认通过真正 ACP client/server 路径执行。
- ACP protocol 可用时，不再调用 `codex exec --json` 作为主路径。
- `trace.events` 中能看到 tool/thinking/command/diff/plan/raw 的结构化事件。
- provider 未返回明文 thinking 时，UI 显示明确 encrypted/unavailable 状态，而不是空白。
- raw ACP event 不被丢弃。
- provider server 不可用时，legacy fallback 可继续给出最终回复，并在 run metadata 中记录 fallback 原因。
- `npm run build` 通过。

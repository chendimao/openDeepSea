# ACP 事件时间线与 UI 集成设计

## 背景

当前项目已经转向纯 ACP 对话，但消息回复里仍缺少完整的思考过程、文件 diff、tool call 生命周期、命令输出、plan/next_steps 等内容。根因不是单个前端组件渲染不足，而是现有后端在接收 Codex/Claude/OpenCode 输出时，提前压扁为 `thinking`、`tool`、`command` 三类 trace，导致大量结构化事件没有进入持久化和 WebSocket 流。

目标是把 ACP agent 返回的结构化事件作为一等数据处理，并在房间消息中呈现接近 Codex/Claude Code 的时间线体验。

## 目标

- 同时支持 Claude Code、OpenCode、Codex 的 ACP server 接入。
- 后端保留原始 ACP/Codex 事件，并归一化为统一 timeline event。
- 前端消息逐条显示执行智能体、审查智能体等返回的原始事件摘要。
- thinking 默认折叠，但可展开查看 provider 返回的全部可读 thinking 文本。
- tool call、tool result、command stdout/stderr、exit code、file diff、plan/next_steps 都能以结构化 UI 展示。
- planner 仍能收集其他智能体结果并分析下一步，但不自动创建项目任务或工作流。
- 默认支持两种继续策略：自动继续下一步，或给出建议后等待确认。

## 非目标

- 不伪造 provider 没有返回的 thinking。
- 不尝试解密 Codex 返回的 `encrypted_content`。
- 不恢复旧的自动任务 / 工作流创建逻辑。
- 不把所有 provider 事件强行映射成完全相同字段；无法归一化的部分保留 raw payload。

## 方案选择

推荐采用“ACP client + 事件时间线”的渐进方案。

### 方案 A：继续扩展现有 trace

在 `MessageTrace` 上直接增加 `diffs`、`plans`、`raw_events`。

优点是改动小，能快速补 UI。缺点是事件顺序、生命周期状态、跨 provider 兼容、后续回放都会继续变复杂。

### 方案 B：新增标准化 timeline event

新增统一事件模型，现有消息 metadata 中保存 `trace.events`，每条事件包含 `seq`、`type`、`status`、`title`、`payload`、`raw`。后端可先从现有 `codex exec --json` / Claude stream-json 转换，后续再切真正 ACP SDK。

这是推荐方案，因为它能先解决用户可见问题，同时避免一次性重写所有 provider。

### 方案 C：立即完整替换为官方 ACP SDK

后端直接使用 `@agentclientprotocol/sdk` 连接 `@agentclientprotocol/claude-agent-acp`、`@zed-industries/codex-acp`、`opencode-ai acp`。

这是长期目标，但首次改动较大。应在方案 B 的事件模型稳定后，把 provider 层替换成真正 ACP client。

## 组件设计

### 1. ACP Agent Registry

新增统一 agent server 配置，至少支持以下默认命令：

- Claude Code：`npx @agentclientprotocol/claude-agent-acp`
- Codex：`npx @zed-industries/codex-acp`
- OpenCode：`opencode-ai acp`

配置项包括：

- `provider`
- `command`
- `args`
- `cwd`
- `env`
- `transport`
- `enabled`

短期可以继续保留现有 CLI adapter，同时把输出转换为统一事件；中期切换为 SDK transport。

### 2. 标准化事件模型

新增 `AgentTimelineEvent`：

```ts
type AgentTimelineEventType =
  | 'thinking'
  | 'assistant_message'
  | 'tool_call'
  | 'tool_result'
  | 'command'
  | 'command_output'
  | 'file_diff'
  | 'plan_update'
  | 'web_search'
  | 'permission_request'
  | 'error'
  | 'raw';
```

每条事件包含：

- `id`：事件唯一 ID。
- `message_id`：所属房间消息。
- `run_id`：一次 agent 调用 ID。
- `agent_id`：房间智能体 ID。
- `seq`：同一 run 内严格递增序号。
- `type`：事件类型。
- `status`：`started`、`delta`、`completed`、`failed`。
- `title`：UI 摘要，例如“读取文件 dispatcher.ts”。
- `payload`：归一化结构。
- `raw`：原始 provider event。
- `created_at`：事件时间。

### 3. 数据存储

短期先把事件存在消息 metadata：

```json
{
  "trace": {
    "events": []
  }
}
```

如果事件量明显增大，再迁移到独立表 `agent_run_events`。独立表字段建议：

- `id`
- `room_id`
- `message_id`
- `run_id`
- `agent_id`
- `seq`
- `event_type`
- `status`
- `payload`
- `raw`
- `created_at`

为避免 metadata 无限膨胀，消息列表接口可返回最近 N 条事件摘要，消息详情或展开时再请求完整事件。

### 4. Provider 事件转换

后端新增 `normalizeAgentEvent(provider, rawEvent)`。

Codex 需要覆盖：

- reasoning / thinking delta -> `thinking`
- function_call started/completed -> `tool_call`
- function_call_output -> `tool_result`
- command_execution / local_shell_call -> `command` 和 `command_output`
- patch / apply_patch / file edit payload -> `file_diff`
- plan_update / next_steps-like payload -> `plan_update`
- 不能识别的事件 -> `raw`

Claude Code 需要覆盖：

- thinking/reasoning content -> `thinking`
- tool_use -> `tool_call`
- tool_result -> `tool_result`
- bash output -> `command_output`
- file edit tool input/result -> `file_diff`

OpenCode 需要覆盖：

- message part update -> `assistant_message`
- tool events -> `tool_call` / `tool_result`
- terminal event -> `command_output`
- edit event -> `file_diff`

### 5. WebSocket 流

扩展 `message:stream`：

```ts
{
  type: 'message:stream',
  message_id: string,
  channel: 'answer' | 'event',
  chunk?: string,
  event?: AgentTimelineEvent
}
```

`answer` 继续负责最终 assistant 文本流。`event` 负责 thinking、tool、diff、plan 等结构化时间线。

前端收到 event 后按 `message_id + event.id` 合并；同一事件的 delta 更新同一个卡片，不再每个 chunk 创建一个新卡片。

### 6. 房间 UI

消息气泡保留最终回复正文，下方新增 `AgentTimeline`。

Timeline 卡片包括：

- Thinking：默认折叠，标题显示“思考过程 · 3.2s”，展开后显示完整文本。
- Tool Call：显示工具名、参数摘要、状态、结果摘要，可展开看完整 JSON。
- Command：显示命令、stdout、stderr、exit code。
- File Diff：显示文件路径、增删统计、diff 代码块，支持折叠。
- Plan / Next Steps：中文字段展示目标、下一步、推荐 agent、执行策略。
- Raw Event：仅开发模式或“显示原始事件”开关下显示。

样式要求：

- 默认不展开长文本。
- JSON 字段显示中文标签，但保留原始字段名作为小号辅助信息。
- diff 使用红绿分行样式。
- command output 使用等宽字体、限制高度、可复制。
- 多智能体消息要能区分 agent 名称、provider、run 状态。

### 7. Planner 行为

planner 接收其他智能体返回的 timeline events 和最终回复，形成上下文摘要：

- 开发智能体结果
- 文件变更摘要
- 测试/命令结果
- 错误和阻塞
- 可用 next_steps

planner 可以决定：

- 自动继续下一批智能体。
- 给出下一步建议并等待确认。
- 要求某个智能体补充信息。
- 停止并总结。

但 planner 不创建项目任务、不创建 workflow、不写入旧任务流表。

## 错误处理

- provider 事件解析失败时，保留为 `raw`，不丢事件。
- thinking 是 encrypted 时，显示“provider 返回了加密 reasoning，无法显示原文”。
- diff payload 缺字段时，降级显示 tool result 或 raw JSON。
- WebSocket event 丢失时，以消息 metadata 中最终事件列表为准恢复。
- provider 不支持某类事件时，UI 不显示空卡片。

## 测试策略

- 后端单元测试：覆盖 Codex/Claude/OpenCode 典型事件到 `AgentTimelineEvent` 的映射。
- 后端集成测试：验证 `message:stream` 能同时发送 answer 与 event。
- 前端单元测试：验证 thinking 默认折叠、tool、command、diff、plan 卡片渲染。
- 前端交互测试：验证展开、复制、长文本截断、事件 delta 合并。
- 构建验证：`npm run build`。
- 手动验收：在房间内让 Codex 执行读文件、改文件、运行命令，确认 UI 显示 thinking、tool call、file diff、command output、final answer。

## 分阶段实施

### 阶段 1：事件模型与现有 adapter 扩展

- 增加 `AgentTimelineEvent` 类型。
- 扩展后端 parser，不再丢弃未知事件。
- 将现有 `thinking/tool/command` 同步转成 timeline events。
- 保留旧 trace 字段兼容现有 UI。

### 阶段 2：房间 Timeline UI

- 新增 `AgentTimeline` 组件。
- 替换当前零散 trace panel。
- 增加 diff、plan、raw JSON 渲染。
- 优化中文字段、默认折叠和复制体验。

### 阶段 3：真正 ACP SDK 接入

- 引入 `@agentclientprotocol/sdk`。
- 接入 Claude Code、Codex、OpenCode ACP server。
- 把现有 CLI JSON adapter 降级为 fallback。
- 增加 agent registry 配置 UI。

### 阶段 4：事件表与大事件按需加载

- 新增 `agent_run_events` 表。
- 消息列表只返回摘要。
- 展开 timeline 时按 message/run 加载完整事件。

## 验收标准

- 房间里每个执行/审查智能体的原始 ACP 事件都能逐条显示或以 raw fallback 保留。
- thinking 默认折叠，展开后显示全部可读 thinking 文本。
- 文件修改能显示 diff，而不是只显示“执行了 apply_patch”。
- tool call 有开始、参数、结果、失败状态。
- command 有 stdout/stderr/exit code。
- planner next_steps 用中文结构化卡片展示，点击继续能实际触发下一步。
- 不再自动创建项目任务或 workflow。
- 现有消息最终回复流式展示不回退。

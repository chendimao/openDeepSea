# 群聊流式逐字渲染设计

## 背景

当前群聊中 agent 的流式消息由后端通过 `message:stream` WebSocket 事件返回。前端在 `RoomPage` 中收到每个 chunk 后，直接把 chunk 追加到 React Query 的 `Message.content`，随后 `MessageContent` 立即按完整内容重渲染。

这能保证实时性，但用户看到的是“后端返回一块，页面显示一块”，阅读体验不够接近 OpenAI 网页端的逐字输出。

## 目标

- 正在流式输出的 agent 消息，普通文字按逐字或短 token 节奏显示。
- 最新流式消息末尾显示闪烁光标。
- 代码块、日志、大段 stdout 保持块级追加，避免长代码逐字渲染造成性能与阅读问题。
- 不改变后端持久化内容，不影响刷新后的历史消息展示。
- 不影响用户消息、系统事件、协作决策卡片、任务事件等非 agent streaming 消息。

## 非目标

- 不把后端协议改成字符级事件。
- 不对历史消息做重新播放动画。
- 不引入新的 markdown 渲染库。
- 不重画群聊整体 UI。

## 已确认决策

- 采用“文字逐字，代码/日志块级追加”的混合体验。
- 效果只应用于正在 streaming 的 `agent_stream` agent 消息。
- 优先在前端展示层实现，不改后端 `message:stream` 协议。

## 推荐方案

在前端增加一个按消息 ID 管理的流式展示缓冲层：

1. 后端仍按现有方式广播 `message:stream` chunk，并继续落库完整内容。
2. 前端收到 chunk 后，真实 `Message.content` 仍保持完整追加，作为最终数据源。
3. 对正在 streaming 的 agent 消息，额外维护 `displayedContentByMessageId`。
4. 可逐字展示的文本进入本地队列，由定时器按节奏释放到 `displayedContent`。
5. 代码块、日志、大段内容直接 flush 到 `displayedContent`。
6. streaming done 后，等待本地队列清空，再用真实 `Message.content` 对齐一次并隐藏光标。

## 组件与职责

### `RoomPage`

- 继续订阅 `message:new` 与 `message:stream`。
- 保留 React Query 中完整消息内容更新。
- 将正在 streaming 的消息 ID、chunk、done 状态传入流式展示缓冲逻辑。
- 传递当前消息是否 streaming 给 `MessageBubble`。

### 流式展示缓冲逻辑

建议抽为局部 hook，例如 `useStreamingMessageDisplay`，职责为：

- 按 `messageId` 存储展示内容、待展示队列、计时器状态。
- 识别 chunk 是否适合逐字展示。
- 在消息完成、切换房间或消息消失时清理状态。
- 保证最终展示内容与真实 `Message.content` 一致。

### `MessageBubble`

- 对 streaming agent 消息使用 `displayedContent`。
- 非 streaming 消息仍使用原始 `message.content`。
- 把 `streaming` 标记传给 `MessageContent`。

### `MessageContent`

- 增加可选 `streaming` 参数。
- 在文本末尾渲染光标。
- 光标只在 streaming 且当前内容可见时显示。
- markdown/source 切换逻辑保持现有行为。

## Chunk 分类策略

优先使用保守规则，避免复杂解析：

- 若当前内容处于 fenced code block 内，chunk 块级追加。
- 若 chunk 包含代码 fence 边界，优先块级追加，防止半截 markdown 状态抖动。
- 若 chunk 长度超过阈值，例如 240 字符，块级追加。
- 若 chunk 换行密度高，例如包含 4 个以上换行，块级追加。
- 若 chunk 命中常见日志特征，例如 shell prompt、堆栈、diff、`[error]`、`stdout`、`stderr`，块级追加。
- 其他普通文本进入逐字队列。

后续如体验需要，可把阈值集中为常量微调。

## 渲染节奏

- 默认每 16 到 25ms 释放 1 个字符或短 token。
- 队列积压时动态加速，例如每 tick 释放 2 到 6 个字符。
- 保留换行和空白，不改变原始文本。
- 使用 `Array.from(text)` 或等价方式按 Unicode code point 拆分，避免中文和 emoji 被错误拆成无效片段。

## 光标设计

- 使用 CSS class，例如 `.streaming-cursor`。
- 用 `@keyframes` 做 700 到 900ms 闪烁。
- 光标颜色使用现有 accent token，避免引入新视觉主题。
- 光标跟随最后一段文本或 markdown preview 末尾。
- streaming done 且本地队列清空后隐藏。

## 数据一致性

真实消息内容仍以 React Query 中的 `Message.content` 为准。`displayedContent` 只是临时 UI 状态。

需要处理以下情况：

- 如果 WebSocket 断线后重新拉取消息，直接使用完整 `Message.content`，不重新逐字播放。
- 如果 `message:stream done` 早于本地队列消耗完成，继续播放队列，最后对齐完整内容。
- 如果切换房间，清理旧房间的展示缓冲和计时器。
- 如果消息未命中本地缓存，沿用现有逻辑 invalidate messages。

## 风险与缓解

- **Markdown 半截解析抖动**：对 fence 边界和代码块使用块级追加。
- **长输出性能问题**：对长 chunk、日志、代码直接块级追加，并对积压队列加速。
- **最终内容不一致**：done 后用真实 `Message.content` 对齐。
- **历史消息重复动画**：只对当前 streaming ID 生效。
- **定时器泄漏**：hook 在房间变化、组件卸载、消息完成时统一清理。

## 验证方式

- 运行 `npm run build`，覆盖 TypeScript 编译与前端打包。
- 针对流式展示判断逻辑补充单元测试，至少覆盖：
  - 普通中文文本进入逐字队列。
  - fenced code block 内 chunk 块级追加。
  - 长 chunk 或多行日志块级追加。
  - done 后展示内容与真实 content 对齐。
- 手动验证群聊中 agent 输出：
  - 普通回复逐字出现。
  - 代码块不逐字抖动。
  - streaming 时光标闪烁，完成后消失。
  - 刷新页面后历史消息不重新播放。

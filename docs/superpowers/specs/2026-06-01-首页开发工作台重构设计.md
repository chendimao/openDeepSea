# 首页开发工作台重构设计

## 背景

当前首页是公共 Header 中“开发”菜单对应的项目入口页，主要展示项目卡片。新的方向是将首页重构为真正的开发工作台：左侧展示项目列表，右侧顶部展示当前项目的群聊标签页，标签页下方直接显示群聊主功能区。

本设计明确撤销“全局页面标签”方向，不再把所有页面纳入全局 tab。群聊标签只属于当前项目，不影响聊天、Agent、Skills、文件、测试等公共页面。

## 目标

1. 首页 `/` 成为完整开发工作台，而不是项目卡片导航页。
2. 工作台主体左侧为项目列表，右侧为当前项目的群聊工作区。
3. 右侧顶部使用群聊 tab 切换当前项目内的群聊。
4. 群聊主功能区保持任务工作区在左、聊天区域在右，比例为 60% / 40%。
5. 项目列表和群聊 tab 都支持置顶、同层拖动排序和创建时间兜底排序。
6. 公共 Header 保持全局导航，不把项目列表重新放回 Header。

## 非目标

1. 不实现全局页面 tab。
2. 不引入气泡目录设计。
3. 不重构聊天、Agent、Skills、文件、测试等公共页面。
4. 不改变极简主题为 Apple 玻璃液化风格。
5. 不在本设计阶段修改业务代码。

## 页面结构

公共 Header 仍由 `AppShell` 负责，包含“开发、聊天、Agent、Skills、文件、测试”等公共菜单。

Header 下方的开发工作台主体采用左右布局：

- 左侧：项目列表 `ProjectRail`。
- 右侧：当前项目工作区。
- 右侧顶部：群聊 tab `RoomTabsBar`。
- 右侧下方：群聊主功能区 `RoomWorkbench`。

群聊主功能区内部保持现有方向：

- 左侧 60%：任务工作区。
- 右侧 40%：聊天区域。

## 路由设计

以下路由统一渲染同一个开发工作台页面 `DevelopmentWorkspacePage`：

- `/`
- `/projects/:projectId`
- `/projects/:projectId/rooms/:roomId`

行为规则：

- 访问 `/` 时，如果没有指定项目，自动选择项目排序第一项。
- 访问 `/projects/:projectId` 时，自动打开该项目排序第一的群聊。
- 访问 `/projects/:projectId/rooms/:roomId` 时，直接打开指定群聊。
- 项目不存在时，回退到项目排序第一项。
- 群聊不存在时，回退到当前项目排序第一的群聊。
- 当前项目没有群聊时，右侧显示空态和新增群聊按钮。

URL 仍跟随当前项目和群聊变化，保证刷新、分享链接和浏览器返回按钮行为稳定。

## 组件设计

### DevelopmentWorkspacePage

开发工作台的路由级页面，负责：

- 读取 URL 中的 `projectId` 和 `roomId`。
- 拉取项目列表、当前项目、当前项目群聊列表。
- 根据排序规则解析当前项目和当前群聊。
- 处理路由回退、空态和新增群聊后的跳转。
- 组装 `ProjectRail`、`RoomTabsBar` 和 `RoomWorkbench`。

### ProjectRail

左侧项目列表，负责：

- 展示项目搜索、置顶项目、普通项目和新增项目入口。
- 支持项目置顶、取消置顶和删除。
- 支持置顶层内部拖动排序。
- 支持普通层内部拖动排序。
- 拖动不改变置顶状态。

### RoomTabsBar

继续作为当前项目内的群聊标签栏，负责：

- 展示置顶群聊和普通群聊。
- 支持右键或更多菜单进行重命名、置顶、取消置顶、删除。
- 支持置顶层内部拖动排序。
- 支持普通层内部拖动排序。
- 最右侧保留新增群聊按钮。

### RoomWorkbench

从现有 `RoomPage` 中抽出群聊主功能区，负责：

- 群聊消息、任务、智能体、工作流、记忆面板等现有能力。
- 保持任务工作区与聊天区域 60% / 40% 的布局。
- 继续复用现有查询、WebSocket 订阅和消息流式渲染逻辑。

## 数据模型

项目需要新增持久化排序字段：

- `pinned_at: number | null`
- `sort_order: number | null`

群聊已有：

- `pinned_at: number | null`
- `last_opened_at: number | null`

群聊需要新增：

- `sort_order: number | null`

统一排序规则：

1. 置顶层在普通层之前。
2. 同一层内优先按 `sort_order` 升序。
3. 没有 `sort_order` 的条目按 `created_at DESC` 排序，新创建的条目靠前。

拖动排序只更新同一层内的 `sort_order`，不改变 `pinned_at`。

## API 设计

项目侧需要扩展：

- `PATCH /api/projects/:id` 支持 `pinned_at`。
- 新增或扩展批量排序接口，用于持久化项目同层拖动排序。

群聊侧需要扩展：

- `PATCH /api/rooms/:id` 支持 `sort_order`。
- 新增或扩展批量排序接口，用于持久化群聊同层拖动排序。

批量排序接口应校验：

- 所有条目必须存在。
- 项目排序只能作用于项目列表。
- 群聊排序只能作用于同一项目内的群聊。
- 置顶层和普通层不能混排。

## 交互规则

### 新增

- 新增项目后左侧列表刷新，并进入新项目。
- 新增群聊后右侧切换到新群聊。

### 删除

- 删除当前群聊后，跳转到当前项目剩余排序第一的群聊。
- 当前项目没有剩余群聊时，显示项目空态。
- 删除项目后，跳转到剩余项目排序第一项。
- 删除存在运行中任务或运行中智能体的项目/群聊时，沿用现有 409 提示。

### 置顶

- 置顶操作把条目移入置顶层。
- 取消置顶把条目移回普通层。
- 置顶和取消置顶不依赖拖动完成。

### 拖动排序

- 拖动只允许在当前层内部发生。
- 保存成功后更新本地缓存。
- 保存失败时回滚到服务端数据并显示 toast。

## 空态

无项目：

- 首页显示创建第一个项目的空态。

项目无群聊：

- 左侧仍选中项目。
- 右侧显示创建第一个群聊的空态。

无匹配搜索结果：

- 左侧展示搜索空态和清除搜索入口。

## 视觉要求

- 公共 Header 保持当前系统统一样式。
- 开发工作台应偏工具型、信息密度适中，不做营销式首页。
- 极简主题保持简单、克制，不使用 Apple 玻璃液化主题的专属视觉。
- Apple 风主题可以保留已有玻璃液化效果，但不能影响极简主题。
- 图标优先使用当前系统中的 `lucide-react`。
- 组件边界要稳定，避免 hover、tab 文案、按钮状态导致布局跳动。

## 验证方式

实现完成后再统一验证，不在设计阶段构建。

最终验证至少包括：

1. `npm run build`。
2. 浏览器访问 `/`，确认公共 Header 和开发工作台显示正常。
3. 浏览器访问 `/projects/:projectId`，确认自动进入该项目排序第一的群聊。
4. 浏览器访问 `/projects/:projectId/rooms/:roomId`，确认可直接打开指定群聊。
5. 验证项目置顶、取消置顶、同层拖动排序和刷新后的持久化。
6. 验证群聊重命名、置顶、取消置顶、删除、新增、同层拖动排序和刷新后的持久化。
7. 验证项目无群聊、无项目、搜索无结果等空态。
8. 验证任务工作区与聊天区域保持 60% / 40%。

## 风险与处理

1. `RoomPage` 当前职责较重，抽出 `RoomWorkbench` 时要保持行为不变，避免一次性重写聊天逻辑。
2. 排序字段涉及数据库迁移，需要兼容已有 SQLite 数据。
3. 批量排序接口必须防止跨项目、跨层排序。
4. 拖动排序失败时必须回滚，否则容易造成前端顺序和服务端顺序不一致。
5. 路由统一后，公共 Header 的“开发”菜单需要在 `/projects/:projectId` 和 `/projects/:projectId/rooms/:roomId` 下保持激活。

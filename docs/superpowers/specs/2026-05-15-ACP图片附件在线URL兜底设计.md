# ACP 图片附件在线 URL 兜底设计

## 背景

聊天附件当前已经上传到 backend 本地目录，并通过 `/uploads/messages/:file` 暴露给前端。上一轮修复后，Codex 与 OpenCode 会收到本地图片路径参数，Claude Code 会在 prompt 中收到本地图片路径。

仍存在一个边界：Claude Code CLI 没有稳定的 `--image` 参数，远端或隔离运行的 ACP 也不一定能读取本机绝对路径。因此图片附件需要同时提供可通过 HTTP 访问的在线地址，作为本地路径失败时的兜底。

## 目标

- 给 claudecode、opencode、codex 都提供图片附件的在线 URL。
- 保留本地路径作为本机 CLI 的高可靠主路径。
- 让 prompt 明确说明优先级：本地文件参数优先，URL 兜底；Claude Code 优先尝试 URL，失败后读取 localPath。
- 保持附件访问来源受控，不引入任意外链或路径穿越风险。

## 非目标

- 不引入第三方对象存储。
- 不改变当前消息上传 API 的 multipart 协议。
- 不要求 Claude Code CLI 支持不存在的本地图片 flag。
- 不为附件增加用户级权限系统；当前仍按本地开发工具定位处理。

## 推荐方案

采用“双通道附件引用”：

1. `localPath`：backend 上传目录下的本机绝对路径。
2. `publicUrl`：由 backend 基址和附件相对 URL 生成的绝对 HTTP 地址。

dispatcher 解析消息附件时生成统一结构：

```ts
interface ResolvedMessageAttachment {
  metadata: MessageAttachmentMetadata;
  localPath: string | null;
  publicUrl: string | null;
}
```

prompt 中每个附件行同时包含：

```text
url=/uploads/messages/xxx.png
publicUrl=http://127.0.0.1:7330/uploads/messages/xxx.png
localPath=/.../data/uploads/messages/xxx.png
```

## Backend 基址配置

新增运行时配置：

```bash
PUBLIC_BASE_URL=http://127.0.0.1:7330
```

规则：

- 优先使用 `PUBLIC_BASE_URL`。
- 未配置时默认 `http://127.0.0.1:${PORT}`。
- 生成 `publicUrl` 时只拼接本系统生成的附件相对 URL，不接受用户输入的任意外链。
- `PUBLIC_BASE_URL` 末尾有无 `/` 都应正常处理。

## ACP 适配策略

Codex：

- 继续通过 `--image localPath` 提供原生图片输入。
- prompt 中同时包含 `publicUrl`，用于模型或工具链无法读取本地路径时兜底。

OpenCode：

- 继续通过 `--file localPath` 提供附件。
- prompt 中同时包含 `publicUrl`。

Claude Code：

- 不伪造不存在的 `--image` 参数。
- prompt 中显式追加 “Claude Code 图片附件” 段：

```text
请优先查看 imageUrl；如果无法访问 URL，再读取 localPath。
1. imageUrl=http://127.0.0.1:7330/uploads/messages/xxx.png
   localPath=/.../data/uploads/messages/xxx.png
```

- 调用时继续把上传目录加入 `--add-dir` 范围，保证 localPath fallback 可读。

## 数据流

1. 前端通过 multipart 上传图片。
2. backend 将文件写入 `packages/backend/data/uploads/messages`。
3. `MessageMetadata.attachments[].url` 保存相对路径。
4. dispatcher 处理消息时：
   - 校验相对 URL 是否属于 `/uploads/messages/`。
   - 解析安全的 `localPath`。
   - 生成受控的 `publicUrl`。
5. adapter 收到：
   - `imagePaths`：仍只包含安全 localPath。
   - prompt：包含 `publicUrl` 与 `localPath`。

## 错误处理

- 如果附件 URL 不属于 `/uploads/messages/`，`localPath` 为 `null`，`publicUrl` 也不生成。
- 如果 `PUBLIC_BASE_URL` 不是合法 URL，启动时不崩溃；生成 `publicUrl` 时返回 `null`，并在日志中警告一次。
- 如果 public URL 无法被 ACP 访问，agent 仍可尝试 localPath。
- 如果 localPath 不可用，agent 至少能看到附件名称、MIME、大小和 publicUrl。

## 安全约束

- 不从 metadata 中接受任意 `publicUrl`。
- 不把用户上传文件路径原样拼接到文件系统路径；继续使用现有 `resolveMessageAttachmentLocalPath` 的目录边界校验。
- 静态文件继续设置 `X-Content-Type-Options: nosniff`。
- 是否保留 `Content-Disposition: attachment` 不影响 ACP 读取图片 URL，但可能影响浏览器预览；本设计不变更该行为。

## 测试计划

- `resolvePublicBaseUrl`：
  - 使用 `PUBLIC_BASE_URL`。
  - 未配置时回退到 `127.0.0.1:${PORT}`。
  - 非法 URL 返回 `null`。
- `buildPromptWithMessageAttachments`：
  - prompt 包含 `publicUrl` 与 `localPath`。
  - 非 `/uploads/messages/` URL 不生成 `publicUrl/localPath`。
- dispatcher：
  - 首轮 fallback agent 能收到 `imagePaths`。
  - prompt 中包含 public URL。
- adapter：
  - Codex 仍生成 `--image localPath`。
  - OpenCode 仍生成 `--file localPath`。
  - Claude Code prompt 包含 imageUrl 与 localPath。
- 最低验证：
  - `npm run test -w @openclaw-room/backend`
  - `npm run build`

## 验收标准

- 上传图片后，聊天记录仍能正常显示。
- ACP run 的 prompt 中能看到可访问的绝对图片 URL。
- Codex/OpenCode 的 CLI 参数仍使用本地文件路径。
- Claude Code 能从 prompt 获得 `imageUrl` 与 `localPath` 两种引用。
- 原有附件路径安全测试继续通过。

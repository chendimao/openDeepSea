# 消息意图自动路由设计

- 状态：设计已确认，待实现计划
- 日期：2026-05-30
- 范围：设计用户发送一条消息后，系统自动判断普通聊天、轻量任务、debugger、workflow、brainstorming 的入口路由
- 非范围：不在本文实现具体代码；不替换 ACP / Superpowers runtime；不改变 M1-M4 已有任务闭环架构目标

---

## 1. 背景

当前系统已经具备 Task Router、任务工作区、TaskExecutor、workflow、planner readiness 等能力，但入口判断仍分散在多个层次：

- `routeMessage()` 主要判断“消息应归属哪个任务”，而不是“消息应该进入什么工作模式”。
- `task_readiness` 可以从 planner 回复中推断是否适合生成任务或 workflow，但它发生在 agent 回复之后。
- `CollaborationDecision` 已支持 `chat_collaboration` 与 `formal_workflow` 的分流，但粒度不足以区分普通聊天、轻量任务、debugger、brainstorming。
- ACP / Superpowers 是流程基座，适合承接后续执行流程，但如果先把所有消息交给 ACP 再判断类型，会让分类太晚，并可能污染任务上下文。

因此需要在消息入口新增一层系统级 **MessageIntentRouter**：先判断消息意图，再决定是否进入 ACP / Superpowers。

---

## 2. 目标

用户发送一条消息后，系统自动判断并进入合适模式：

| 意图 | 含义 | 默认动作 |
|---|---|---|
| `chat` | 普通聊天、问答、解释、闲聊 | 留在全局聊天，不创建任务 |
| `light_task` | 小范围明确任务，可直接执行或单 agent 处理 | 创建轻量任务，走任务上下文 |
| `debugger` | 报错、失败、复现、修复、日志分析 | 进入 systematic-debugging 风格流程 |
| `brainstorming` | 头脑风暴、设计讨论、方案探索 | 进入 Superpowers brainstorming gate |
| `workflow` | 明确要实现、开发、完整闭环、多阶段交付 | 进入正式 workflow / M1-M4 闭环 |

核心体验目标：

1. 用户不需要手动选择“聊天 / 任务 / workflow”。
2. 系统要给出可解释分类结果，便于用户纠正。
3. 高置信自动执行，低置信先确认。
4. 分类结果成为系统自己的结构化元数据，不直接依赖 ACP 原始输出。
5. 与 AI Task OS 的 M1-M4 架构兼容：消息仍是入口，Task 仍是系统本体。

---

## 3. 设计原则

1. **分类必须早于执行**  
   消息进入 ACP agent / Superpowers workflow 之前，先形成 `MessageIntentResult`。

2. **ACP 可辅助分类，但不作为唯一事实源**  
   低置信度时可以调用只读 ACP / model classifier，但最终落库的是系统级意图结果。

3. **分类器无副作用**  
   分类阶段不得创建任务、修改文件、启动 workflow、进入 Superpowers 实现阶段。

4. **高置信自动，低置信确认**  
   明确意图直接进入对应模式；模糊意图生成确认卡片，等待用户选择。

5. **复用现有 Task Router**  
   `MessageIntentRouter` 判断“走什么模式”；`routeMessage()` 继续判断“归属哪个任务”。

---

## 4. 方案对比

### 方案 A：纯规则分类

使用关键词、命令前缀、正则和上下文状态判断意图。

优点：
- 快速、便宜、可测试。
- 无模型调用延迟。
- 行为稳定。

缺点：
- 对自然语言表达泛化弱。
- 中英文混合、隐含意图、上下文续写容易误判。

适用：作为第一层快速判断。

### 方案 B：纯 ACP / LLM 分类

所有消息先发给 classifier agent，让其输出 JSON 意图结果。

优点：
- 语义理解强。
- 对复杂表达更稳。

缺点：
- 分类太晚或成本较高。
- 如果复用真实执行 agent，可能产生副作用。
- 依赖 provider 输出稳定性。

适用：只适合作为低置信度辅助，不适合作为唯一入口。

### 方案 C：Hybrid Intent Router（推荐）

先用本地规则给出快速分类与置信度；低置信度或冲突时调用只读 classifier；最后通过策略门禁决定自动执行还是询问用户。

优点：
- 稳定、可解释、低延迟。
- 能利用 ACP / 模型语义能力。
- 与现有任务闭环兼容。

缺点：
- 需要定义一套意图 contract 和确认交互。
- 初期规则需要根据真实使用持续校准。

结论：采用方案 C。

## 5. 核心模型

### 5.1 MessageIntent

```ts
type MessageIntent =
  | 'chat'
  | 'light_task'
  | 'debugger'
  | 'brainstorming'
  | 'workflow';
```

### 5.2 MessageIntentResult

```ts
interface MessageIntentResult {
  intent: MessageIntent;
  confidence: number;
  reason: string;
  source: 'rule' | 'classifier' | 'user_override';
  suggestedAction:
    | 'reply_in_chat'
    | 'create_light_task'
    | 'start_debugger'
    | 'start_brainstorming'
    | 'start_workflow'
    | 'ask_user';
  signals: string[];
}
```

### 5.3 Message Metadata 扩展

用户消息 metadata 增加：

```ts
interface MessageMetadata {
  intent_result?: MessageIntentResult;
}
```

兼容要求：

- 旧消息没有 `intent_result` 时按 `chat` 展示。
- `route_result` 继续保留，用于任务归属。
- `task_readiness` 继续保留，用于 planner 产出的任务准备度。

## 6. 路由顺序

推荐入口顺序：

```txt
POST /rooms/:roomId/messages
  ↓
MessageIntentRouter.classify()
  ↓
策略门禁：auto / ask_user
  ↓
routeMessage() 判断任务归属
  ↓
按 intent 分流
```

具体逻辑：

1. 先运行 `MessageIntentRouter`。
2. 如果 `intent=chat`：
   - 不创建任务。
   - 不强制启动 workflow。
   - 按现有普通群聊 / fallback reply 逻辑处理。
3. 如果 `intent=light_task`：
   - 创建轻量任务，或追加到激活任务。
   - 默认不进入完整 workflow。
   - 可由一个 fallback agent 或指定执行 agent 处理。
4. 如果 `intent=debugger`：
   - 创建/追加 debug 类型任务。
   - 后续实现阶段映射到 systematic-debugging 流程。
5. 如果 `intent=brainstorming`：
   - 创建/追加设计讨论任务。
   - 进入 Superpowers brainstorming gate。
   - 不允许直接进入实现。
6. 如果 `intent=workflow`：
   - 创建/追加正式任务。
   - 进入 writing-plans → implementation → review → verification 的完整闭环。

## 7. 分类信号

### 7.1 普通聊天 `chat`

典型信号：

- “解释一下……”
- “这个是什么？”
- “怎么看……”
- 闲聊、确认、短反馈。

默认策略：

- 置信度高时直接聊天。
- 若同时包含实现/修复信号，则降级到确认。

### 7.2 轻量任务 `light_task`

典型信号：

- “把默认主题改成极简风”
- “临时插入一点修改”
- “改一下文案”
- “加一个小按钮”

边界：

- 小范围、低风险、通常单文件或局部组件。
- 可以执行，但不需要完整 M1-M4 工作流。

### 7.3 Debugger `debugger`

典型信号：

- “为什么页面没有变化”
- “报错了”
- “测试失败”
- “找根因”
- “修复这个 bug”

默认策略：

- 进入问题排查模式。
- 先确认现象、复现、日志、影响范围，再修复。
- 不直接跳到实现，除非根因明确。

### 7.4 Brainstorming `brainstorming`

典型信号：

- “头脑风暴”
- “重新优化交互逻辑”
- “讨论设计”
- “给几个方案”
- “先别实现”

默认策略：

- 进入 Superpowers brainstorming。
- 产出设计并等待用户确认。
- 确认后写 spec，再进入 implementation plan。

### 7.5 Workflow `workflow`

典型信号：

- “完整闭环”
- “需要 M1 到 M4”
- “完成后浏览器实际测试”
- “开发完成需要代码审查”
- “实现这个功能”

默认策略：

- 进入正式任务闭环。
- 要求计划、实现、测试、浏览器验证、代码审查、完成前验证。

## 8. 置信度与确认策略

建议阈值：

| 置信度 | 动作 |
|---|---|
| `>= 0.85` | 自动进入对应模式 |
| `0.60 - 0.84` | 显示确认卡片，默认选中推荐模式 |
| `< 0.60` | 作为普通聊天或询问用户 |

冲突策略：

- 同时出现 `debugger` 与 `workflow`：优先 `debugger`，因为修复前应先确认根因。
- 同时出现 `brainstorming` 与 `workflow`：优先 `brainstorming`，因为用户可能还在设计阶段。
- 同时出现 `light_task` 与 `workflow`：若包含“完整闭环、代码审查、浏览器测试、提交”等强信号，升为 `workflow`。
- 当前已有激活任务时，意图判断仍先执行；任务归属由 `routeMessage()` 决定。

## 9. ACP / Superpowers 集成方式

ACP 不直接接管入口分类，而是承担两个角色。

### 9.1 只读分类辅助

低置信度时调用只读 classifier：

- 输入：用户消息、最近上下文、激活任务摘要、现有任务标题。
- 输出：严格 JSON `MessageIntentResult`。
- 限制：不得调用工具、不得修改文件、不得创建任务、不得启动 workflow。

### 9.2 后续流程执行

分类完成后再进入对应 Superpowers 流程：

| 意图 | Superpowers / ACP 行为 |
|---|---|
| `chat` | 普通 agent 回复或 model fallback |
| `light_task` | 轻量实现流程，可跳过完整 workflow，但仍保留验证 |
| `debugger` | systematic-debugging |
| `brainstorming` | brainstorming |
| `workflow` | writing-plans → implementation → requesting-code-review → verification-before-completion |

## 10. UI 交互

### 10.1 自动分类反馈

消息发送后，气泡或 activity 层显示轻量提示：

```txt
已识别为：Debugger · 置信度 0.91
原因：包含“页面没有变化”“为什么”等排查信号
```

高置信场景不阻塞用户。

### 10.2 低置信确认卡片

低置信场景显示确认卡片：

```txt
我判断这条消息可能是：Brainstorming
原因：你提到“重新优化交互逻辑”

[按 Brainstorming 继续] [改为普通聊天] [改为轻量任务] [进入 Workflow]
```

用户选择后写入 `source='user_override'`，后续同类消息可作为校准信号。

### 10.3 任务工作区联动

- `light_task/debugger/brainstorming/workflow` 都可以在任务区形成任务对象。
- `chat` 不强制出现在任务栏。
- 进入 workflow 后，任务详情面板展示完整闭环进度。

## 11. 与 M1-M4 的关系

本设计不是替代 AI Task OS，而是补齐入口大脑：

| Milestone | 关系 |
|---|---|
| M1 事件地基 | `intent_result` 可作为 chat/activity 事件记录 |
| M2 Task Router | `MessageIntentRouter` 在 `routeMessage()` 前；二者职责分离 |
| M3 任务工作区 | UI 展示意图结果、确认卡片、任务模式 |
| M4 TaskExecutor | 只有 task 类意图才进入 TaskExecutor 隔离上下文 |

关键边界：

- MessageIntentRouter 决定“模式”。
- Task Router 决定“任务归属”。
- TaskExecutor 决定“执行会话归属”。

## 12. 测试策略

实现阶段建议按 Level 2 处理，因为这是入口级行为变更。

### 12.1 单元测试

覆盖：

- 关键词与规则分类。
- 冲突意图优先级。
- 置信度阈值。
- 低置信返回 `ask_user`。
- ACP classifier JSON 解析与非法输出降级。

### 12.2 集成测试

覆盖：

- `POST /rooms/:roomId/messages` 写入 `intent_result`。
- `chat` 不创建任务。
- `light_task` 创建/追加任务。
- `debugger` 进入 debug 类型流程入口。
- `brainstorming` 阻止直接实现，要求设计确认。
- `workflow` 进入完整闭环入口。

### 12.3 浏览器测试

完成实现后必须在真实浏览器中验证：

- 在群聊发送普通聊天消息。
- 发送轻量任务消息。
- 发送 debug 消息。
- 发送 brainstorming 消息。
- 发送 workflow 消息。
- 验证确认卡片、任务栏、任务详情、workflow 进度、UI 交互都符合预期。

---

## 13. 验收标准

1. 用户只发送自然语言消息，系统能自动分类。
2. 分类结果可见、可解释、可纠正。
3. 普通聊天不会误创建任务。
4. Brainstorming 不会绕过设计确认直接进入实现。
5. Debugger 不会跳过排查直接修改代码。
6. Workflow 能进入完整闭环，并继承代码审查、浏览器测试、完成前验证要求。
7. 分类结果写入 message metadata，可被前端和后续流程读取。
8. ACP / Superpowers 被用于分类辅助和后续执行，但不作为入口唯一事实源。

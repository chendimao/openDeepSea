# 会话与历史记录新模型设计

## 背景

当前产品以项目群聊、任务列表、任务动作和 agent run 叠加组成主工作流。这个模型在轻量聊天时可用，但在真实 coding agent 场景下会出现几个明显问题：

1. 用户把页面理解为“群聊”，系统却在后台自动创建任务、路由阶段、派发 agent，心智模型不一致。
2. 任务列表既承担待办、执行阶段、历史记录、agent evidence 展示，多种职责混在一起。
3. `brainstorming`、`writing-plans`、执行、review、verification 被暴露成任务动作按钮，用户需要理解内部 Superpowers 阶段。
4. `compact`、`resume`、上下文来源、工具调用证据、未提交 diff 和验证状态缺少统一控制面。
5. 社区中的 Codex、Claude Code、OpenCode 等 CLI agent 工具已经证明终端执行体验足够强，但普遍存在会话状态、历史恢复、上下文治理、审计和多 session 协作薄弱的问题。

本设计将 OpenClaw 从“项目群聊 + 多任务”改为“项目级 Agent Session OS”。产品不复制 CLI 的终端执行体验，而是做 CLI 上层缺失的会话资产化、历史治理、上下文透明、多 agent 可视化、恢复与审计。

## 目标

- 直接引入新的 `sessions` 与 `history_records` 数据模型，不保留旧 `rooms/tasks` 工作流兼容。
- 将当前项目工作方式改为 Codex/Claude Code CLI 类似的会话流：`/new` 切分历史，`/compact` 压缩上下文，`/resume` 恢复历史，`/fork` 分叉试验。
- 将当前任务列表替换为历史记录列表。两次 `/new` 之间的对话、工具调用、文件变化、验证、commit 和 compact 结果固化为一条历史记录。
- 将任务阶段降级为 session 内 checklist / plan items，不再作为主实体驱动页面。
- 建立 Session Control Layer，统一承载 `/status`、Context Inspector、Compact Preview、Resume Brief、Fork Session 和 Evidence Timeline。
- 使用 `frontend-design` 与 `ui-ux-pro-max` 重新定义 UI 风格、布局、组件体系和交互密度；不依赖旧 `index.css`、`shell-*`、`chat-*`、`task-*` 样式体系。
- 补齐 CLI 社区常见痛点：compact/resume 丢上下文、历史索引不可靠、上下文和成本不可见、工具调用缺少审计、多 session/worktree 冲突、subagent 交接弱、provider 配置碎片化。

## 非目标

- 不保留旧的 room/task 工作流兼容路径。
- 不在新模型中继续把“群聊”作为主抽象。
- 不把旧任务列表简单改名为历史记录。
- 不继续把 `brainstorming`、`writing-plans`、执行、review 等 Superpowers 阶段暴露成主按钮。
- 不复用旧群聊/任务工作区的视觉语言、布局骨架或 CSS class 命名。
- 不在第一版实现所有高级恢复能力，例如完整文件系统 checkpoint restore，可先记录 checkpoint 元数据和 git 状态。
- 不重新实现 Codex、Claude Code、OpenCode 的底层 CLI 能力；OpenClaw 通过 adapter 调用并治理它们。

## 核心概念

### Session

`session` 是项目中的一次连续 agent 工作上下文，等价于 CLI 中一个可持续推进的 thread。

Session 负责保存：

- 当前目标
- 当前 mode
- 当前 phase
- 最近消息
- agent runs
- compact 摘要
- checklist
- evidence timeline
- 当前上下文 manifest
- 与 git/worktree/provider 的绑定关系

### History Record

`history_record` 是一个已经固化的 session 段。默认由 `/new` 生成，也可以由用户手动归档。

History Record 展示：

- 标题
- 摘要
- 状态
- 时间范围
- 关键决策
- 文件改动
- 最近验证
- commit refs
- compact 次数
- resume brief
- fork 来源与分支关系

### Session Mode

Session mode 替代旧的群聊意图自动建任务机制：

- `ask`：只读问答、解释、分析。
- `plan`：方案设计、头脑风暴、写 spec/plan。
- `code`：实现、重构、文档修改。
- `debug`：bug 排查、测试失败分析。
- `review`：代码审查、完成前验证、收尾。

Mode 可以由用户显式设置，也可以由系统建议，但不应偷偷改变主工作流。

### Session Phase

Phase 是 session 内部状态，不直接等于主 UI 任务按钮：

- `idle`
- `brainstorming`
- `planning`
- `implementing`
- `debugging`
- `reviewing`
- `verifying`
- `blocked`
- `completed`
- `archived`

### Evidence Event

`session_evidence_events` 是 append-only 事件流，替代旧 task events 的主地位。

每个 evidence event 都要能回答：

- 发生了什么
- 谁触发的
- 来源 run 是什么
- 影响了哪些文件
- 是否通过验证
- 是否可重试、恢复或展开原始输出

## 数据模型

### sessions

```sql
CREATE TABLE sessions (
  id TEXT PRIMARY KEY,
  project_id TEXT NOT NULL,
  title TEXT NOT NULL,
  current_goal TEXT,
  mode TEXT NOT NULL CHECK (mode IN ('ask', 'plan', 'code', 'debug', 'review')),
  phase TEXT NOT NULL,
  status TEXT NOT NULL CHECK (status IN ('active', 'blocked', 'completed', 'archived', 'failed')),
  provider TEXT,
  model TEXT,
  workspace_path TEXT,
  worktree_path TEXT,
  branch_name TEXT,
  forked_from_session_id TEXT,
  forked_from_history_record_id TEXT,
  latest_compaction_id TEXT,
  latest_context_manifest_id TEXT,
  created_at INTEGER NOT NULL,
  updated_at INTEGER NOT NULL,
  archived_at INTEGER,
  FOREIGN KEY (project_id) REFERENCES projects(id) ON DELETE CASCADE,
  FOREIGN KEY (forked_from_session_id) REFERENCES sessions(id) ON DELETE SET NULL
);
```

关键索引：

- `(project_id, status, updated_at DESC)`
- `(project_id, archived_at DESC)`
- `(forked_from_session_id)`

### session_messages

```sql
CREATE TABLE session_messages (
  id TEXT PRIMARY KEY,

  role TEXT NOT NULL CHECK (role IN ('user', 'assistant', 'system')),

  status TEXT NOT NULL DEFAULT 'completed',
  metadata TEXT,
  created_at INTEGER NOT NULL,

```

## 命令设计

### /new

`/new` 结束当前 session 段，生成 history record，并开启新 session。

流程：

1. 检查当前 session 是否存在未终态 run。
2. 生成 session summary。
3. 收集 changed files、最近验证、commit refs、blockers、未完成 plan items。
4. 生成 resume brief。
5. 写入 `history_records`。
6. 写入 `session_evidence_events`：`new`。
7. 创建新的 active session。

可选参数：

```text
/new
/new blank
/new inherit-summary
/new title: 重构会话模型
```

### /compact

`/compact` 不结束 session，只生成 compact preview。用户确认 apply 后才影响后续 prompt。

流程：

1. 生成 `preview_summary`。
2. 展示 Keep / Drop / Risk。
3. 用户可编辑 summary。
4. Apply 后写入 `applied_summary`，更新 `sessions.latest_compaction_id`。
5. 写入 evidence event：`compact`。

可选参数：

```text
/compact
/compact focus: 保留 UI 决策和未完成 bug
/compact aggressive
/compact conservative
```

### /status

`/status` 输出 session 驾驶舱。

必须展示：

- 当前目标
- mode / phase / status
- context token 估算
- latest compact
- 未提交 diff
- 最近验证
- 当前 blocker
- 下个建议动作
- provider / model / permission profile
- active run 状态

### /context

`/context` 打开 Context Inspector。

必须展示：

- 本轮 prompt 实际来源
- included / excluded source
- token estimate
- inclusion reason
- priority
- excerpt
- content hash

### /resume

`/resume <history_record_id>` 从历史记录恢复 session。

恢复前必须展示 resume brief，并要求模型先对齐当前目标，不允许直接执行无关修改。

### /fork

`/fork` 从当前 session、历史记录或 checkpoint 分叉新 session。

可选参数：

```text
/fork current
/fork history:<id>
/fork checkpoint:<id>
/fork compact-only
```

### /checkpoint

记录当前 git head、diff summary、关键 evidence，供恢复和 fork 使用。

## Session Control Layer

### 1. /status

`/status` 不是普通消息，而是可复用的 session snapshot。

后端生成 `StatusSnapshot`：

```ts
interface StatusSnapshot {
  goal: string | null;
  mode: SessionMode;
  phase: SessionPhase;
  status: SessionStatus;
  context: {
    totalTokenEstimate: number;
    latestCompactionId: string | null;
    retainedRecentMessages: number;
    pressure: 'low' | 'medium' | 'high';
  };
  git: {
    branchName: string | null;
    changedFileCount: number;
    hasUncommittedDiff: boolean;
    conflictRisk: 'none' | 'low' | 'high';
  };
  verification: {
    lastCommand: string | null;
    status: 'passed' | 'failed' | 'unknown';
    completedAt: number | null;
  };
  blocker: {
    reason: string;
    since: number;
    requiredAction: string;
  } | null;
  nextAction: {
    label: string;
    command: string | null;
    reason: string;
  };
}
```

### 2. Context Inspector

Context Inspector 每次 agent run 前生成，不依赖用户手动打开。

展示维度：

- AGENTS / RTK
- compact summary
- history record
- memory
- recent messages
- file snippets
- git diff
- tool results
- user latest request

交互能力：

- 下轮强制包含
- 下轮排除
- 查看 excerpt
- 查看 hash
- 复制 manifest
- 生成“为什么模型不知道 X”的诊断

### 3. Compact Preview

Compact Preview 必须包含：

```ts
interface CompactPreview {
  keep: Array<{ title: string; reason: string; sourceRefs: string[] }>;
  drop: Array<{ title: string; reason: string; sourceRefs: string[] }>;
  risks: string[];
  previewSummary: string;
  suggestedStrategy: 'aggressive' | 'conservative';
}
```

Compact 后必须允许编辑摘要，并记录 `user_edited=1`。

### 4. Resume Brief

Resume Brief 是历史记录一键恢复的启动提示词。

模板：

```text
你正在恢复 OpenClaw session：<title>

目标：
<goal>

已完成：
<completed>

未完成：
<remaining>

关键决策：
<decisions>

当前文件状态：
<changed_files>

最近验证：
<verification>

阻塞 / 风险：
<blockers>

建议下一步：
<next_action>

优先读取文件：
<files>
```

### 5. Fork Session

Fork Session 用于试验方案和多 provider 对比。

Fork 必须记录来源：

- `forked_from_session_id`
- `forked_from_history_record_id`
- base compaction
- inherited context sources
- branch/worktree 绑定

Fork 类型：

- current
- history
- checkpoint
- compact-only

### 6. Evidence Timeline

Evidence Timeline 是新模型主视图，替代旧任务时间线和任务详情。

过滤器：

- All
- Decisions
- Files
- Verification
- Errors
- Commits
- Compactions
- Tool Calls
- Reviews

每个事件支持：

- 展开原始 payload
- 定位相关 message/run/file
- 重试可重试动作
- 生成摘要
- 标记为关键决策

## 社区痛点映射

### Compact / Resume 丢上下文

问题：

- compact 后忘记目标、约束、当前任务。
- resume 后 active task pointer 丢失。
- 摘要不可见、不可编辑、不可指定重点。

OpenClaw 补强：

- Current Objective Contract：目标、约束、下一步单独持久化。
- Compact Preview：保留/丢弃/风险可见。
- Resume Guard：恢复后先对齐目标再执行。
- Anti-drift Detector：发现行为偏离目标时提示确认。

### 历史记录索引不可靠

问题：

- transcript 损坏导致 session 消失。
- resume 列表漏 session。
- 历史只显示最后消息，无法快速判断内容。

OpenClaw 补强：

- append-only evidence events。
- Session Health Check。
- Rebuild Index。
- Export Session Bundle。

### 上下文和成本不可见

问题：

- context bloat。
- auto compact 阈值不清。
- prompt cache 重建成本不可见。

OpenClaw 补强：

- `/status` 显示上下文压力。
- Context Inspector 显示来源和 token 占比。
- Cost Timeline 记录 provider、model、输入、输出、cache 命中。
- Context Policy 按 mode 选择 compact 策略。

### 工具调用缺少审计

问题：

- 用户不知道 agent 读了哪些文件、执行了哪些命令。
- 危险操作和越界访问不够显眼。

OpenClaw 补强：

- Evidence Timeline 默认展示 tool call、file read、file diff、test、commit。
- File Access Log。
- Suspicious Action Flag。
- 每个 tool call 记录 reason、scope、result。

### 多 session / worktree 并行冲突

问题：

- 多个 CLI session 触碰同一文件。
- worktree / detached HEAD / branch 关系不清。
- session 被另一个 session 的状态打断。

OpenClaw 补强：

- Workspace Lock Manager。
- Parallel Lanes。
- Conflict Radar。
- Branch Attach Wizard。

### Subagent 交接弱

问题：

- subagent 上下文不足。
- 主线程口头转述遗漏约束。
- 子任务结果缺少结构化证据。

OpenClaw 补强：

- Subagent Brief Builder。
- Scoped Context Pack。
- Agent Handoff Record。
- Cross-agent Review。

### Provider 配置碎片化

问题：

- Codex、Claude Code、OpenCode 的权限、模型、上下文、resume 行为不同。
- 用户难以判断当前 provider 是否可用。

OpenClaw 补强：

- Provider Health Matrix。
- Permission Profile。
- Run Replay。
- Provider-specific Caveats。

## UI 风格与前端信息架构

### 设计原则

新 UI 采用 **Session Operations Console** 方向：它是项目级 agent 会话控制台，不是聊天软件，也不是任务看板。

设计目标：

- 高信息密度，但主次清晰。
- 当前目标、mode、阻塞、下个动作必须一眼可见。
- 所有工具调用、文件 diff、验证和 commit 都可审计。
- Compact、resume、fork、context inclusion 这些关键动作不能隐藏在消息流中。
- 视觉风格服务长期开发工作：安静、锋利、可扫描、低装饰。
- 桌面优先支持三栏操作台；移动端降级为 History drawer、Session main、Inspector bottom sheet。

### 视觉方向

推荐视觉基调：

- 风格：industrial editorial console，偏 CLI 控制台与项目审计台结合。
- 背景：warm white / graphite / ink black，避免玻璃拟态和旧有渐变装饰。
- 强调色：
  - signal cyan：当前运行、上下文、选中态。
  - verification green：验证通过、commit 成功。
  - warning amber：compact 风险、上下文压力、等待用户。
  - danger red：失败、危险命令、越界操作。
- 字体：
  - `IBM Plex Sans`：正文、导航、表单、面板标题。
  - `JetBrains Mono`：命令、session id、token、diff、工具输出、状态数字。
- 圆角：默认 4px，最大 6px；禁止 12px 以上大圆角卡片。
- 分隔：1px rule、细网格、状态色竖线和左侧 accent bar。
- 阴影：只用于浮层与 active focus，主布局不用重阴影。
- 动效：150-220ms，用于面板切换、行展开、compact preview apply，不做装饰动画。

### 样式隔离

新 UI 不依赖原来的全局样式文件。

实现要求：

### 顶部 Command Bar

Command Bar 是 session OS 的主操作区，不是传统站点导航。

内容：

- 项目名和 workspace path。
- 当前 session title。
- provider/model/permission profile。
- 主命令：New、Compact、Fork、Resume、Status、Context。
- 全局搜索：history、files、evidence、commands。
- context pressure indicator。
- active run indicator。

交互要求：

- 所有 icon-only 按钮必须有 tooltip 和 aria-label。
- 主命令按钮高度不低于 44px。
- 快捷键提示使用 mono，小尺寸但不可低于 12px。
- 当前危险状态，例如未提交 diff + fork，必须在 Command Bar 显示 warning indicator。

### 左侧：History Records

展示：

- title
- summary
- status
- mode
- 时间范围
- compact count
- changed files count
- verification status
- commit count
- fork count
- provider/model badge
- health status

操作：

- Resume
- Fork
- Copy Resume Brief
- Export Bundle
- Rebuild Index

布局：

- 顶部固定 search + filters。
- 列表项采用密集 row，不做大卡片。
- 每条记录左侧使用 3px status rail。
- title 一行，summary 最多两行。
- metadata 用 mono pill：`code`、`debug`、`3 files`、`tests passed`。
- 当前 active session 对应的历史来源必须高亮。

空状态：

- 不使用营销式大插画。
- 显示 3 个可执行入口：Start Session、Import Bundle、Open Project Context。

### 中间：Active Session

展示：

- session messages
- agent streaming
- plan items
- command composer
- compact preview dialog
- objective contract
- active run banner
- inline tool call groups

中间区域按以下顺序组织：

1. Objective Contract：当前目标、边界、风险、next action。
2. Active Run Banner：provider、model、status、elapsed、stop/retry。
3. Transcript：用户消息、assistant 输出、tool group、diff summary、verification summary。
4. Session Plan Items：只显示当前会话 checklist，不再模拟任务列表。
5. Composer：命令输入区。

Transcript 规则：

- 用户消息使用明确的 request block。
- Assistant 回复使用正文 + evidence summary，不把 JSON 原样大面积展示。
- Tool calls 默认折叠为 group，展示命令、状态、耗时、影响文件。
- 原始 stdout/stderr 折叠，超过 200 行必须虚拟滚动或分页。
- 文件 diff 只展示摘要和关键 hunks，支持跳转到 Files inspector。

Composer 规则：

- 视觉上是 command input，不是普通聊天 textarea。
- 支持 slash commands：`/new`、`/compact`、`/status`、`/context`、`/fork`、`/resume`、`/checkpoint`。
- 支持 file chips、history chips、context source chips。
- 输入区必须显示当前发送 mode，例如 `mode: code`。
- 发送前如果有高风险状态，例如未应用 compact preview，应给出 inline warning。

交互：

过滤：

- Tool Calls

每个 evidence row 展示：

桌面：

- 三栏常驻。

- 默认只显示 Active Session。
- History 是左侧 drawer。

- Composer 不遮挡最后一条消息。

- Compact Preview、Fork、Resume 等关键流程必须有 Escape 退出和明确取消按钮。

## Runtime Prompt 组装

每次 session run 前必须生成 context manifest：

1. 加载系统和项目规则。
2. 加载 AGENTS / RTK。
3. 加载 latest applied compact。
4. 加载当前目标和 session mode。
5. 加载最近 N 条消息。
6. 加载相关 history / memory。
7. 加载用户显式引用文件。
8. 加载必要 git diff。
9. 估算 token。
10. 写入 `session_context_manifests` 和 `session_context_sources`。

如果 token 压力为 high，系统应提示 compact，而不是静默丢上下文。

## 迁移策略

因为本设计明确不保留旧兼容，迁移采用硬切换：

1. 新增 session/history schema。
2. 新增 session runtime 和 API。
3. 前端入口改为 session workspace。
4. 旧 room/task 页面从主导航移除。
5. 旧数据可选只读导出，不进入新 runtime。
6. 删除或归档旧任务工作区代码。
7. 后续清理旧 API、旧 repos、旧 task actions。

第一版不要求自动把旧 rooms/tasks 全量迁移为新 history_records。若需要保留历史，可提供一次性导出脚本。

## 验收标准

- 新项目工作台默认进入 active session，不再进入群聊任务工作区。
- 用户发送消息不会自动创建旧 task。
- `/new` 能生成 history record，并开启新 session。
- `/status` 能展示目标、mode、上下文压力、未提交 diff、最近验证、blocker 和 next action。
- `/compact` 默认先展示 preview，不会静默应用。
- compact apply 后，下一次 run 的 context manifest 包含 applied summary。
- History record 支持 resume brief。
- Fork session 能记录来源，并不污染原 history record。
- Evidence Timeline 能展示 message、tool call、file diff、test、commit、review 和 compact。
- 新 workflow 不依赖旧 `rooms/tasks/task_events` 作为主状态源。
- 新 Session OS UI 不依赖旧 `index.css` 视觉 token，不复用旧 `shell-*`、`chat-*`、`task-*`、`workspace-*` class。
- 第一屏同时展示 History Records、Active Session 和 Status Inspector。
- `/status`、Context Inspector、Compact Preview、Resume Brief、Fork Session 和 Evidence Timeline 均有明确主入口与响应式降级方案。
- 桌面无嵌套卡片式任务看板；移动端无横向滚动，主要操作 touch target 不低于 44px。

## 风险与约束

- 硬切换会导致旧页面和旧数据不可直接继续使用，需要明确用户预期。
- 新模型涉及后端 schema、runtime prompt、前端信息架构、provider adapter，必须分期实现。
- 完全重设 UI 会使旧组件无法直接渐进复用，需要控制第一期交付范围，避免一次性改造过大。
- 新样式隔离要求会增加短期重复组件，但能避免旧群聊/任务工作区视觉债继续扩散。
- Context token 估算不可能完全准确，应标注为 estimate。
- Compact summary 可编辑会引入用户误删关键上下文的风险，需要保留 preview 原始版本。
- Fork session 若绑定 worktree，需要明确文件锁和 branch 策略。
- Evidence Timeline 数据量会增长，需要分页、过滤和摘要缓存。

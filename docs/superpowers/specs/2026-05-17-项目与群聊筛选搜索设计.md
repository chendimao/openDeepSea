# 项目与群聊筛选搜索设计

## 背景

当前项目列表页面展示全部项目，项目详情页展示该项目下全部群聊。用户需要：

- 在项目列表页按项目名称筛选项目。
- 在项目详情页筛选群聊，支持按群聊名称和描述筛选。
- 在群聊筛选中支持自然语言搜索，例如输入“修复页面显示不完整的bug的群聊”，能够找出聊天消息或任务中提到“页面显示不完整 bug”的群聊。

现有前端页面：

- `packages/frontend/src/pages/DashboardPage.tsx`：项目列表。
- `packages/frontend/src/pages/ProjectPage.tsx`：项目详情与群聊列表。

现有后端数据：

- `rooms`：群聊名称、描述。
- `messages`：群聊消息内容。
- `tasks`：任务标题、描述。
- 已有模型配置与调用入口：`invokeConfiguredModelText`。

## 目标

1. 项目列表页提供轻量项目名称筛选。
2. 群聊列表页提供统一搜索框，搜索范围包括群聊名称、描述、聊天消息、任务标题和任务描述。
3. 群聊自然语言搜索优先使用模型语义搜索。
4. 模型不可用、调用失败或超时时，自动降级为本地关键词搜索。
5. UI 使用紧凑内联筛选形态，保持当前页面信息密度。

## 非目标

- 第一版不做 embedding 向量索引。
- 第一版不新增 SQLite FTS 同步表。
- 第一版不引入高级筛选抽屉、复杂排序配置或多条件组合筛选。
- 第一版不修改消息、任务、记忆等既有创建流程。

## 用户体验设计

### 项目列表筛选

在“所有项目”标题行右侧增加搜索输入框：

- 占位文案：`按项目名称筛选`。
- 输入为空时显示全部项目。
- 输入非空时，仅按项目名称进行前端本地筛选。
- 数量展示改为筛选结果数量与总数，例如 `3 / 12 个项目`。
- 无匹配时显示轻量空状态，提示没有匹配项目。

项目列表数据量预期较小，前端本地筛选足够，避免新增不必要接口。

### 群聊列表筛选

在“群聊”标题行中增加紧凑搜索输入框：

- 占位文案：`搜索名称、描述、消息或任务`。
- 输入为空时显示完整群聊列表。
- 输入非空时调用后端群聊搜索接口。
- 搜索结果数量展示为 `匹配数 / 总数`。
- 搜索中显示轻量 loading 状态。
- 模型降级时显示轻提示，例如 `语义搜索不可用，已使用关键词搜索`。
- 无匹配时显示轻量空状态，提示调整搜索词。

群聊搜索保持紧凑布局，但允许在列表上方显示一行状态提示，用于承载语义搜索、降级和错误信息。

## 技术方案

### 推荐方案：LLM rerank 候选群聊

后端新增项目级群聊搜索接口，例如：

```http
GET /api/projects/:projectId/rooms/search?q=...
```

接口职责：

1. 校验项目存在。
2. 查询该项目下所有群聊。
3. 为每个群聊聚合候选文本：
   - 群聊名称。
   - 群聊描述。
   - 最近或截断后的聊天消息内容。
   - 任务标题。
   - 任务描述。
4. 如果模型配置可用：
   - 调用 `invokeConfiguredModelText`。
   - 让模型根据用户查询对候选群聊打分并排序。
   - 只返回明确相关的群聊。
5. 如果模型不可用、模型返回不可解析、调用失败或超时：
   - 自动走关键词搜索。
   - 返回 `mode: 'keyword'` 和降级原因。

返回结构建议：

```ts
interface RoomSearchResponse {
  query: string;
  mode: 'semantic' | 'keyword';
  degraded: boolean;
  degradationReason: string | null;
  total: number;
  results: RoomSearchResult[];
}

interface RoomSearchResult {
  room: Room;
  score: number;
  matchedFields: Array<'room_name' | 'room_description' | 'message' | 'task_title' | 'task_description'>;
  highlights: string[];
}
```

### 模型提示词策略

模型输入应是受控 JSON，而不是长篇自然语言拼接。每个候选群聊提供：

- `id`
- `name`
- `description`
- `messages`：截断后的代表性消息片段。
- `tasks`：任务标题和描述片段。

模型输出只允许 JSON，例如：

```json
{
  "results": [
    {
      "roomId": "room-id",
      "score": 0.92,
      "reason": "聊天消息提到页面显示不完整 bug"
    }
  ]
}
```

后端只接受已存在的 `roomId`，忽略未知 ID、非法分数和不可解析字段，避免模型输出影响数据边界。

### 关键词降级策略

关键词搜索用于两类场景：

- 模型不可用、失败或超时。
- 模型返回空结果时，可作为兜底提供弱匹配结果。

关键词提取规则：

- 对中文、英文、数字做基础归一化。
- 去除常见助词和低信息词，例如“的”“了”“群聊”“修复”等可配置停用词。
- 保留关键名词、动词和技术词，例如“页面”“显示”“不完整”“bug”。
- 对群聊名称、描述、消息、任务标题、任务描述做大小写不敏感匹配。

排序优先级：

1. 群聊名称命中。
2. 群聊描述命中。
3. 任务标题命中。
4. 任务描述命中。
5. 消息内容命中。
6. 最近更新时间或创建时间作为同分排序。

## 数据与性能边界

第一版应限制候选文本大小，避免一次搜索把全部历史消息塞给模型：

- 每个群聊最多取最近若干条消息，或总字符数上限。
- 每个群聊最多取若干个任务。
- 每个候选群聊文本总长度设置上限。
- 项目群聊数量较多时，可先用关键词粗筛出候选，再交给模型 rerank。

建议第一版使用“关键词粗筛 + 模型 rerank”的混合流程：

1. 若关键词粗筛有结果，只把粗筛候选交给模型。
2. 若粗筛无结果，但查询非空，可把最近活跃的一批群聊交给模型判断。
3. 模型超时后直接返回关键词结果。

## 前端数据流

### 项目列表

`DashboardPage.tsx`：

- 新增本地 state：`projectQuery`。
- 使用 `useMemo` 计算 `filteredProjects`。
- 列表渲染改为 `filteredProjects.map(...)`。
- 标题数量使用筛选数量和总数。

### 群聊列表

`ProjectPage.tsx`：

- 新增本地 state：`roomQuery`。
- `roomQuery` 为空时使用现有 `api.listRooms(projectId)` 结果。
- `roomQuery` 非空时调用新增 `api.searchRooms(projectId, { query: roomQuery })`。
- 为输入增加 debounce，避免每个字符立即请求。
- 搜索结果优先展示接口返回顺序。
- 删除群聊、新建群聊成功后，同时刷新普通群聊列表和搜索结果。

## API 与类型扩展

前端 `api.ts` 增加：

```ts
searchRooms: (projectId: string, input: { query: string }) =>
  request<RoomSearchResponse>(`/projects/${projectId}/rooms/search?q=${encodeURIComponent(input.query)}`)
```

前后端类型增加 `RoomSearchResponse` 与 `RoomSearchResult`。若项目继续保持前后端类型分离，需要分别在：

- `packages/backend/src/types.ts`
- `packages/frontend/src/lib/types.ts`

中维护对应类型。

## 错误处理

- 项目不存在：返回 404。
- 查询为空：前端不调用搜索接口；后端若收到空查询可返回空结果或 400，推荐返回 400 以保持调用方明确。
- 模型未配置：不返回 500，自动降级关键词搜索。
- 模型超时或异常：记录已脱敏错误，返回关键词结果和降级状态。
- 模型输出不可解析：忽略模型结果，返回关键词结果和降级状态。

### 后端测试

新增或扩展 routes/repo 测试：

- 搜索接口能匹配群聊名称。
- 搜索接口能匹配群聊描述。
- 搜索接口能匹配消息内容。
- 搜索接口能匹配任务标题和任务描述。
- 模型未配置时自动降级关键词搜索。
- 模型返回非法 JSON 时自动降级。
- 搜索不会返回其他项目的群聊。

### 前端验证

- 项目名称筛选输入后只显示匹配项目。
- 群聊搜索输入后显示搜索结果和数量。
- 降级状态能显示轻提示。
- 输入清空后恢复完整群聊列表。
- 移动端标题行和搜索框不发生文本溢出或按钮遮挡。

## 风险与权衡

- LLM rerank 不等于稳定全文检索。模型可能漏判或误判，因此必须保留关键词降级和受控输出解析。
- 大量消息会增加模型调用成本和延迟，因此必须限制候选文本长度。
- 关键词粗筛可能漏掉纯语义相关内容。第一版接受该权衡，后续可升级 embedding 或 FTS。
- 复用现有模型配置可以减少设置入口，但搜索行为会依赖系统模型配置质量。

## 后续演进

如果群聊和消息规模增大，可分阶段升级：

1. 增加 SQLite FTS 表，提高关键词检索性能。
2. 增加 embedding 索引，支持更稳定的语义召回。
3. 增加搜索范围开关，例如“仅名称/描述”“包含消息”“包含任务”。
4. 增加匹配片段高亮和跳转到原消息/任务。

# 单次 ACP 流式意图分析设计

## 背景

当前群聊消息入口会先执行本地规则意图判断，并在低置信度时同步调用 ACP classifier。这个流程会阻塞 `POST /rooms/:roomId/messages` 返回，导致短消息或模糊消息在发送时等待 ACP 分类完成。

新的目标是取消“本地规则优先 + 置信度补判”的入口判断，让 ACP 在一次只读调用中同时完成用户回复和意图分析。正文保持流式输出，意图判断作为隐藏控制块在末尾返回。

## 目标

- 用户消息保存和前端发送反馈不再等待单独的意图 classifier。
- 单次 ACP 调用负责分析用户内容并生成回复正文。
- ACP 调用必须只读，不能修改文件、运行写入类操作或触发实现。
- 回复正文继续流式展示，并能和 ACP 调用记录按时间线穿插展示。
- ACP 在末尾输出隐藏 intent JSON，后端解析后写入消息 metadata。
- intent JSON 不展示给用户。

## 非目标

- 不在本次设计中引入第二次 ACP 调用。
- 不要求 ACP 在回复开始前先输出 intent。
- 不改变现有 ACP timeline 的基本事件模型。
- 不重新设计任务创建、workflow 启动和任务详情页。

## 输出协议

ACP 使用同一次调用输出两类内容：

1. 可见正文：正常流式输出，按现有 agent stream message 展示。
2. 隐藏控制块：在回复末尾输出，格式固定为：

```text
<openclaw_intent_json>
{"intent":"chat","suggestedAction":"reply_in_chat","reason":"普通问候","signals":["hi"]}
</openclaw_intent_json>
```

控制块中的 JSON 字段：

- `intent`: `chat | light_task | debugger | brainstorming | workflow`
- `suggestedAction`: `reply_in_chat | create_light_task | start_debugger | start_brainstorming | start_workflow | ask_user`
- `reason`: 简短中文原因
- `signals`: 字符串数组，记录命中的关键信号

后端解析后补充或规范化字段：

- `source`: 固定为 `classifier`
- `confidence`: 可选；如果 ACP 未返回，后端可写入默认值 `1`

## 数据流

1. 前端提交群聊消息。
2. 后端立即校验、保存用户消息，并返回 `201`。
3. 后端异步派发用户消息给目标 ACP agent。
4. ACP 以 read-only 权限运行，prompt 明确要求：
   - 分析用户请求。
   - 正文可流式回复。
   - 不修改文件。
   - 不执行用户请求中的实现动作。
   - 末尾输出隐藏 intent JSON 控制块。
5. 后端接收 ACP answer chunks：
   - 普通正文 chunk 继续广播到 agent stream message。
   - 控制块内容进入内部 buffer，不展示给用户。
6. ACP 完成后，后端解析控制块 JSON。
7. 解析成功时，更新源用户消息 metadata 中的 `intent_result`，并按 intent 更新 route/task 决策。
8. 解析失败或缺失时，写入系统活动消息提示意图解析失败，但保留已流式输出的正文。

## 路由语义

- `chat`: 留在群聊，只展示 ACP 回复。
- `light_task`: 创建等待确认的轻量任务。
- `debugger`: 创建等待确认的 debugger 任务。
- `brainstorming`: 创建等待确认的 brainstorming 任务。
- `workflow`: 创建等待确认的 workflow 任务。
- `ask_user`: 写入活动提示，请用户选择后续意图。

任务类 intent 不应让 ACP 在同一次调用中直接改文件。该次 ACP 只负责解释、判断、建议下一步，并通过 intent JSON 触发后续任务流程。

## 错误处理

- ACP 调用失败：保留用户消息，写入系统活动消息说明 ACP 分析失败。
- 控制块缺失：保留正文，写入 `message_intent_parse_failed` 活动消息。
- 控制块 JSON 非法：保留正文，写入解析失败活动消息。
- 控制块出现在中途：后端仍隐藏并解析，但 prompt 要求只在末尾输出。
- 控制块被拆分到多个 chunk：后端通过状态机跨 chunk 识别。

## 测试计划

- 后端单元测试：跨 chunk 隐藏并解析 `<openclaw_intent_json>` 控制块。
- 后端路由测试：发送 `hi` 不再等待 classifier，消息立即创建。
- 后端路由测试：ACP 流式正文正常广播，控制块不进入可见正文。
- 后端路由测试：合法 intent JSON 更新用户消息 metadata。
- 后端路由测试：非法或缺失控制块写入解析失败活动消息。
- 构建验证：运行 `npm run build`。

## 风险

- 模型可能不稳定遵守隐藏控制块格式，需要后端解析容错和 prompt 强约束。
- 正文和 intent 同源于一次 ACP 回复，稳定性好于双调用，但仍依赖模型最终输出控制块。
- 如果用户内容本身包含 `<openclaw_intent_json>`，需要后端只识别 ACP 输出末尾控制块，避免正文误触发。

## 决策

采用单次 ACP 调用：正文流式输出，末尾隐藏控制块返回 intent JSON。后端负责隐藏控制块、解析 JSON、更新 metadata 和触发后续路由。

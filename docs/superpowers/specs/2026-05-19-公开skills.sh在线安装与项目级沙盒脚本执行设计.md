# 公开 skills.sh 在线安装与项目级沙盒脚本执行设计

## 背景

OpenDeepSea 当前的 `/skills` 已经支持本地导入、系统绑定、范围选择和 prompt 注入，但它本质上仍是“内部指令型 skill”。用户现在希望把它升级成真正可安装、可执行、可更新的技能系统，并且明确要接入公开的 `skills.sh` 生态。

本设计的目标不是把 skill 变成任意本机权限插件，而是把它做成一种 **可安装的脚本型能力包**：由 OpenDeepSea 安装、登记、校验，再由 workflow / agent 显式调用，在项目级沙盒中运行。

## 目标

1. 支持从公开 `skills.sh` 在线发现和安装 skill。
2. 支持 skill 包携带 JS、Python、Shell 脚本，并由 OpenDeepSea 在受限环境中执行。
3. 执行只在 workflow / agent 显式调用时发生，不做自动后台触发。
4. 默认限制在当前项目目录内运行，允许联网，但必须可审计。
5. 更新检查和更新后处理都可配置，且默认采用启动时检查、提示后更新。
6. 保留现有 prompt-only skills 的兼容能力，不破坏当前 `/skills` 管理页。

## 非目标

1. 不做私有 registry。
2. 不做跨机器同步或云端账号体系。
3. 不把 skill 直接提升为无限制本机插件。
4. 不做自动关键词后台触发。
5. 不要求第一版兼容所有社区 skill 的全部奇异格式，只要求兼容 `skills.sh` 的公开安装语义和可执行 skill 包结构。

## 总体方案

系统分成四层：

1. **Skill Source**：公开 `skills.sh`，负责发现和安装源。
2. **Skill Package**：文件夹化包，包含 `SKILL.md`、机器 manifest、脚本和资源。
3. **Skill Runtime**：由 OpenDeepSea 启动 runner，在项目级沙盒里执行脚本。
4. **Skill Orchestrator**：workflow / agent 显式调用 skill，负责传参、权限校验、结果回流和审计。

## 包格式

建议 skill 包采用以下目录结构：

```text
skill-root/
  SKILL.md
  skill.json
  scripts/
    run.sh
    main.js
    main.py
  assets/
  references/
```

### 文件职责

- `SKILL.md`：人和模型都能读的说明文档，保留现有 prompt 型 skill 的习惯。
- `skill.json`：机器可读 manifest，声明运行器、入口、权限、输入输出约束。
- `scripts/`：真正执行的脚本。
- `assets/` / `references/`：附属资源。

### Manifest 核心字段

```json
{
  "name": "example-skill",
  "version": "1.0.0",
  "description": "Short summary",
  "runtime": "node",
  "entrypoint": "scripts/main.js",
  "permissions": {
    "filesystem": "project",
    "network": true,
    "commands": ["node", "python3", "bash"]
  }
}
```

第一版建议强制要求：

- `runtime`：`node` / `python` / `shell`
- `entrypoint`：相对路径
- `permissions.filesystem`：默认 `project`
- `permissions.network`：布尔值，默认按配置允许
- `permissions.commands`：允许的命令集合

如果 skill 只有 `SKILL.md`，则仍可按现有 prompt-only skill 导入，但不能执行脚本。

## 安装与更新

### 在线安装源

第一版只接入公开 `skills.sh`。

兼容目标是它公开站点的安装语义，例如 `npx skills add owner/repo` 这一类公开安装方式。OpenDeepSea 可以在内部实现等价安装器，不要求前端一定直接调用外部 CLI。

### 安装流程

1. 在 `/skills` 页面搜索或选择 `skills.sh` 上的公开 skill。
2. 后端解析公开来源，拉取 package 元数据和文件内容。
3. 校验 package 结构、文件大小、路径边界和 manifest 合法性。
4. 复制到 OpenDeepSea 管理目录。
5. 写入数据库登记，保留来源、版本、校验和、manifest 信息。
6. 安装完成后不执行任何脚本。

### 更新策略

更新检查和更新后处理都放到设置里配置。

建议提供两组配置：

- **检查方式**：关闭 / 启动时检查 / 手动检查 / 定时检查
- **更新后处理**：仅提示 / 先下载待确认 / 自动替换

默认值：

- 启动时检查更新
- 发现更新后仅提示，不自动替换

更新只影响后续执行，不会中断正在运行的 skill。

## 执行模型

### 调用方式

skill 只在 workflow 或 agent 显式调用时执行。第一版不做自动关键词触发，也不做静默后台运行。

建议在 runtime 中引入一个统一调用入口，例如：

- `invokeSkill(skillId, input, context)`

其中 `context` 至少包含：

- `projectId`
- `roomId`
- `agentId`
- `workingDirectory`
- `allowedPaths`
- `networkPolicy`

### 运行器

第一版支持三类运行器：

- `node`
- `python`
- `shell`

运行时行为：

- 在项目级沙盒目录内启动
- 以当前项目目录作为默认工作目录
- 输入通过标准输入或临时输入文件传递
- 输出收集 stdout、stderr、exit code、耗时和结果 JSON

### 结果模型

每次执行都应产生可审计记录：

- skill id
- package version / revision
- 触发来源
- 调用者
- 参数摘要
- 工作目录
- 允许的路径
- 网络开关
- 起止时间
- exit code
- stdout / stderr 摘要
- 结构化结果

## 安全边界

### 项目级沙盒

默认只允许访问当前项目目录内的文件。禁止访问：

- 其他项目目录
- `~/.ssh`
- `~/.codex`
- 系统目录
- 未授权的上层路径

### 网络

第一版允许联网，但必须满足：

- 仅在 manifest 或全局设置允许时开放
- 运行记录中保留网络开关状态
- 不把网络权限默认为无条件自由

### 进程和命令

- 不执行安装阶段的 postinstall
- 不允许 skill 在安装阶段自修改执行环境
- shell 运行器只使用白名单命令
- 默认不透传敏感环境变量

### 文件系统约束

- 禁止 symlink 逃逸
- 禁止读取安装目录之外的未授权文件
- 禁止写入项目目录之外的路径

## 数据模型

### `skills` 表扩展

现有 `skills` 表建议增加或补充以下字段：

- `source_type` 扩展为包含 `skills_sh`
- `package_version`
- `package_revision`
- `runtime_type`
- `entrypoint`
- `permissions_json`
- `install_source_label`
- `update_check_mode`
- `update_apply_mode`
- `last_update_checked_at`
- `available_version`
- `available_revision`

### 新增 `skill_runs` 表

建议新增执行审计表：

```sql
CREATE TABLE skill_runs (
  id TEXT PRIMARY KEY,
  skill_id TEXT NOT NULL,
  project_id TEXT,
  room_id TEXT,
  agent_id TEXT,
  invoked_by TEXT NOT NULL,
  runtime TEXT NOT NULL,
  entrypoint TEXT NOT NULL,
  input_json TEXT,
  allowed_paths_json TEXT,
  network_enabled INTEGER NOT NULL,
  status TEXT NOT NULL,
  exit_code INTEGER,
  stdout TEXT,
  stderr TEXT,
  result_json TEXT,
  error TEXT,
  created_at INTEGER NOT NULL,
  updated_at INTEGER NOT NULL
);
```

## API 设计

建议在现有 `/skills` router 上扩展以下能力：

- `GET /skills/marketplace?q=`：查询公开 `skills.sh`
- `POST /skills/import/skills-sh`：安装公开 skill
- `POST /skills/:id/run`：由 workflow / agent 显式执行
- `GET /skills/runs`：查看运行记录
- `GET /skills/:id/updates`：检查更新
- `POST /skills/:id/update`：应用更新
- `PATCH /skills/settings`：更新检查策略和更新后处理策略

## 前端设计

`/skills` 页面建议分成四块：

1. **在线安装**
   - 搜索公开 `skills.sh`
   - 浏览结果
   - 安装按钮
   - 安装前展示来源、版本、权限、运行器

2. **已安装技能**
   - 显示来源、版本、运行器、权限、更新状态
   - 允许启用/禁用
   - 允许卸载

3. **运行记录**
   - 展示最近执行、退出码、耗时、摘要、错误
   - 可按 skill 过滤

4. **更新设置**
   - 检查方式
   - 更新后处理
   - 手动检查按钮

## 与现有系统的关系

- 现有 prompt-only skills 保留。
- 内部模型路径仍可继续使用 `selectSkills` + `formatSkillPrompt`。
- 执行型 skills 不替代 planner / model chat / workflow / memory 的 prompt 注入能力，而是作为新的显式动作能力。
- 外部 ACP CLI 默认仍不注入 OpenDeepSea skills。

## 验证标准

1. 能从公开 `skills.sh` 搜索并安装一个 skill。
2. 安装后能看到版本、来源、运行器和权限。
3. 启动时能检查到更新并按配置提示。
4. workflow / agent 能显式调用 skill 执行脚本。
5. skill 只能在项目级沙盒内读写。
6. 运行记录可追踪 stdout、stderr、exit code 和结果。
7. 现有 prompt-only skills 行为不回归。

## 风险与权衡

1. **联网带来供应链风险**：因此必须保留来源、版本、校验和和安装审查。
2. **脚本执行带来安全风险**：因此必须限制到项目级沙盒和受控命令集。
3. **更新策略太自动会引入不确定性**：因此默认只提示，不自动替换。
4. **包格式可能碎片化**：因此第一版只锁定公开 `skills.sh`，减少兼容面。

## 结论

第一版把 skill 明确成“公开市场可安装、项目级沙盒可执行、由 workflow / agent 显式调用、可审计可更新”的能力包。这个边界足够小，能落地；也足够强，能覆盖你想要的执行型 skill 场景。

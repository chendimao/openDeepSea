# 聊天富文本编辑器与附件上传设计

## 背景

当前聊天室发送区使用普通 textarea。它可以输入文本并通过正则识别 `@agent`，但无法把提及对象显示为一个整体，也无法在删除时把提及整体删除。用户还需要在同一输入区上传图片和文件，并在发送前预览图片、文件与文本。

本设计聚焦聊天室消息发送编辑器的首版升级：把输入区改为结构化富文本编辑器，支持整体 mention token、图片/文件预览，以及发送消息时统一上传附件。

## 目标

- `@前端开发工程师` 这类已匹配当前群聊 agent 的提及显示为整体 token。
- mention token 在 Backspace/Delete 删除时作为整体被删除。
- 用户从候选菜单选择 agent 后插入 mention token。
- 用户手打 `@agent_name` 或 `@agent_id`，若能唯一匹配当前群聊 agent，则自动转为 mention token。
- 用户可以选择、粘贴或拖入图片/文件，并在发送前预览。
- 图片和文件在点击发送时统一上传到后端，后端返回可访问路径并写入消息 metadata。
- 消息列表可以展示文本、mention 样式、图片缩略图和文件卡片。

## 非目标

- 首版不实现多人协同编辑输入框。
- 首版不解析附件内容，也不要求 agent 直接读取文件内容。
- 首版不新增附件管理页。
- 首版不把文件二进制内容写入 SQLite。
- 首版不支持任意未匹配 `@xxx` 自动成为 mention；未匹配或有歧义时保留普通文本。

## 交互设计

### 编辑器结构

编辑器显示为单个输入容器，但内部维护两类状态：

- 正文节点：`text` 和 `mention`，负责光标、输入、删除和序列化。
- 附件列表：待发送本地文件，负责预览、移除、校验和发送时上传。

mention 节点在 UI 中渲染为 chip，例如 `@前端开发工程师`。chip 不能被拆开编辑；光标在 chip 前后移动，删除键一次删除整个 chip。

### @ 输入与匹配

- 输入 `@` 后弹出当前群聊 agent 候选菜单。
- 选择候选项后插入 mention token，并在 token 后插入一个空格，便于继续输入。
- 用户手打 `@xxx` 后，当边界明确时执行匹配：
  - `xxx` 能唯一匹配某个 agent 的 `agent_name` 或 `agent_id`：替换为 mention token。
  - 无匹配：保留为普通文本。
  - 多个匹配：保留普通文本，用户可通过候选菜单明确选择。
- 发送前再做一次规范化扫描，避免粘贴文本或快速输入漏转 token。

### 附件输入

- 支持点击按钮选择文件。
- 支持粘贴剪贴板中的图片或文件。
- 支持拖拽文件到编辑器区域。
- 图片在发送前显示缩略图。
- 非图片显示文件卡片，至少包含文件名、文件大小和 MIME 类型或扩展名。
- 用户可以移除任意待发送附件。

首版限制沿用既有附件范围确认：

- 单条消息最多 5 个文件。
- 单个文件最大 10MB。
- 允许纯附件消息。
- 输入内容以 `/task` 命令开头时不允许携带附件，应阻止发送并提示先移除附件。

## 数据模型

### 前端编辑器状态

正文状态只描述文本和 mention；附件作为独立列表挂在编辑器外层，避免附件节点干扰正文光标模型。

```ts
type ComposerNode =
  | { type: 'text'; text: string }
  | {
      type: 'mention';
      roomAgentId: string;
      agentId: string;
      agentName: string;
      label: string;
    };

interface PendingAttachment {
  id: string;
  file: File;
  name: string;
  mimeType: string;
  size: number;
  isImage: boolean;
  previewUrl: string | null;
}
```

发送时从 `ComposerNode[]` 派生：

- `content`：面向消息正文和 agent prompt 的纯文本，例如 `请 @前端开发工程师 看一下`。
- `mentions`：已匹配 mention 的 `roomAgentId[]`。
- `files[]`：待上传附件。

### 后端消息 metadata

继续复用 `messages.metadata`，不新增数据库表：

```ts
interface MessageAttachmentMetadata {
  id: string;
  name: string;
  mimeType: string;
  size: number;
  url: string;
  isImage: boolean;
}

interface MessageMetadata {
  attachments?: MessageAttachmentMetadata[];
}
```

`url` 是后端暴露的受控静态路径，不使用用户原始文件名拼接。

## API 设计

### 发送消息

`POST /rooms/:roomId/messages`

支持两种请求：

- `application/json`：兼容现有纯文本消息。
- `multipart/form-data`：用于文本加附件或纯附件消息。

multipart 字段：

- `content`：字符串，可为空；当没有文本但有附件时允许为空。
- `sender_id`：可选，默认 `user`。
- `sender_name`：可选。
- `mentions`：JSON 字符串数组，元素为 `roomAgentId`。
- `files`：一个或多个文件。

后端行为：

1. 校验房间存在性和输入。
2. 校验文件数量、大小和基本文件名。
3. 使用服务端生成的安全文件名保存到受控上传目录。
4. 创建消息，并把附件信息写入 `metadata.attachments`。
5. 广播 `message:new`。
6. 根据 `content` 与显式 `mentions` 触发调度。

如果文件保存或消息创建失败，接口返回错误，不创建部分消息。

### 附件访问

后端暴露只读静态路由，例如：

`GET /uploads/messages/:storedFileName`

该路由只服务上传目录内文件，不能接受路径穿越。

## 渲染设计

### 发送区

发送区保持当前视觉风格，但输入主体从 textarea 替换为富文本编辑器区域：

- 左侧工具按钮：上传图片/文件。
- 中间编辑区域：普通文本、mention chip 和光标。
- 下方附件预览条：图片缩略图和文件卡片。
- 右侧发送按钮：文本为空但有附件时仍可用。

### 消息列表

消息正文仍使用 `MessageContent` 渲染文本与 mention 高亮。附件从 `metadata.attachments` 渲染：

- 图片：缩略图，可点击打开后端文件 URL。
- 文件：文件卡片，包含文件名、大小和类型，可点击下载或打开。

旧消息没有 metadata 或 metadata 解析失败时正常退化为纯文本。

## 错误处理

- 文件数量超过限制：阻止新增，并提示最多 5 个文件。
- 单文件超过 10MB：阻止新增，并提示文件过大。
- `/task` 命令携带附件：阻止发送，并提示先移除附件。
- 上传或保存失败：保留编辑器内容和附件预览，提示失败原因。
- 消息发送成功：清空文本节点和附件，并释放图片 `objectURL`。
- 组件卸载或附件移除：释放对应图片 `objectURL`。

## 测试与验证

最低验证：

- `npm run build`
- 手动验证候选菜单插入 mention token。
- 手动验证手打当前群聊 agent 名称自动转 token。
- 手动验证 Backspace/Delete 一次删除整个 mention token。
- 手动验证粘贴图片并显示预览。
- 手动验证选择或拖入普通文件并显示文件卡片。
- 手动验证移除附件会更新发送状态。
- 手动验证纯附件消息可发送。
- 手动验证文本加附件消息写入 `metadata.attachments`。
- 手动验证超过数量、超过大小和 `/task` 带附件的错误提示。

## 实现边界

- 首版优先使用本地轻量 `contenteditable` 封装，不引入 TipTap、Slate 或 Lexical。
- 编辑器组件应从 `RoomPage.tsx` 中拆出，避免继续扩大页面文件。
- 后端上传逻辑应集中在路由附近或独立 helper，避免把文件保存细节散落到消息仓储。
- 不修改调度器的核心 prompt 语义；附件首版仅作为消息 metadata 展示，不进入 agent 文件读取流程。

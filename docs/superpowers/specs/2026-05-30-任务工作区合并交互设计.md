# 任务工作区合并交互设计

- 状态：设计草案（已确认方向 B：当前任务驾驶舱，待书面规格评审）
- 日期：2026-05-30
- 范围：将房间页左侧任务栏与右侧任务详情合并为一个任务工作区，重新组织任务队列、当前任务详情、事件层、执行会话和任务操作
- 非范围：不新增数据库字段、不修改 TaskEvent / TaskRouter / TaskExecutor 后端契约、不引入新的任务状态机、不重写聊天消息模型
- 关联总纲：`docs/superpowers/specs/2026-05-30-ai-task-os-总体架构-design.md`

---

## 1. 背景

当前房间页已经具备 AI Task OS 的关键地基：任务是一等对象，消息可绑定任务，任务事件可按层投影，任务执行会话可展示，群聊可通过激活任务影响路由。

但前端体验仍然是分裂的三块：

```txt
左侧 TaskBoard     中间 Chat / Files     右侧 TaskDetailPanel
任务列表/筛选/活动  群聊命令入口           当前任务详情/事件/日志/执行会话
```

这个布局的问题是：

1. 任务对象被拆成“列表”和“详情”两个空间，用户需要跨列理解一个任务的完整状态。
2. 中间聊天被两侧任务 UI 挤压，尤其在存在右侧详情时可读宽度下降。
3. 当前任务优先级不够明确，用户无法一眼判断“现在群聊消息会落到哪个任务”。
4. M1-M4 的闭环能力已经散落在 UI 中，但没有形成一条连续工作流。

本设计选择方向 B：**当前任务驾驶舱**。任务工作区成为左侧一体化面板，中间群聊保留为命令入口，右侧 inspector 默认释放给记忆、设置或未来临时面板。

---

## 2. 目标

### 2.1 用户目标

- 一眼看清当前激活任务、状态、负责人、执行会话和最近事件。
- 在同一个任务工作区内完成任务选择、任务状态推进、查看计划、查看时间线、查看变更、查看日志。
- 在群聊中发送任务消息时，明确知道消息会进入当前激活任务、显式 @task，还是全局聊天。
- 支撑 M1 到 M4 的完整闭环测试：创建/路由任务、激活任务、发送任务消息、观察事件流、启动执行、查看执行会话、推进到完成。

### 2.2 产品目标

- 强化“Task 是系统本体，Chat 是命令入口”的信息架构。
- 把现有 `TaskBoard` 和 `TaskDetailPanel` 的能力合并为一个更高密度、更可扫描的任务控制面板。
- 默认主题保持极简风格：低装饰、细边框、紧凑密度、清晰 active 状态。
- 为后续 M4 TaskExecutor 并发隔离留出稳定展示位置。

### 2.3 工程目标

- 优先复用现有数据源：`tasks`、`room-task-events`、`task-executors`、`workflows`、`agent-runs`。
- 优先复用现有投影逻辑：`filterRootTasks`、`selectActivityEvents`、`selectTaskDetailEvents`。
- 通过新组件合并 UI 职责，而不是修改后端模型。
- 保持 `npm run build` 为最低验证门禁，实施阶段补充浏览器真实操作测试。

---

## 3. 非目标

- 不把聊天流挪进任务面板；群聊仍是中间主工作区。
- 不删除文件面板、记忆面板、Agent 配置面板。
- 不改变消息路由优先级：显式 @task > 当前激活任务 > 语义匹配 > 自动建任务 / ask_user。
- 不把 Runtime 日志重新塞回聊天流；事件仍按 Activity / Timeline / Runtime / Diff 分层。
- 不在本设计阶段实现代码。实现需在本规格评审通过后进入 writing-plans。

---

## 4. 推荐方案

### 4.1 方案选择

采用 **当前任务驾驶舱**：

```txt
App Sidebar | TaskWorkspacePanel | Chat / Files Workbench | Optional Utility Panel
```

其中 `TaskWorkspacePanel` 由两个内部区域组成：

```txt
TaskWorkspacePanel
├── Queue Rail             任务队列、筛选、数量、最近活动入口
└── Active Task Surface    当前任务头部、操作、详情、事件视图、执行会话
```

这不是简单把右侧详情搬到左侧，而是重新组织任务工作流：

- 任务队列只负责“选择与快速扫描”。
- 当前任务区域负责“理解、执行、验收”。
- 群聊负责“发指令、补上下文、触发路由”。
- 右侧 inspector 不再被任务长期占用。

### 4.2 备选方案及放弃理由

**方案 A：列表内展开**

在现有任务列表中展开详情。实现成本最低，但计划、时间线、Diff、日志会被压缩到卡片内部，无法承载 M1-M4 的完整闭环。

**方案 C：任务抽屉**

把任务详情做成聊天下方或右下抽屉。适合移动端，但桌面生产力不足，用户需要频繁开合抽屉才能查看执行状态。

## 5. 信息架构

### 5.1 页面级布局

桌面端保留三段式视觉节奏，但任务相关能力集中到左侧：

```txt
┌──────────────┬────────────────────────────┬───────────────────────┐
│ App Sidebar  │ Task Workspace             │ Chat / Files           │
│              │  Queue + Active Task       │ command entry          │
└──────────────┴────────────────────────────┴───────────────────────┘
```

建议宽度：

- `TaskWorkspacePanel`：`440px - 520px`，可通过 CSS clamp 控制。
- `Chat / Files`：占剩余空间，最低不小于 `520px`。
- 右侧 utility panel：仅在记忆、设置、临时详情需要时出现；任务详情不再默认占用右侧。

### 5.2 任务工作区结构

```txt
TaskWorkspacePanel
├── Header
│   ├── 标题：任务工作区
│   ├── 当前任务短标识：#abcd12
│   └── 创建任务 / 收起队列
├── Queue Rail
│   ├── 状态筛选：全部、待办、进行中、评审、完成、失败
│   ├── 任务列表：标题、状态、优先级、负责人、更新时间、执行状态
│   └── 最近活动摘要：最多 4 条 room activity
└── Active Task Surface
    ├── Command Header
    │   ├── 当前任务标题
    │   ├── 状态、优先级、负责人、执行模式
    │   └── 状态推进、启动工作流、定位来源、关闭激活
    ├── Execution Strip
    │   └── TaskExecutor 会话：agent、provider、状态、最近运行
    ├── Tabs
    │   ├── 概览
    │   ├── 时间线
    │   ├── 变更
    │   └── 日志
    └── Layer Controls
        └── Activity / Timeline / Runtime / Diff 可见性
```

### 5.3 空状态

无任务时，任务工作区不显示右侧空 inspector，而是在合并面板内部展示：

- “暂无任务”
- 创建任务入口
- 群聊发送任务型消息的提示
- 最近 room activity 仍可显示

有任务但未激活时，不静默自动激活任务。任务工作区展示“选择一个任务开始”的空状态，并显示最近更新的未完成任务作为推荐选择。只有用户点击任务、创建任务后选择激活，或收到后端 `task:activated` 事件时，才改变 `activeTaskId` 和 composer 发送目标。这样可以避免用户以为自己在全局聊天，实际消息被路由进某个自动选中的任务。

## 6. 交互逻辑

### 6.1 激活任务

点击队列任务卡：

1. 调用现有 `api.activateTask(roomId, task.id)`。
2. 本地立即设置 `activeTaskId`，保持当前乐观交互。
3. 收起记忆面板，避免任务上下文与记忆 inspector 同时争抢注意力。
4. 中间聊天 composer 的 `activeTaskId` 使用当前任务的可路由 ID。

收到 `task:activated` WebSocket 事件：

1. 同步 `activeTaskId`。
2. 更新任务工作区高亮。
3. 不强制滚动聊天，除非事件来源是“定位来源消息”。

### 6.2 群聊发送任务消息

Composer 上方或输入框附近需要展示当前路由上下文：

```txt
发送到：当前任务 #abcd12 · 修复任务事件回放
```

当 `activeTaskId` 不可路由或为空时：

```txt
发送到：全局聊天
```

消息发送仍使用现有 `activeTaskId` 参数，不新增 API。UI 只负责让用户明确路由结果。

### 6.3 状态推进

状态推进保留现有快速路径：

```txt
todo -> in_progress -> review -> done
```

任务工作区中至少提供两种入口：

- 队列卡片上的快速推进按钮，适合批量扫任务。
- 当前任务头部的主操作按钮，适合完成验收流。

失败状态仍作为显式操作，不放进默认推进链。

### 6.4 事件视图

任务详情 tab 从现有 `plan / timeline / diff / logs` 调整为中文语义：

- `概览`：任务描述、来源、计划事件、基础字段、关键操作。
- `时间线`：Activity + Timeline 层事件。
- `变更`：Diff 层事件。
- `日志`：Runtime 层事件。

层过滤仍保留，但位置从详情顶部挪到 tab 区域附近，避免用户误以为它是全局消息过滤。

### 6.5 执行会话

TaskExecutor 会话从详情中段提升到当前任务头部下方，作为 `Execution Strip`。

展示优先级：

1. 正在运行 / retrying / queued 的执行会话。
2. blocked / failed 的执行会话。
3. 最近完成的执行会话。

每条会话展示：

- agent 名称
- provider
- 状态
- 最近更新时间
- 关联 run / session 短 ID

这让 M4 的“会话绑 task”在 UI 上成为显性能力。

### 6.6 右侧面板行为

任务合并后，右侧不再展示 `TaskDetailPanel`。右侧区域仅用于：

- MemoryPanel
- AcpConfigPanel / 设置类弹层
- 未来文件预览或临时详情

当用户打开记忆面板时，中间聊天可适度收窄，但任务工作区不应被替换。

## 7. 组件边界

### 7.1 新组件

新增 `TaskWorkspacePanel`，承接现有任务 UI：

```ts
type TaskWorkspacePanelProps = {
  tasks: Task[];
  activeTask: Task | null;
  activeTaskId: string | null;
  statusFilters: TaskStatusFilter[];
  activityEvents: TaskEvent[];
  agents: RoomAgent[];
  workflows: WorkflowRun[];
  layerVisibility: TaskLayerVisibility;
  startingTaskId?: string | null;
  onStatusFiltersChange(filters: TaskStatusFilter[]): void;
  onSelectTask(task: Task): void;
  onChangeStatus(task: Task, status: Task['status']): void;
  onStartWorkflow?(task: Task): void;
  onLocateSourceMessage(messageId: string, task: Task): void;
  onLayerVisibilityChange(layer: MessageLayer, visible: boolean): void;
  onClearActiveTask(): void;
};
```

组件内部可拆分：

- `TaskQueueRail`
- `TaskQueueCard`
- `ActiveTaskHeader`
- `TaskExecutionStrip`
- `TaskWorkspaceTabs`
- `TaskOverviewView`
- `TaskTimelineView`

### 7.2 迁移策略

第一步不要求删除旧组件文件。推荐方式：

1. 提取 `TaskBoard` 的列表、筛选、活动逻辑到 `TaskQueueRail`。
2. 提取 `TaskDetailPanel` 的事件选择、执行会话、tab 内容到 `TaskWorkspacePanel` 内部子组件。
3. `RoomPage` 只渲染一个 `TaskWorkspacePanel`。
4. 确认无调用后再删除或保留兼容 `TaskBoard` / `TaskDetailPanel`。

这样可以降低一次性重构风险，并保护已有 `TaskDetailPanel.render.test.tsx` 中的纯展示测试。

### 7.3 状态归属

继续由 `RoomPage` 管理跨区域状态：

- `activeTaskId`
- `taskStatusFilters`
- `layerVisibility`
- `showMemoryPanel`

`TaskWorkspacePanel` 不拥有业务源状态，只负责展示和回调。

## 8. 数据流

### 8.1 查询

保留现有查询：

- `api.listRoomTasks(roomId)`
- `api.listRoomTaskEvents(roomId, { layer: 'activity', limit: 80 })`
- `api.listRoomTaskEvents(task.room_id, { taskId: task.id, limit: 80 })`
- `api.listTaskExecutors(task.id)`
- `api.listRoomWorkflows(roomId)`
- `api.listAgentRuns(roomId)`

实施阶段可把 `TaskDetailPanel` 内部的 task event / executor query 上移到 `TaskWorkspacePanel`，避免详情组件消失后查询散落。

### 8.2 WebSocket

保留现有事件处理：

- `task:created`
- `task:updated`
- `task:deleted`
- `task:activated`
- `task_event:new`
- `agent_run:created`
- `agent_run:updated`
- `message:new`
- `message:stream`

合并 UI 不新增 WebSocket 事件。`task_event:new` 继续 invalidate room task events；如果事件属于当前任务，当前任务 tabs 自动刷新。

### 8.3 写入

保留现有写操作：

- `api.activateTask(roomId, task.id)`
- `api.updateTask(task.id, patch)`
- `api.startWorkflow(...)` 或现有启动工作流入口
- 消息发送继续携带 `activeTaskId`

## 9. M1-M4 闭环承接

该设计必须支持完整闭环，而不是只做基础切片。

### M1 事件地基

任务工作区必须能消费 `task_events` 的按层投影：

- room activity 显示在队列区域。
- task timeline / diff / runtime 显示在当前任务 tabs。
- `task_event:new` 到达后 UI 自动更新，不需要手动刷新页面。

### M2 Task Router

任务工作区必须强化路由可解释性：

- 激活任务高亮必须明显。
- Composer 必须显示当前发送目标。
- 路由解释消息或 activity 可以定位到对应任务。
- 用户切换任务后，下一条无显式 @task 的群聊消息应进入新激活任务。

### M3 前端体验

本设计本身就是 M3 的体验升级：

- 任务队列与任务详情合并。
- 聊天不再被左右任务面板夹住。
- 当前任务的计划、事件、变更、日志成为连续工作流。

### M4 上下文隔离

TaskExecutor 展示必须成为当前任务的一等区域：

- 每个任务展示自己的 executor session。
- 正在运行的 task-agent 会话优先展示。
- 并发任务切换时，执行会话列表随 active task 切换，不混入 room-agent 全局会话。

## 10. 视觉规范

### 10.1 极简风默认方向

任务工作区采用极简风：

- 背景以纯白 / 近白为主。
- 使用 `1px` 分隔线表达层级。
- 卡片圆角不超过 `8px`。
- 不使用大面积渐变、发光阴影、玻璃拟态模糊。
- active 状态使用左侧细色条、浅底色、深色文字和明确边框。
- 数据标签、短 ID、时间继续使用 monospace。

### 10.2 密度

任务工作区是生产力面板，应优先支持扫描：

- 队列卡片标题最多两行。
- 描述默认最多两行。
- 负责人、状态、优先级、时间在一行内紧凑展示。
- 当前任务描述可以完整展示，但超过固定高度后内部滚动或折叠。
- Runtime 日志和 Diff 默认折叠长内容。

### 10.3 避免的问题

- 不要把任务工作区做成多层嵌套卡片。
- 不要用大 hero、营销式空状态或装饰性插画。
- 不要让任务面板抢占聊天输入区。
- 不要在移动端让三列硬挤在同一屏。

## 11. 响应式设计

### 11.1 桌面

桌面默认显示：

```txt
TaskWorkspacePanel + Chat/Files
```

当 MemoryPanel 打开时：

```txt
TaskWorkspacePanel + Chat/Files + MemoryPanel
```

如果宽度不足，优先收窄任务队列 rail，再收窄聊天，不隐藏当前任务头部。

### 11.2 平板

平板使用双模式：

- `任务`
- `聊天`

切换不丢失 active task。任务工作区占满主区域，聊天作为另一 tab。

### 11.3 移动端

移动端采用 route-level 或页面内 segmented control：

- `聊天`
- `任务`

任务工作区内部先显示当前任务，队列可通过顶部切换或抽屉进入。移动端不展示三列。

## 12. 验收标准

### 12.1 功能验收

- 房间页默认只出现一个任务工作区，不再同时出现左侧任务栏和右侧任务详情。
- 点击任务后，任务队列高亮、当前任务详情、composer 发送目标同步更新。
- 当前任务可以查看概览、时间线、变更、日志。
- 当前任务可以查看 TaskExecutor 会话。
- 可以从任务工作区定位来源消息，并滚动到聊天中的对应消息。
- 可以从任务工作区推进状态、标记失败、启动工作流。
- 打开记忆面板时，任务详情不会重新出现在右侧。

### 12.2 M1-M4 闭环验收

实施完成后必须通过真实浏览器测试以下流程：

1. 进入群聊房间。
2. 创建或选择一个任务并激活。
3. 在群聊中发送任务消息，确认 composer 显示发送目标。
4. 确认消息进入当前任务路由，并产生可见 activity / timeline 事件。
5. 启动任务执行或工作流。
6. 观察 TaskExecutor 会话状态变化。
7. 查看执行过程产生的 timeline / runtime / diff。
8. 推进任务状态到 review / done。
9. 切换到另一个任务，确认事件、执行会话和发送目标全部切换。

### 12.3 视觉验收

- 极简主题下无明显玻璃拟态残留。
- 任务工作区、聊天、记忆面板之间分隔清晰。
- 长标题、长任务描述、长 agent 名称不溢出。
- 任务列表和当前任务区域滚动互不干扰。
- 桌面和平板视口无明显重叠。

### 12.4 工程验收

- `npm run build` 成功。
- 相关组件测试通过；若迁移影响 `TaskDetailPanel.render.test.tsx`，需同步调整或新增 `TaskWorkspacePanel` 测试。
- 完成实现后进行代码审查。
- 全部任务完成后再次进行代码审查。
- 完成后自动提交 commit。

## 13. 风险与处理

### 风险 1：合并组件导致文件过大

处理：`TaskWorkspacePanel` 只做组合，队列、头部、执行会话、tabs 拆成小组件。单文件接近 500 行时继续拆分。

### 风险 2：当前任务区域过窄

处理：任务工作区宽度不低于 `440px`；队列 rail 可以压缩，详情区域优先保证可读。

### 风险 3：右侧任务详情消失导致用户找不到能力

处理：把原右侧能力逐项映射到当前任务区域，并在首次实现中保持文案一致，例如“时间线”“变更”“日志”。

### 风险 4：移动端体验退化

处理：移动端不沿用桌面合并布局，直接切为 `聊天 / 任务` 双 tab。

### 风险 5：任务激活策略影响路由

处理：不采用静默自动激活。composer 必须明确展示发送目标，并允许一键清除当前任务回到全局聊天。

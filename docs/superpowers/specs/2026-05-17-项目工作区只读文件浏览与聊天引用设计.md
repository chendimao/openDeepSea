# 项目工作区只读文件浏览与聊天引用设计

## 背景

当前项目在新建时已经保存本地目录 `projects.path`，后端也会校验该路径存在且是目录。ACP 运行时也会把项目目录作为当前工作区或可写目录传给 agent。

已有“项目级文件库与群聊引用”设计覆盖的是上传附件库：文件被复制到 `packages/backend/data/uploads/files/<projectId>/`，并通过 `files` 表和 `message_file_refs` 管理。该模型适合上传和复用附件，但不适合直接浏览真实项目源码目录。

本次设计新增“项目工作区只读文件浏览与聊天引用”：用户可以浏览项目目录内的真实文件，并在聊天消息框中快捷引用这些文件。引用不复制文件、不写入项目级附件库，只在消息 metadata 中记录相对路径快照，agent 调度时解析为本地绝对路径用于读取上下文。

## 目标

1. 以 `projects.path` 作为项目工作区根目录，提供只读文件浏览能力。
2. 前端支持在聊天消息框中快速选择工作区文件。
3. 发送消息时可携带工作区文件引用。
4. 后端校验引用路径必须位于项目目录内，防止路径穿越。
5. Agent prompt 中包含被引用工作区文件的文件名、相对路径、大小、MIME 类型和 `localPath`。
6. 与现有项目级文件库并存，不混用数据模型。
7. 首版必须同步收紧本地 API 访问控制，防止第三方网页跨站读取本机文件。

## 非目标

1. 不支持写入、删除、重命名、移动、创建目录。
2. 不把工作区文件复制进 `data/uploads/files`。
3. 不把工作区文件写入现有 `files` 表。
4. 不做跨项目文件引用。
5. 不做全文索引、版本管理、Git 状态展示或权限系统。
6. 不暴露裸 `express.static(project.path)` 静态服务。
7. 不默认允许浏览高风险路径，例如 `.git`、`node_modules`、构建产物和环境变量文件。

## 核心概念

### 项目级文件库

已有上传附件库。文件存储在后端数据目录，拥有数据库记录和可下载 URL，适合上传图片、PDF、设计稿、外部资料，并可跨同项目聊天室复用。

### 工作区文件

真实存在于 `projects.path` 下的文件。它的身份是“项目内相对路径”，例如：

```text
packages/frontend/src/components/RichMessageComposer.tsx
```

工作区文件不进入 `files` 表。它的引用随消息保存为 metadata 快照，后端每次调度 agent 时按项目根目录重新解析本地路径。

## 本地访问控制硬门禁

工作区文件 API 会暴露本机真实项目目录结构和文本内容，因此访问控制是首版前置需求，不是后续优化。

首版采用双层策略：

1. 收紧 CORS：后端不再对所有来源开放 `cors()`。开发环境只允许无 Origin 的本地工具请求、同源请求，以及明确可信的前端 Origin，例如 `http://localhost:5173`、`http://127.0.0.1:5173`。允许列表应由常量或环境变量集中管理。
2. 本地会话 token：后端启动时生成随机 token，通过只读启动日志或受控 bootstrap API 提供给前端开发服务。所有 `/api/projects/:projectId/workspace/*` 接口必须校验该 token，例如 `X-OpenDeepSea-Local-Token`。缺失或错误时返回 403。

实现计划必须先落访问控制，再暴露 workspace tree/file/search API。测试必须覆盖：

- 非可信 Origin 请求被拒绝。
- 缺失 token 的 workspace API 请求被拒绝。
- 错误 token 的 workspace API 请求被拒绝。
- 可信 Origin 加正确 token 才能访问 workspace API。

如果实现阶段发现 token 分发和 Vite dev server 集成成本过高，允许先把工作区 API 限定为同源生产访问，并在开发环境通过 Vite proxy 注入 token；但不能回退到开放 CORS 加无 token 的 workspace API。

## 后端设计

### 路径解析

新增共享工具模块，建议命名为 `workspace-files.ts`。

核心函数：

- `resolveWorkspacePath(projectPath, relativePath)`
- `isIgnoredWorkspacePath(relativePath)`
- `statWorkspaceEntry(projectPath, relativePath)`
- `listWorkspaceDirectory(projectPath, relativePath)`
- `buildWorkspaceAttachmentMetadata(project, relativePath)`

路径规则：

1. API 只接受相对路径。
2. 空路径表示项目根目录。
3. 禁止绝对路径。
4. 禁止 `..` 路径穿越。
5. 使用 `resolve(project.path, relativePath)` 后，结果必须等于项目根目录或位于项目根目录之下。
6. 先计算 `realpath(project.path)` 作为真实根目录；项目路径不存在时返回明确错误。
7. 对请求路径的每个组件使用 `lstat` 检查。
8. 目录 symlink 不进入、不列出为可进入目录。
9. 文件 symlink 只有在 `realpath(file)` 仍位于真实根目录下时才允许作为文件处理，否则拒绝。
10. 列表、预览、搜索、聊天引用校验、dispatch 解析必须复用同一套路由解析函数，避免不同入口出现安全差异。
11. 路径分隔符对外统一为 `/`。

### 忽略规则

首版使用内置忽略规则，不引入完整 `.gitignore` 解析。

默认忽略：

- `.git`
- `node_modules`
- `dist`
- `build`
- `.next`
- `.turbo`
- `coverage`
- `.env`
- `.env.*`
- `*.pem`
- `*.key`
- `*.sqlite`
- `*.db`

忽略规则适用于目录列表、文件读取、聊天引用校验和搜索。

### 文件大小限制

首版建议限制：

- 目录列表最多返回 500 项。
- 单个可预览文本文件最大 512 KB。
- 聊天可引用文件最大 2 MB。
- 超过限制的文件仍可在目录中显示，但不能预览或引用，并返回明确原因。

这些限制应集中成常量，便于后续调整。

### 文本与 MIME 判定

MIME 类型不能只依赖浏览器或文件扩展名。首版策略：

1. 对常见源码和文本扩展名使用白名单判定，例如 `.ts`、`.tsx`、`.js`、`.jsx`、`.json`、`.md`、`.css`、`.html`、`.txt`、`.yml`、`.yaml`、`.toml`、`.sh`。
2. 对无扩展名或未知扩展名文件，读取前 8 KB 做二进制探测；包含 NUL 字节或明显二进制控制字符时视为二进制。
3. 文本预览 API 只返回 UTF-8 文本；无法按 UTF-8 解码时拒绝预览。
4. `mimeType` 使用保守值，例如源码文件可统一返回 `text/plain` 并附加 `language` 字段；避免使用非标准 MIME 作为关键逻辑。
5. 二进制文件可以出现在目录列表中，但首版不可预览；是否可引用取决于大小限制和是否命中忽略规则。图片文件首版可作为普通 workspace 引用，但不作为前端缩略图或 ACP image 参数。

### 搜索遍历限制

文件名搜索必须受限，不能为了返回 50 个结果扫描整个巨大 monorepo。

首版限制：

- 最大遍历目录数：1000。
- 最大检查文件数：10000。
- 最大递归深度：12。
- 最大耗时：1500 ms。
- 最大返回结果：50。

搜索遇到权限错误、损坏 symlink、删除中的文件或其他可恢复 FS 错误时跳过该路径，并在响应中返回 `truncated: true` 或 `skippedCount`。遇到项目根目录不可访问时返回错误。

### API

#### 浏览目录

```http
GET /api/projects/:projectId/workspace/tree?path=<relativePath>
```

返回：

```json
{
  "path": "packages/frontend/src",
  "entries": [
    {
      "name": "components",
      "path": "packages/frontend/src/components",
      "kind": "directory",
      "size": 0,
      "mtime": 1778940000000,
      "ignored": false,
      "selectable": false
    },
    {
      "name": "main.tsx",
      "path": "packages/frontend/src/main.tsx",
      "kind": "file",
      "size": 1234,
      "mtime": 1778940000000,
      "mimeType": "text/plain",
      "language": "tsx",
      "ignored": false,
      "selectable": true
    }
  ]
}
```

排序规则：

1. 目录在前，文件在后。
2. 同类按名称自然排序。
3. 忽略项默认不返回；如果后续需要调试，可增加 `includeIgnored=1`，但首版不做。

#### 读取文件预览

```http
GET /api/projects/:projectId/workspace/file?path=<relativePath>
```

仅支持文本类文件预览。返回：

```json
{
  "path": "packages/frontend/src/main.tsx",
  "name": "main.tsx",
  "mimeType": "text/plain",
  "language": "tsx",
  "size": 1234,
  "content": "import React from 'react';\n...",
  "truncated": false
}
```

二进制、大文件、被忽略文件或目录返回 400，并给出错误原因。图片预览首版不通过工作区 API 暴露真实文件 URL；如需要图片查看，应后续设计独立的安全预览接口。

#### 搜索文件

```http
GET /api/projects/:projectId/workspace/search?q=<query>
```

首版只做文件名搜索，不做内容搜索。返回最多 50 个匹配文件，跳过忽略目录和超过引用限制的文件。

### 发送消息扩展

现有消息发送接口继续保留 `fileIds`，新增 `workspaceFilePaths`：

```http
POST /api/rooms/:roomId/messages
Content-Type: application/json

{
  "content": "看看这些文件",
  "mentions": ["room_agent_id"],
  "fileIds": ["uploaded-file-id"],
  "workspaceFilePaths": [
    "packages/frontend/src/components/RichMessageComposer.tsx"
  ]
}
```

multipart 消息也支持 `workspaceFilePaths` 字段，格式为 JSON 字符串数组。

服务端校验：

1. `roomId` 必须存在。
2. room 所属项目必须存在。
3. 每个 `workspaceFilePaths` 必须是项目内相对路径。
4. 路径必须对应普通文件。
5. 文件不能命中忽略规则。
6. 文件大小不能超过聊天引用限制。
7. 同一消息内重复路径去重。
8. 上传附件、项目文件库引用、工作区文件引用的总数受 `MAX_MESSAGE_FILES` 约束，或新增统一的 `MAX_MESSAGE_REFERENCES` 常量。
9. 只有本地上传文件和 `fileIds` 对应的项目级文件库文件写入 `message_file_refs`。
10. `workspaceFilePaths` 只写入 `messages.metadata.attachments`，不写 `files` 表，也不写 `message_file_refs`。
11. JSON 与 multipart 的 `workspaceFilePaths` 解析失败时返回 400，不创建消息，也不清空前端 composer。

## Message Metadata

将 `MessageAttachmentMetadata` 明确为 discriminated union，同时保持历史 metadata 兼容。没有 `source` 的旧附件按 `upload` 处理。

```ts
interface BaseMessageAttachmentMetadata {
  id: string;
  name: string;
  mimeType: string;
  size: number;
  isImage: boolean;
  deleted?: boolean;
}

interface UploadedMessageAttachmentMetadata extends BaseMessageAttachmentMetadata {
  source?: 'upload' | 'project-file';
  fileId?: string;
  url: string;
}

interface WorkspaceMessageAttachmentMetadata extends BaseMessageAttachmentMetadata {
  source: 'workspace';
  workspacePath: string;
  language?: string;
  url?: never;
}

type MessageAttachmentMetadata =
  | UploadedMessageAttachmentMetadata
  | WorkspaceMessageAttachmentMetadata;
```

工作区文件 metadata 示例：

```json
{
  "attachments": [
    {
      "id": "workspace:7a4f2d8c1e9b0a63",
      "source": "workspace",
      "workspacePath": "packages/frontend/src/main.tsx",
      "name": "main.tsx",
      "mimeType": "text/plain",
      "language": "tsx",
      "size": 1234,
      "isImage": false
    }
  ]
}
```

工作区文件不写 `url`。前端渲染时如果 `source === 'workspace'`，必须使用 `workspacePath` 展示路径，禁止当作下载链接或图片地址。

工作区附件的 `id` 不直接拼接原始路径。首版使用规范化路径作为去重键，并生成稳定短 ID：

```text
workspace:<sha256(normalizedWorkspacePath).slice(0, 16)>
```

其中 `normalizedWorkspacePath` 是去重、排序和展示前统一规范化后的 `/` 分隔相对路径。

前端 `parseMessageMetadata` / sanitizer 必须同步改为 union 解析：

1. `source === 'workspace'` 时要求 `workspacePath`、`name`、`mimeType`、`size`、`isImage` 合法，不要求 `url`。
2. `source` 缺失、`upload` 或 `project-file` 时沿用 URL 校验，但允许 `/uploads/messages/` 和 `/uploads/files/` 两类安全前缀。
3. workspace 附件卡片禁止渲染 `<a href>`、`<img src>` 或任何可直接打开本机路径的元素。

## Agent 调度

当前 dispatch 会从附件 URL 解析上传文件本地路径。workspace 解析不能从 URL 推导，必须依赖项目路径，因此需要明确重构解析入口。

改造点：

1. `getResolvedMessageAttachments(userMessage)` 改为接收项目路径，例如 `getResolvedMessageAttachments(userMessage, project.path)`。
2. `resolveMessageAttachmentLocalPath(attachment)` 改为 `resolveMessageAttachmentLocalPath(attachment, projectPath)`。
3. 上传附件和项目级文件库附件继续按 `/uploads/messages/`、`/uploads/files/` URL 解析到上传根目录。
4. `source === 'workspace'` 的附件必须读取 `workspacePath`，并通过 `workspace-files.ts` 的统一解析函数得到 `localPath`。
5. 如果文件已不存在、被忽略、变为目录、变为项目外 symlink 或项目路径不可访问，prompt 中标记 `localPath=unavailable`。
6. 工作区附件不参与现有图片 `imagePaths` 传参，首版只通过 prompt 的 `localPath` 告知 agent。

Agent prompt 文案调整：

```text
消息引用文件：
请结合以下文件回答。如果需要查看内容，请读取对应 localPath。
1. main.tsx | source=workspace | relativePath=packages/frontend/src/main.tsx | mimeType=text/plain | size=1234 | localPath=/abs/project/packages/frontend/src/main.tsx
```

图片文件首版不作为 ACP image 参数传入，除非后续确认 ACP adapters 对本地 workspace 图片路径的处理边界。

## 前端设计

### 文件浏览入口

首版以聊天框内选择器为主，不新增独立页面。

在 `RichMessageComposer` 工具栏中保留现有两个入口：

- 本地文件上传
- 项目级文件库选择

新增第三个入口：

- 工作区文件选择

建议图标使用 `FolderTree` 或 `FileCode`，tooltip 文案为“引用工作区文件”。

### `WorkspaceFilePickerDialog`

新增组件职责：

1. 显示当前目录面包屑。
2. 展示目录和文件列表。
3. 支持进入目录、返回上级。
4. 支持文件名搜索。
5. 支持多选可引用文件。
6. 已选文件显示勾选状态。
7. 不可引用文件显示禁用态和原因。
8. 确认后把文件元信息返回给 composer。

列表项展示：

- 文件名
- 相对路径
- 文件大小
- 修改时间
- 文件类型

### Composer attachment 类型

当前 composer attachment 分为本地待上传和项目文件库引用。新增工作区引用：

```ts
type ComposerAttachment =
  | { kind: 'local'; id: string; file: File; previewUrl: string | null }
  | { kind: 'project'; id: string; file: ProjectFile }
  | { kind: 'workspace'; id: string; file: WorkspaceFileEntry };
```

发送时：

- `local` 走 multipart `files`
- `project` 走 `fileIds`
- `workspace` 走 `workspaceFilePaths`

附件栏展示：

- 工作区文件显示文件名和相对路径。
- 不显示图片缩略图。
- 支持移除。

### 消息渲染

消息 metadata 解析需要允许 `source` 和 `workspacePath` 字段。

渲染规则：

1. 上传附件和项目文件库附件继续按 URL 下载或预览。
2. 工作区文件显示为引用卡片，展示文件名和相对路径。
3. 工作区文件不提供直接下载链接。
4. 如果后端后续提供预览接口，可在卡片上打开只读预览弹窗。

实现注意：

- `MessageAttachmentMetadata` 类型改为 union 后，所有读取 `attachment.url` 的 UI 必须先判断非 workspace 类型。
- 现有项目级文件库附件的 `/uploads/files/` URL 也应加入 sanitizer 白名单，避免已有文件库引用被过滤。
- 工作区文件引用卡片应提供“复制相对路径”动作，而不是下载动作。

## 安全边界

1. 不挂载 `express.static(project.path)`。
2. 不返回真实绝对路径给前端。
3. 只在 agent prompt 内使用 `localPath`，因为 agent 运行在本地并已围绕项目目录工作。
4. API 始终以项目 ID 和相对路径定位文件。
5. 禁止路径穿越、绝对路径和项目外 symlink。
6. 默认忽略敏感文件和重型目录。
7. 工作区文件引用失败时不阻断历史消息展示，只在后续 agent 调度中标记 unavailable。
8. 工作区 API 必须强制通过“可信 Origin + 本地 token”校验。未完成访问控制前不得合并 workspace tree/file/search API。

## 错误处理

常见错误：

- `project not found`：项目不存在。
- `workspace path is invalid`：路径非法或越界。
- `workspace path is ignored`：路径命中忽略规则。
- `workspace file not found`：文件不存在。
- `workspace path is not a file`：引用对象不是普通文件。
- `workspace file is too large`：文件超过引用或预览限制。
- `too many files`：总引用数量超过限制。

前端行为：

- 浏览目录失败：在弹窗内展示错误和重试按钮。
- 引用文件失败：toast 提示，并保留编辑器内容。
- 发送时校验失败：沿用当前发送失败提示，不清空 composer。

## 验证策略

### 后端测试

1. 目录浏览只返回项目根内文件。
2. `../` 路径穿越被拒绝。
3. 绝对路径被拒绝。
4. 忽略目录和敏感文件不出现在列表中。
5. symlink 指向项目外时被拒绝。
6. 文件名搜索跳过忽略目录。
7. 发送消息时 `workspaceFilePaths` 写入 metadata。
8. 不存在、过大、被忽略、越界路径不能作为聊天引用。
9. dispatch 能把 workspace metadata 解析为 `localPath`。
10. 文件删除或移动后 dispatch 标记 `localPath=unavailable`。
11. 非可信 Origin 访问 workspace API 被拒绝。
12. 缺失或错误本地 token 访问 workspace API 被拒绝。
13. JSON 与 multipart 的 `workspaceFilePaths` 解析错误返回 400。
14. 混合上传附件、`fileIds` 和 `workspaceFilePaths` 时总数限制生效。
15. workspace 引用不写入 `files` 表和 `message_file_refs`。
16. 搜索遇到大目录、权限错误和 symlink loop 时受限返回。

### 前端验证

1. 工作区文件选择器能打开根目录并进入子目录。
2. 搜索能按文件名筛选。
3. 多选文件后附件栏显示工作区引用。
4. 移除附件后不随消息发送。
5. 发送消息 payload 包含 `workspaceFilePaths`。
6. 历史消息能显示工作区文件引用卡片。
7. metadata parser 保留 workspace 附件，并继续保留上传附件和项目文件库附件。
8. workspace 引用卡片不可下载、不可作为图片预览。
9. 发送失败后 composer 内容和已选引用不被清空。

### 最低门禁

实现完成后至少运行：

```bash
npm run build
```

若补充测试，应运行相关后端和前端测试命令。

## 风险与权衡

1. 工作区文件会随本地目录变化而变化；消息保存的是路径快照，不保证未来仍存在。
2. 不返回绝对路径给前端会减少泄露面，但前端不能直接打开本地文件。
3. 不复制文件避免污染附件库，但历史可追溯性弱于上传附件。
4. 忽略规则首版内置，简单可靠；代价是可能和项目 `.gitignore` 不完全一致。
5. 访问控制是本功能的合并前置条件；如果 token 分发方案变复杂，应缩小 workspace API 暴露范围，而不是放宽校验。
6. MIME 和文本判定采用保守策略，可能拒绝部分可读文件；这是首版可接受权衡。
7. 搜索有遍历上限，巨大项目中可能找不到深层文件；用户仍可通过目录浏览进入目标路径。

## 已确认决策

1. 只做只读文件浏览。
2. 聊天消息框支持快捷引入工作区文件。
3. 工作区文件不进入项目级文件库。
4. 不开放真实项目目录静态服务。
5. 首版不支持写入、删除、重命名。
6. Agent 通过 prompt 中的 `localPath` 读取被引用文件。
7. 工作区 API 首版必须具备可信 Origin 和本地 token 保护。
8. 工作区附件 metadata 使用 discriminated union，不写 `url`。
9. Dispatcher 必须接收项目路径后解析 workspace `localPath`。
10. 工作区引用不写入 `files` 和 `message_file_refs`。

# 多智能体长生命周期会话设计

## 背景

当前 Session 页面已经具备业务会话、消息历史、运行记录、ACP 适配器和 WebSocket 事件通知，但普通消息发送仍偏向“HTTP 提交一次消息，后端启动一次新的 run”。每次 run 记录自己的 `acp_session_id`，后续普通消息没有稳定继承上一轮 ACP 会话，因此还没有形成 Agent CLI 风格的长生命周期智能体会话。

目标状态是：一个 OpenDeepSea 会话可以同时存在多个智能体，每个智能体拥有自己的可恢复 ACP runtime 和独立事件流；前端通过 WebSocket 实时接收这些流并按智能体、运行、通道进行分流展示。

## 目标

1. 一个业务会话内允许多个智能体并行或串行工作。
2. 每个智能体在同一业务会话内复用自己的 ACP/provider session。
3. 每个智能体拥有独立的运行状态、事件序列、暂停/继续控制和错误边界。
4. 前端实时展示 thinking、tool、command、answer、status 等事件，不再只依赖 HTTP 轮询或完成后刷新。
5. 暂停只中断当前生成或工具循环，不销毁业务会话、智能体 runtime 或 provider session。

## 非目标

1. 不要求每个智能体都建立一条物理浏览器 WebSocket 连接。
2. 不要求第一阶段实现完整 token cache、hidden reasoning cache 或可恢复子进程。
3. 不把导出、通知、会话计划编辑纳入本设计。
4. 不改变 ACP provider 的底层能力边界；provider 不支持的能力由 OpenDeepSea runtime 层做降级。

## 推荐架构

推荐使用“单前端 WebSocket 连接 + 多智能体逻辑事件流 + 后端多 Agent Runtime”的结构。

```text
Frontend
  WebSocket /ws
    subscribe sessionId
    send session.message.send / agent.run.pause / agent.run.resume
    receive agent.run.delta / agent.run.status / session.message.created

Backend Session Gateway
  Session Runtime Manager
    Map<sessionId, SessionRuntime>

SessionRuntime
  shared message history
  shared workspace context
  agents: Map<agentId, AgentRuntime>

AgentRuntime
  agentId
  provider
  providerSessionId
  currentRunId
  status
  event sequence
  abort controller
  checkpoint summary

ACP Adapter
  provider session resume
  event stream normalization
```

前端可以只有一条物理 WebSocket。后端事件必须携带 `sessionId`、`agentId`、`runId` 和 `seq`，让前端可靠分流。

## 核心数据模型

### SessionRuntime

```ts
type SessionRuntime = {
  sessionId: string;
  workspacePath: string;
  sharedMessages: SessionMessage[];
  agents: Map<string, AgentRuntime>;
};
```

### AgentRuntime

```ts
type AgentRuntime = {
  sessionId: string;
  agentId: string;
  provider: 'codex' | 'claudecode' | 'opencode';
  providerSessionId: string | null;
  currentRunId: string | null;
  status: 'idle' | 'running' | 'paused' | 'failed' | 'completed';
  nextSeq: number;
  abortController: AbortController | null;
  latestCheckpointId: string | null;
};
```

### 持久化建议

需要新增或等价补齐以下持久化关系：

```text
session_agent_runtimes
  id
  session_id
  agent_id
  provider
  provider_session_id
  status
  latest_run_id
  latest_checkpoint_id
  created_at
  updated_at

session_agent_events
  id
  session_id
  agent_id
  run_id
  seq
  channel
  event_type
  content
  payload_json
  created_at
```

如果第一阶段不新增表，也至少要保证新 run 能读取同一 `sessionId + agentId` 下最近可用的 `provider_session_id`。

## WebSocket 事件协议

### 客户端到服务端

```ts
type ClientEvent =
  | { type: 'session.subscribe'; sessionId: string }
  | { type: 'session.message.send'; sessionId: string; target: AgentTarget; content: string }
  | { type: 'agent.run.pause'; sessionId: string; agentId: string; runId: string }
  | { type: 'agent.run.resume'; sessionId: string; agentId: string; content?: string }
  | { type: 'agent.run.cancel'; sessionId: string; agentId: string; runId: string };

type AgentTarget =
  | { kind: 'agent'; agentId: string }
  | { kind: 'broadcast'; agentIds: string[] }
  | { kind: 'auto' };
```

### 服务端到客户端

```ts
type ServerEvent =
  | { type: 'session.message.created'; sessionId: string; message: SessionMessage }
  | { type: 'agent.run.created'; sessionId: string; agentId: string; run: SessionRun }
  | {
      type: 'agent.run.delta';
      sessionId: string;
      agentId: string;
      runId: string;
      seq: number;
      channel: 'answer' | 'thinking' | 'tool' | 'command' | 'event';
      chunk: string;
      payload?: Record<string, unknown>;
    }
  | { type: 'agent.run.status'; sessionId: string; agentId: string; runId: string; status: AgentRuntime['status'] }
  | { type: 'agent.run.completed'; sessionId: string; agentId: string; runId: string }
  | { type: 'agent.run.failed'; sessionId: string; agentId: string; runId: string; error: string };
```

## 消息路由

用户输入分为三类：

1. 定向消息：`@planner 继续拆计划` 只进入 planner 的 runtime。
2. 广播消息：`大家评审这个方案` 会写入共享消息，并派发给多个 AgentRuntime。
3. 自动路由：后端根据当前任务阶段、活跃智能体和系统策略选择目标智能体。

每条消息应持久化一次，并记录目标：

```text
message_targets
  message_id
  session_id
  agent_id
  delivery_status
```

这样可以区分“用户在会话里说过什么”和“哪些智能体消费了这条消息”。

## 暂停与继续

暂停语义必须从取消语义中拆开。

### 暂停单个智能体

```text
agent.run.pause
  -> abort 当前生成或工具循环
  -> run.status = paused
  -> agent_runtime.status = paused
  -> 保留 providerSessionId
  -> 保留事件序列和 checkpoint
```

### 取消单个智能体

```text
agent.run.cancel
  -> abort 当前运行
  -> run.status = cancelled
  -> agent_runtime.status = idle 或 failed
  -> 是否保留 providerSessionId 由策略决定
```

### 继续单个智能体

```text
agent.run.resume
  -> append 可选用户消息
  -> 使用同一个 providerSessionId 调 ACP resume
  -> 创建新的 run 或恢复 paused run
```

第一阶段建议“继续时创建新 run，但复用 providerSessionId”。这样审计清晰，也兼容 provider 不支持恢复同一个 run 的场景。

## ACP Adapter 要求

每个 adapter 都要支持同一调用契约：

```ts
invoke({
  projectPath,
  sessionId: providerSessionId,
  prompt,
  onSession,
  onChunk,
  signal,
});
```

关键要求：

1. 新建 provider session 后必须通过 `onSession` 或 result 返回 `providerSessionId`。
2. 后续同一智能体消息必须传入上一次保存的 `providerSessionId`。
3. `signal.abort()` 只代表中断当前 run，不代表业务 session 销毁。
4. adapter 输出统一归一化为 answer、thinking、tool、command、event。

## 前端展示

中间聊天区应按会话时间线展示共享消息，同时允许按智能体折叠事件：

```text
USER: 修复登录页面

PLANNER
  thinking...
  answer...

CODER
  command npm test
  tool_result ...
  answer...

REVIEWER
  answer...
```

右侧状态区按智能体显示：

```text
Planner   paused   run-12
Coder     running  run-13
Reviewer  idle     -
```

输入区支持：

1. `@agent` 定向。
2. 多选智能体广播。
3. 暂停当前智能体。
4. 暂停全部智能体。

## 分阶段落地

### 阶段一：修正 ACP session 复用

新 run 创建时继承同一 `sessionId + agentId` 下最近可用的 `providerSessionId`。完成后，一个业务会话中的同一智能体可以连续对话。

### 阶段二：前端消费增量事件

前端不再忽略 `done=false` 的 stream 事件，而是根据 `agentId/runId/seq/channel` 增量渲染。

### 阶段三：多智能体 runtime registry

后端建立 `SessionRuntimeManager`，维护 `Map<sessionId, AgentRuntime>`，支持多个智能体并行运行和独立状态。

### 阶段四：暂停、继续、取消分离

新增 paused 状态和对应 API/WebSocket 事件，明确 pause 与 cancel 的不同语义。

### 阶段五：checkpoint 与压缩

为长生命周期会话增加 checkpoint、summary、context compression，避免 token 膨胀和 provider session 失效导致上下文丢失。

## 风险与处理

1. 事件乱序：每个 `agentId + runId` 使用单调递增 `seq`，前端按 seq 合并。
2. WebSocket 断线：前端重连后通过 HTTP 拉取 workspace snapshot，再继续接收后续事件。
3. provider 不支持真实 pause：降级为 abort 当前调用并保留 providerSessionId，继续时新 run resume。
4. 多智能体同时写文件：运行前计算 write scope，冲突时串行化或要求用户确认。
5. provider session 失效：使用最近 checkpoint 和 message history 创建新 provider session，并记录 handoff evidence。

## 验收标准

1. 同一业务会话中，同一智能体连续两次发送消息时，第二次 ACP 调用使用第一次保存的 `providerSessionId`。
2. 一个业务会话内可同时存在至少两个智能体 runtime，事件按 `agentId` 正确分流。
3. 前端能实时显示 `thinking/tool/answer` 增量事件，而不是等待 run 完成后才刷新。
4. 暂停单个智能体不会影响同会话内其他智能体运行。
5. 暂停后继续输入，仍能沿用同一智能体的上下文。
6. WebSocket 断开重连后，页面能通过 snapshot 恢复当前会话状态。

## 推荐结论

采用“单 WebSocket 连接承载多智能体逻辑流，后端为每个智能体维护独立长生命周期 ACP runtime”的方案。它比每个智能体一条物理 WebSocket 更稳，也比 HTTP request/response 更适合 Agent CLI 风格的暂停、继续和实时可观测性。

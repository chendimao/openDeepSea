# 群聊页面紧凑头部与功能切换设计

## 背景

当前群聊页面的 header 垂直空间偏大，`toolbar-tabs` 只是静态占位，实际功能切换不清晰。页面同时展示聊天与任务看板，主视图区被拆成双栏，用户在群聊、任务和文件之间切换时不够直接。

本设计优化群聊页面的信息架构：

- 压缩 header 高度，减少顶部空白。
- 将原 `toolbar-tabs` 改造为聊天室切换列表。
- 将 `segmented-control` 改为功能 tabs：`群聊`、`任务`、`文件`。
- `群聊`、`任务`、`文件` 互斥展示在主内容区。
- 文件 tab 复用现有文件页面能力，只筛选当前群聊关联文件。
- 支持从任务或文件定位回群聊消息上下文。

## 目标

1. Header 默认保持单行紧凑布局，降低页面顶部占高。
2. 用户可以在当前项目的多个聊天室之间无缝切换。
3. 主内容区通过 tabs 切换 `群聊`、`任务`、`文件`，避免聊天和任务固定双栏同时占用空间。
4. `智能体消息` 文案改为 `群聊`。
5. 当前群聊文件列表复用现有文件页面列表、卡片、预览、下载、删除等能力。
6. 用户可以从文件定位到群聊中引用该文件的消息，也可以从任务定位到群聊中任务开始的来源消息。

## 非目标

- 不重新设计全局文件库页面。
- 不改变文件存储模型和任务创建模型。
- 不引入新的大型布局框架。
- 不在本轮设计中实现跨项目聊天室切换。

## 页面结构

### 紧凑 Header

Header 采用单行布局，建议高度约 46px 到 52px：

- 左侧：返回按钮、当前聊天室名称、项目名称简要信息。
- 中间：聊天室切换列表，替代原 `toolbar-tabs`。
- 右侧：创建任务、设置、记忆、邀请智能体等操作按钮。

中间聊天室列表只展示当前可视区域内能容纳的聊天室 chip。列表末尾提供一个 `更多` 下拉按钮，点击后展示未显示的剩余聊天室。

### 聊天室切换列表

聊天室列表来源为当前项目的 rooms。

交互规则：

- 当前 room chip 高亮。
- 点击其他 room chip 后跳转到 `/projects/:projectId/rooms/:roomId`。
- `更多` 下拉中展示未在主列表中显示的聊天室。
- 如果聊天室数量较少，`更多` 按钮可以隐藏。
- 移动端可降级为当前聊天室 + `更多` 下拉，避免横向挤压操作按钮。

### 功能 Tabs

`segmented-control` 改为当前 room 内的功能切换：

- `群聊`
- `任务`
- `文件`

默认进入 `群聊` tab。三个 tab 互斥展示主内容区，不再默认左右双栏同时展示聊天和任务看板。

## 功能行为

### 群聊 Tab

群聊 tab 展示现有聊天流和 composer。原 `智能体消息` 文案改为 `群聊`。

当用户从文件或任务触发定位时：

1. 自动切换到 `群聊` tab。
2. 滚动到目标消息。
3. 短暂高亮目标消息，帮助用户识别上下文。

### 任务 Tab

任务 tab 展示当前 room 的任务看板，复用现有 `TaskBoard` 能力。

任务详情的呈现可以延续现有交互，但不再作为页面固定右栏默认占位。点击任务后可在任务 tab 内展示详情，或后续实现为详情面板。

任务定位规则：

- 任务存在 `source_message_id` 时，任务卡片或详情中提供 `定位开始消息` 操作。
- 点击后切换到 `群聊` tab，并滚动到 `source_message_id` 对应消息。
- 如果任务没有来源消息，隐藏定位入口或置灰并提示无法定位。

### 文件 Tab

文件 tab 复用现有文件页面能力，但数据范围固定为当前 room：

- 使用现有 `api.listFiles({ projectId, roomId })` 获取当前群聊关联文件。
- 复用现有 `ProjectFileView` 的列表 / 卡片视图能力。
- 保留搜索、预览、下载、删除等操作。
- 删除文件沿用现有 `api.deleteProjectFile(fileId)` 逻辑，并同步刷新 `files` 与当前 room 相关缓存。

文件定位规则：

- 文件项提供 `定位消息` 操作。
- 默认定位到该文件在当前 room 中最近一次被引用的消息。
- 如果后续需要展示全部引用位置，可扩展为下拉列表或弹层，但第一版只需最近引用即可满足主流程。
- 当前 `ProjectFile` 已有 `last_referenced_at`、`last_referenced_room_id`、`last_referenced_room_name`，但缺少直接暴露的 `last_referenced_message_id`。实现时建议补充轻量后端字段，或新增文件引用查询接口。

## 数据与状态

RoomPage 新增本地状态：

- `activeTab: 'chat' | 'tasks' | 'files'`
- `highlightMessageId: string | null`

建议新增跳转函数：

```ts
function focusMessage(messageId: string): void {
  setActiveTab('chat');
  setHighlightMessageId(messageId);
}
```

消息列表中的每条消息需要可被 `message.id` 定位。实现可以在消息节点上增加 `data-message-id`，并维护 refs map，切到群聊 tab 后调用 `scrollIntoView`。

文件 tab 需要当前 room 文件数据：

```ts
api.listFiles({ projectId, roomId })
```

若要支持文件直接定位消息，后端需要满足以下任一条件：

1. 在 `ProjectFile` 列表返回中增加 `last_referenced_message_id`。
2. 新增 `GET /files/:fileId/refs?roomId=:roomId`，返回该文件在当前 room 的引用消息列表。

第一版推荐方案 1，改动小，能支持默认定位最近引用消息。

## 组件边界

建议拆分为以下小组件，避免继续放大 `RoomPage.tsx`：

- `RoomSwitcher`：负责当前项目聊天室 chips 与 `更多` 下拉。
- `RoomFeatureTabs`：负责 `群聊 / 任务 / 文件` 切换。
- `RoomFilesPanel`：复用文件页逻辑，只接收 `projectId`、`roomId` 和 `onLocateMessage`。
- `MessageFocusController` 或局部 hook：封装消息滚动和高亮。

现有 `FilesPage` 中可抽出共用的文件列表区域，让独立文件页和 room 文件 tab 复用同一套展示与操作逻辑。

## 响应式行为

桌面端：

- Header 单行展示。
- 聊天室 chips 占据中间可用空间。
- 操作按钮保留在右侧。
- 主内容区只显示当前 tab。

移动端：

- Header 可以保持两行以内。
- 聊天室列表优先展示当前 room 和 `更多`。
- 操作按钮可横向滚动或收敛为图标按钮。
- 功能 tabs 保持可见，避免用户迷失在单一内容区中。

## 风险与处理

1. 文件定位缺少 message id。
   - 处理：后端补充 `last_referenced_message_id` 或新增 refs 查询接口。

2. `RoomPage.tsx` 已经较长，继续堆叠会增加维护成本。
   - 处理：实现时优先提取 `RoomSwitcher`、`RoomFilesPanel` 和消息定位 hook。

3. Header 中间聊天室列表与右侧操作按钮可能挤压。
   - 处理：chips 区域使用 `minmax(0, 1fr)`，超过可视容量的房间进入 `更多` 下拉；移动端进一步收敛。

4. 删除文件会影响消息附件展示。
   - 处理：沿用现有删除逻辑，后端已有消息附件 deleted 标记更新能力；前端刷新消息和文件缓存。

## 验证方式

- `npm run build`
- 手动验证桌面端：
  - Header 高度明显降低。
  - 可切换聊天室。
  - `更多` 下拉能展示剩余聊天室并跳转。
  - `群聊 / 任务 / 文件` 互斥切换。
  - 文件 tab 只展示当前 room 文件。
  - 文件删除后列表刷新，聊天附件状态正确。
  - 文件定位能切回群聊并高亮目标消息。
  - 任务定位能切回群聊并高亮来源消息。
- 手动验证移动端：
  - Header 不发生文字重叠。
  - tabs 可操作。
  - 聊天室切换不挤压主要操作。

## 当前设计确认

用户已确认以下决策：

- 采用顶部聊天室列表，不使用左侧房间列表。
- 聊天室列表默认不展示全部聊天室，只展示可视区域内的部分，剩余放入下拉。
- `群聊` 和 `任务` 通过 tab 切换互斥展示。
- 增加 `文件` tab。
- 文件 tab 直接复用原文件页面能力，只筛选当前群聊文件。
- 文件和任务都需要支持定位到群聊中的相关消息。

# 子代理事件采集与原生编排设计

## 背景

任务 `IQ7qudivzq34` 的最新执行消息中，前端执行者在正文里声明“会派一个只读子代理”“审查子代理已启动”，但时间线中没有内部子代理事件。排查本地 SQLite 后确认：

- `agent_runs` 在对应时间段只有 `planner` 和 `frontend-executor` 两条 run。
- 消息 trace 中 `subagent_*` 事件数为 0。
- trace 中 `spawn_agent`、`wait_agent`、`dispatch_agent`、`Task` 工具事件数为 0。
- 只有 assistant 正文包含“子代理”自然语言声明。

这说明当前问题不是前端过滤，也不是后端子代理事件归一化失败，而是 ACP/CLI 源头没有返回结构化子代理事件。

## 社区方案分析

社区里获取 agent CLI 子代理事件通常有三类方式。

### 1. ACP 工具调用流

ACP 标准只提供通用 `session/update` 流，包含 message、thought、tool call、plan、usage 等更新。子代理不是 ACP 的一等标准类型。

如果 CLI 把子代理派发暴露为工具调用，例如：

- Claude Code 的 `Task`
- Codex 的 `spawn_agent`
- Codex 的 `wait_agent`
- 自定义 `subagent_result`

那么 ACP 客户端可以从 `tool_call` / `tool_call_update` 中归一化出 `subagent_started`、`subagent_progress`、`subagent_completed`、`subagent_failed`。

这种方案实时、干净，但前提是 CLI 通过 ACP 明确上报子代理工具事件。

### 2. 本地 transcript 旁路读取

Claude Code 和部分社区 devtools 会直接读取本地 JSONL transcript。Claude Code 主 transcript 中可能出现 `Task` 工具调用，子代理还可能写入独立的 subagent transcript。

这种方案可以补齐 ACP 未暴露的信息，但依赖每个 CLI 的私有文件格式和存储路径。Codex、Claude Code、OpenCode 都需要独立适配，历史兼容成本高。

### 3. 平台原生编排

平台不再要求被调起的 agent CLI 自己创建子代理，而是由 OpenDeepSea 后端直接创建父子 agent run：

- 父 run 负责调度和汇总。
- 子 run 是 OpenDeepSea 一等 `agent_runs`。
- 子 run 绑定 `task_id`、`parent_run_id`、`role`、`status`、`started_at`、`completed_at`。
- 子 run 的 ACP trace 按现有链路投影为任务 runtime/diff/timeline 事件。

这种方案最可控，也最符合任务工作区需要。

## 目标

1. 用户在时间线中能看到真实子代理的启动、等待、完成、失败和结果摘要。
2. 若 agent 只在正文中声明子代理，但没有结构化事件，系统应明确显示诊断，不把文本当成真实子代理。
3. Superpowers `subagent-driven-development` 不再依赖 CLI 是否开放内部子代理工具；OpenDeepSea 可以原生编排实现者、规格审查者、代码质量审查者。
4. ACP tool call 归一化继续保留，作为兼容外部 CLI 原生子代理事件的轻量路径。
5. transcript 旁路读取只作为可选增强，不作为主链路。

## 非目标

- 不把 ACP 标准扩展为私有协议。
- 不要求所有 CLI 都支持同一套子代理工具名。
- 不在前端通过文本正则伪造子代理完成态。
- 不让子代理事件依赖 assistant 正文可信度。

## 推荐架构

采用三层方案：

### 第一层：ACP-first 归一化

继续解析 ACP `tool_call` / `tool_call_update` / `tool_result`：

- `Task`
- `spawn_agent`
- `dispatch_agent`
- `wait_agent`
- `agent_result`
- `subagent_result`

这些事件统一映射为 `subagent_*` timeline event，并投影到 `task_events` 的 runtime 层。

### 第二层：OpenDeepSea 原生子代理编排

为任务动作 `subagent_execution` 增加平台级执行计划：

1. 后端解析当前任务阶段，生成子代理角色：
   - `implementer`
   - `spec_reviewer`
   - `code_quality_reviewer`
   - 可选 `final_reviewer`
2. 每个子代理通过现有 ACP adapter 创建独立 `agent_run`。
3. `agent_runs` 增加父子关系字段，或新增 `agent_run_links` 表：
   - `parent_run_id`
   - `child_run_id`
   - `relationship`
   - `role`
   - `task_id`
4. 子代理 run 的状态变化生成任务 runtime event：
   - `subagent_started`
   - `subagent_progress`
   - `subagent_completed`
   - `subagent_failed`
5. 前端时间线按父 run 分组展示子代理执行树。

### 第三层：文本声明诊断

若 assistant 正文出现“子代理已派发”“等待子代理”“审查子代理已启动”等声明，但当前 message/run 没有 `subagent_*` 事件，也没有子 run 关联，则显示诊断：

> 检测到文本中的子代理声明，但没有收到结构化子代理事件或子代理 run。

该诊断只用于解释缺口，不改变任务状态。

## 数据模型建议

优先新增轻量关系表，减少对 `agent_runs` 迁移风险：

```sql
CREATE TABLE agent_run_links (
  id TEXT PRIMARY KEY,
  room_id TEXT NOT NULL,
  task_id TEXT,
  parent_run_id TEXT NOT NULL,
  child_run_id TEXT NOT NULL,
  relationship TEXT NOT NULL,
  role TEXT NOT NULL,
  created_at INTEGER NOT NULL
);
```

如果后续需要更强查询性能，再把 `parent_run_id` 和 `role` 冗余到 `agent_runs`。

## 前端展示

任务时间线展示两类来源：

1. ACP 归一化子代理事件。
2. OpenDeepSea 原生 `agent_run_links` 推导出的子代理事件。

展示规则：

- 父 run 下展示子代理卡片。
- 子代理卡片显示角色、agent 名称、状态、开始/结束时间、摘要。
- 子代理完成后显示结果摘要和关键验证命令。
- 子代理失败时显示错误和可重试入口。
- 只有文本声明时显示诊断，不显示为真实子代理。

## 验收标准

1. 当 ACP trace 中有 `spawn_agent` 和 `wait_agent` 时，时间线显示子代理启动和完成。
2. 当 OpenDeepSea 原生编排子代理时，即使 ACP CLI 不支持内部子代理工具，时间线也显示完整子代理 run。
3. 当只有 assistant 文本声明子代理、没有结构化事件和子 run 时，时间线显示诊断。
4. `IQ7qudivzq34` 这类历史消息能解释为“文本声明但无结构化事件”，不会误显示虚假子代理结果。
5. `subagent_execution` 完成后，任务事件能追溯每个子代理 run 的状态和结果。

## 风险

- 原生编排会改变 `subagent_execution` 的执行责任边界，需要避免和 CLI 内部子代理重复派发。
- 多子代理并行会带来写文件冲突，需要继承现有 ownership / scope_write 规则。
- transcript 旁路读取格式不稳定，只适合作为后续增强。
- 历史数据只能补诊断，不能恢复从未记录的子代理结果。

## 推荐推进顺序

1. 先补诊断：文本声明子代理但无结构化事件/子 run 时明确展示。
2. 增加 `agent_run_links` 和后端投影能力。
3. 将 `subagent_execution` 改为 OpenDeepSea 原生编排实现者与两级审查者。
4. 前端展示父子 run 执行树。
5. 最后评估是否接入 Claude Code / Codex transcript 旁路读取。

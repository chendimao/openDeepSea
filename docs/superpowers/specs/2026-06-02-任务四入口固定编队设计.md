# 任务四入口固定编队设计

## 背景

当前任务创建后，任务卡和任务详情顶部主要展示 `Owner`、`Status`、`Priority`、`ETA` 等静态元信息。实际用户在任务创建后的核心动作不是查看这些字段，而是决定任务进入哪种 Superpowers 流程：直接执行、头脑风暴、编写计划或子代理执行。

现有 `task_execution.next_steps` 支持 planner 在执行结果返回后继续提出下一轮派发。这会导致执行中途临时拉入新智能体，例如实现完成后再由 planner 输出 `review-validator`。这种模式让任务边界、参与智能体和阶段顺序在运行过程中漂移，也容易出现自动续派发断链。

## 目标

任务创建后，将任务入口明确改为四个动作：

- 开始执行
- 头脑风暴
- 编写计划
- 子代理执行

点击 `开始执行` 时，系统必须一次性固定完整编队和阶段计划，包括参与智能体、阶段顺序、并行/串行关系。后续执行只能推进这个锁定计划，不允许中途新增智能体。

## 非目标

- 不重新设计完整任务工作区布局。
- 不保留旧的“按建议继续”执行语义。
- 不让 planner 在 executor 完成后临时追加未在初始计划中的 agent。
- 不把 Superpowers 技能内容硬编码进前端。

## 用户体验

### 四个入口

任务创建消息和任务详情顶部的四个元信息块改为四个动作入口：

1. `开始执行`
   - 适用于需求已经明确、可以直接进入完整开发闭环的任务。
   - 点击后直接创建固定编队 workflow。
   - 系统显示启动中、运行中、失败、完成等状态。

2. `头脑风暴`
   - 适用于需要澄清需求、边界、验收标准或多方案选择的任务。
   - 通过 ACP 调用 Superpowers `brainstorming`。
   - 阶段完成后产出 spec 文件，例如 `docs/superpowers/specs/YYYY-MM-DD-topic-design.md`。

3. `编写计划`
   - 适用于已有 spec，需要产出可执行 implementation plan 的任务。
   - 通过 ACP 调用 Superpowers `writing-plans`。
   - 输入以上一步 spec 为准。
   - 如果没有 spec，入口进入 `blocked`，提示先执行头脑风暴。

4. `子代理执行`
   - 适用于已有 implementation plan，需要按计划执行的任务。
   - 通过 ACP 调用 Superpowers `subagent-driven-development`。
   - 输入以上一步 plan 为准。
   - 如果没有 plan，入口进入 `blocked`，提示先执行编写计划。

### 状态反馈

四个入口都需要展示可见状态：

- `idle`：可点击
- `queued`：已排队
- `running`：运行中
- `failed`：运行失败
- `completed`：阶段完成
- `blocked`：缺少前置产物或边界不清

入口按钮不能只触发请求后静默等待。对应 agent run、workflow step 或系统消息必须能在任务记录中看到。

## 后端执行契约

### 固定编队计划

`开始执行` 不再使用可无限续派发的 `task_execution.next_steps` 作为核心执行契约，而是生成一个可锁定的任务 workflow 计划。

`开始执行` 的初始 roster 默认必须包含 executor、reviewer 和 acceptor。任务类型会影响 executor 数量和是否并行，但不会省略审查与验收阶段。

示例结构：

```json
{
  "task_workflow": {
    "mode": "fixed_roster",
    "entry_action": "start_execution",
    "locked": true,
    "agents": [
      { "agent_id": "backend-executor", "role": "executor" },
      { "agent_id": "reviewer", "role": "reviewer" },
      { "agent_id": "acceptor", "role": "acceptor" }
    ],
    "stages": [
      {
        "id": "execute",
        "agent_ids": ["backend-executor"],
        "parallel": false,
        "goal": "实现任务需求"
      },
      {
        "id": "review",
        "agent_ids": ["reviewer"],
        "parallel": false,
        "goal": "审查实现结果与风险"
      },
      {
        "id": "acceptance",
        "agent_ids": ["acceptor"],
        "parallel": false,
        "goal": "根据验收标准判断任务是否完成"
      }
    ]
  }
}
```

### 锁定规则

当 `locked` 为 `true`：

- 后续 planner 只能更新阶段状态、总结、失败原因和返工目标。
- 后续 planner 不能新增 `agents` 中不存在的智能体。
- 后续 planner 不能新增不在 `stages` 中的阶段。
- 如果执行中发现缺少必要智能体，workflow 必须进入 `blocked` 或 `needs_boundary_confirmation`，等待用户修改计划或重新开始。
- 返工只能回到已锁定 roster 内的 executor/reviewer/acceptor。

### 入口到 Superpowers 阶段的映射

| 入口 | ACP / Superpowers 阶段 | 前置条件 | 产物 |
| --- | --- | --- | --- |
| 开始执行 | 固定编队 workflow | 任务已创建 | 锁定 workflow plan |
| 头脑风暴 | `brainstorming` | 任务已创建 | design spec |
| 编写计划 | `writing-plans` | 已有 design spec | implementation plan |
| 子代理执行 | `subagent-driven-development` | 已有 implementation plan | 代码改动、测试、review 记录 |

## 前端展示

任务卡和任务详情顶部应优先展示动作入口，而不是静态元信息。

当前类似：

- `Status`
- `Owner`
- `Priority`
- `ETA`

改为：

- `开始执行`
- `头脑风暴`
- `编写计划`
- `子代理执行`

静态元信息可以降级到详情区、tooltip 或任务记录中，不再占据主操作区。

每个入口需要：

- 明确 icon。
- 明确 loading 状态。
- 明确 disabled 原因。
- 明确失败提示。
- 与任务记录中的 workflow step / agent run 对应。

## 错误处理

- `开始执行` 生成固定编队失败：创建系统消息，入口状态为 `failed`。
- 初始 roster 无可用 executor：进入 `blocked`，提示需要添加可执行智能体。
- 初始 roster 无 reviewer 或 acceptor：可根据任务风险阻塞，或使用默认全局 `reviewer` / `acceptor` 加入初始 roster，但必须发生在启动前。
- `编写计划` 缺少 spec：入口进入 `blocked`，提示先运行头脑风暴。
- `子代理执行` 缺少 plan：入口进入 `blocked`，提示先运行编写计划。
- 执行中 planner 请求新增 agent：拒绝新增，将 workflow 标记为 `blocked` 并记录原因。

## 验收标准

- 任务创建后，任务卡或任务详情顶部展示四个动作入口：`开始执行`、`头脑风暴`、`编写计划`、`子代理执行`。
- 点击 `开始执行` 后，后端一次性生成并锁定完整编队和阶段计划。
- `开始执行` 不再依赖中途 planner 输出的新 `next_steps` 来追加智能体。
- 执行中如果 planner 输出未在初始 roster 中的 agent，系统不会自动拉入该 agent。
- `头脑风暴` 会通过 ACP Superpowers `brainstorming` 产出 spec。
- `编写计划` 会通过 ACP Superpowers `writing-plans` 基于 spec 产出 plan。
- `子代理执行` 会通过 ACP Superpowers `subagent-driven-development` 基于 plan 执行。
- 四个入口都有 loading、失败、完成和阻塞状态。
- 任务记录中能追踪每个入口触发的 run、产物和错误。

## 风险与权衡

- 固定编队会降低中途自适应能力，但能显著提升任务边界和执行可追踪性。
- 需要后端明确区分“启动时扩编”和“运行中扩编”。前者允许，后者禁止。
- 若 planner 初始判断不足，任务可能更早进入 blocked；这是可接受的，因为用户可以明确修改计划，而不是让系统静默漂移。
- 当前已有 Superpowers runtime 和阶段提示可复用，实施重点应放在入口映射、锁定契约和 UI 状态反馈。

## 设计决策

- `开始执行` 始终包含 executor、reviewer 和 acceptor。低风险任务可以减少 executor 数量，但不跳过审查和验收。
- `编写计划` 缺少 spec 时直接阻塞并提示先运行头脑风暴。
- `子代理执行` 缺少 plan 时直接阻塞并提示先运行编写计划。
- 不自动串联缺失的前置阶段，避免用户不清楚当前实际运行阶段。

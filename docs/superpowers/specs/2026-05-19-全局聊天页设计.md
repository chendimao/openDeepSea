# 全局聊天页设计

## 背景

当前 OpenDeepSea 的聊天能力主要绑定在项目群聊中，面向项目开发、任务创建、workflow 编排和智能体协作。用户还需要一个不绑定项目的全局聊天入口，用于查询全局记忆、项目记忆和系统配置，也可以进行普通对话。

## 目标

- 在侧边栏新增独立的全局聊天页面。
- 全局聊天不绑定项目、房间、任务或 workflow。
- 每次用户提问默认自动检索可用记忆和配置摘要。
- 聊天会话和消息需要持久化。
- 用户可以手动把消息或片段保存为全局记忆。

## 非目标

- 第一版不触发项目任务创建。
- 第一版不启动 workflow。
- 第一版不邀请或调度房间智能体。
- 第一版不做自动记忆沉淀。
- 第一版不做复杂右侧引用面板，引用来源先放在 assistant 消息下方折叠展示。

## 产品形态

采用独立全局聊天页方案。

- 侧边栏新增“聊天”导航入口。
- 路由建议为 `/chat`。
- 页面默认打开最近会话。
- 没有会话时显示空状态，用户发送第一条消息后创建会话。
- 支持新建会话、切换历史会话、删除或归档会话。
- 主区域是消息流和输入框。
- assistant 消息下方展示“引用来源”折叠区，包括命中的记忆和配置摘要。

## 数据模型

### 全局聊天会话

新增 `global_chat_sessions`。

字段建议：

- `id`
- `title`
- `archived`
- `created_at`
- `updated_at`

会话标题可由用户第一条消息截断生成，后续可支持重命名。

### 全局聊天消息

新增 `global_chat_messages`。

字段建议：

- `id`
- `session_id`
- `role`: `user | assistant | system`
- `content`
- `status`: `completed | failed`
- `metadata`
- `created_at`

`metadata` 用于保存：

- `memory_refs`
- `config_refs`
- `error`
- `model_chat`

### 全局记忆

扩展 `memory_entries.scope`，新增 `global`。

全局记忆特点：

- 不绑定 `project_id`、`room_id`、`room_agent_id` 或 `task_id`。
- 适合保存用户偏好、通用约定、跨项目经验、系统使用习惯。
- 全局聊天默认检索全局记忆。
- 项目/房间上下文是否检索全局记忆可作为后续增强，不纳入第一版强制范围。

## 后端 API

建议新增全局聊天 API：

- `GET /global-chat/sessions`
- `POST /global-chat/sessions`
- `PATCH /global-chat/sessions/:id`
- `DELETE /global-chat/sessions/:id`
- `GET /global-chat/sessions/:id/messages`
- `POST /global-chat/sessions/:id/messages`
- `POST /global-chat/messages/:id/save-memory`

`POST /global-chat/sessions/:id/messages` 负责：

1. 保存用户消息。
2. 自动检索上下文。
3. 调用系统模型。
4. 保存 assistant 回复。
5. 返回 assistant 消息和引用来源。

第一版可使用普通 HTTP 响应；如果后续需要逐字流式输出，再扩展 WebSocket 或 SSE。

## 检索策略

默认每次提问自动检索上下文。

检索范围：

- 全局记忆。
- 所有项目记忆。
- 相关房间/任务记忆。
- 非敏感系统配置摘要。

排序建议：

1. pinned 全局记忆。
2. 与用户输入关键词相关的全局记忆。
3. 与用户输入关键词相关的项目、房间、任务记忆。
4. 最近更新的配置摘要。

需要设置数量和字符预算，避免 prompt 过长。第一版建议：

- 全局记忆最多 8 条。
- 跨项目相关记忆最多 12 条。
- 配置摘要固定输出非敏感字段。
- 总上下文字符数设置上限。

## 配置信息安全

全局聊天可以查询配置，但不能暴露敏感值。

允许进入模型上下文的信息：

- 是否已配置模型。
- 模型名称。
- base URL 的脱敏摘要。
- 是否已设置 API Key。
- API Key preview。
- 默认 workflow 配置名称或 id。
- 全局 routing、interaction、auto distill 等开关。

禁止进入模型上下文的信息：

- 完整 API Key。
- token。
- 未脱敏凭证。
- 不必要的本机绝对敏感路径。

## 模型调用

复用现有系统模型配置和 `model_chat` runtime scope。

全局聊天的系统提示应明确：

- 这是 OpenDeepSea 的全局聊天助手。
- 可以基于检索到的记忆和配置回答。
- 如果上下文没有证据，应说明不确定。
- 不要声称已经修改文件、创建项目或执行 workflow。
- 用户提出开发需求时，只能解释、建议或引导去项目群聊/任务流程。

## 保存为记忆

用户可以在消息操作中选择“保存为全局记忆”。

第一版保存规则：

- 默认保存到 `scope = global`。
- 由用户手动触发。
- 不自动从所有全局聊天中抽取记忆。
- 保存时可用消息发送者和内容前缀生成默认标题。
- 支持后续编辑记忆标题和内容。

## 前端设计

新增页面：`packages/frontend/src/pages/GlobalChatPage.tsx`。

侧边栏：

- 在现有全局导航中新增“聊天”。
- 使用聊天类图标。
- 不依赖当前项目。

页面布局：

- 左侧为会话列表，宽度固定或响应式折叠。
- 主区域为聊天消息流。
- 底部为输入框。
- assistant 消息展示引用来源折叠区。
- 消息操作包含“保存为全局记忆”。

复用现有组件：

- 消息内容渲染可复用 `MessageContent`。
- 输入体验可参考房间聊天的 composer。
- 空状态风格跟现有页面一致。

## 错误处理

- 模型未配置：页面提示用户去系统设置配置模型。
- 检索失败：仍允许普通聊天，但 system 消息或 toast 提示检索失败。
- 模型调用失败：保存失败的 assistant/system 消息，显示脱敏错误。
- 保存记忆失败：toast 提示，不影响聊天历史。
- 会话不存在或已归档：返回明确错误。

## 测试计划

后端测试：

- 创建、列表、更新、删除或归档全局聊天会话。
- 发送消息后同时持久化 user 和 assistant 消息。
- 默认检索全局记忆和跨项目记忆。
- 配置摘要不包含完整 API Key。
- assistant 消息 metadata 记录引用来源。
- 手动保存消息为全局记忆。

前端测试：

- 侧边栏出现全局聊天入口。
- `/chat` 可以打开并加载会话。
- 发送消息后显示 user 和 assistant 消息。
- assistant 消息展示引用来源。
- 保存为全局记忆按钮调用正确 API。
- 模型未配置时显示可操作提示。

最终验证：

- `npm run test -w @openclaw-room/backend`
- `npm run build`

## 分阶段实施

### 第一阶段

- 数据表和 repo。
- 全局记忆 scope。
- 基础 API。
- 简单 HTTP 回复。
- 前端页面和侧边栏入口。
- 手动保存为全局记忆。

### 第二阶段

- 流式回复。
- 引用来源侧栏。
- 会话重命名。
- 更细的检索排序和过滤。

### 第三阶段

- 全局聊天对项目/房间的跳转建议。
- 全局记忆在项目群聊中的可选注入。
- 更完整的引用审计和上下文调试视图。

## 验收标准

- 用户可以从侧边栏进入全局聊天页。
- 用户可以在不选择项目的情况下发起普通聊天。
- 每次回答默认检索全局记忆、项目记忆和非敏感配置摘要。
- 刷新页面后会话和消息仍存在。
- assistant 回复能显示本次使用的引用来源。
- 用户可以手动把消息保存为全局记忆。
- API Key 等敏感配置不会出现在模型上下文、响应或消息 metadata 中。

# 子代理进度 ACP 展示设计

## 背景

在群聊 `xPR1AKSuZxq9` 的任务 `_bWdakVJfFcN` 中，`frontend-executor` 正文显示已经派发并等待子代理，但页面只展示父执行器的普通 ACP 执行过程，没有展示子代理进度或结果。数据库证据显示：

- `agent_runs` 只有父执行器 run，没有 parent/child run 关系字段。
- `task_events` 只投影了父 run 的工具调用、计划更新和运行事件，没有 `subagent_*` 语义事件。
- 该 run 的正文包含“子代理已派发”“子代理已返回 DONE”，但结构化 trace 中没有 `spawn_agent` 或 `wait_agent` 工具事件。
- `subagent_execution` 和 `auto_advance` 曾早于后续 runtime event 被标记完成，旧任务仍保持 `todo`。

## 目标

让 ACP 时间线能够完整、可追溯地表达子代理状态：当 provider 返回结构化子代理事件时，后端应归一化并投影到任务事件；当 provider 只在正文里提到子代理时，前端应明确显示诊断，说明缺少结构化 ACP 子代理事件。

## 非目标

- 不实现真正的多进程子代理调度器。
- 不从普通文本伪造完整子代理执行结果。
- 不迁移历史数据库结构来补 parent/child run 表关系。
- 不修改当前已有结构化 JSON 样式任务的实现内容。

## 方案

### 后端事件语义

扩展 `AgentTimelineEventType`，新增：

- `subagent_started`
- `subagent_progress`
- `subagent_completed`
- `subagent_failed`

在 `protocol-events.ts` 中识别 provider 可能返回的子代理工具或事件：

- `spawn_agent`、`dispatch_agent`、`Task` 等开始类事件归一化为 `subagent_started`。
- `wait_agent`、`subagent_result`、`agent_result` 等完成类事件按状态归一化为 `subagent_completed` 或 `subagent_failed`。
- 其他包含 `subagent` 且没有终态的事件归一化为 `subagent_progress`。

事件 payload 至少保留：

- `child_agent_id`
- `model`
- `reasoning_effort`
- `parent_run_id`
- `tool_call_id`
- `summary`
- `result`

### 任务事件投影

`projectTimelineEventToTaskEvent` 对 `subagent_*` 事件投影到 `task_events`，类型仍使用现有 `runtime_event`，避免扩大任务事件枚举和数据库迁移范围。payload 中增加：

- `timeline_type: subagent_started | subagent_progress | subagent_completed | subagent_failed`
- `parent_run_id`
- `child_agent_id`
- `subagent_status`

这样任务工作区可以和普通 runtime event 一起回放，同时前端可以按 `timeline_type` 识别。

### 前端展示

`AgentTimeline` 渲染新增子代理类事件：

- 摘要显示为“子代理启动 / 子代理进度 / 子代理完成 / 子代理失败”。
- 详情中展示子代理、模型、推理等级、结果摘要和原始结构化字段。
- 子代理事件使用独立样式标记，但仍保持现有 timeline card 交互。

同时新增诊断：如果某条父执行器正文或 trace assistant message 中出现“子代理已派发、等待子代理、子代理已返回”等文本，但 trace 中没有任何 `subagent_*` 事件，则显示“检测到文本中的子代理声明，但 ACP 未返回结构化子代理事件”。这避免用户误以为页面漏渲染了不存在的数据。

### 自动推进状态

新增一个后端补偿判断：当任务已经存在 `auto_advance completed` 且委派动作为终态动作，但任务仍为 `todo` 时，任务列表响应应能补偿为完成状态，或至少让自动推进入口不可点击。优先采用服务端补偿任务状态，避免前端重复实现判断。

## 验收标准

- 后端测试覆盖结构化 `spawn_agent` / `wait_agent` 事件归一化。
- 后端测试覆盖 `subagent_*` timeline event 能投影为 task runtime event。
- 前端测试覆盖子代理事件的可见展示。
- 前端测试覆盖“正文提到子代理但没有结构化事件”诊断。
- `_bWdakVJfFcN` 这种历史任务不应继续显示可自动推进状态。
- `npm run build` 能完成，若因工作区既有无关改动失败，必须说明失败文件和原因。

## 风险

- 当前 Codex ACP 未必返回结构化子代理事件，第一版只能对未来结构化事件完整展示，对历史文本只能诊断。
- 工作区已有大量无关改动，提交时必须精确暂存。
- `AgentTimelineEventType` 是前后端共享概念，需要同时更新后端和前端类型。

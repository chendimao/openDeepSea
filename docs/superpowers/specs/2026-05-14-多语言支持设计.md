# 多语言支持设计

## 背景

OpenClaw Room 当前前端文案以中文硬编码为主，少量技术词保持英文。用户需要新增中文和英文两种界面语言，默认中文，并明确语言类型使用 `zh` / `en`。

本设计只覆盖前端界面国际化，不改变后端 API、数据库 schema、项目数据、聊天消息、任务内容或 agent 输出内容。

## 目标

1. 支持 `zh` 和 `en` 两种界面语言。
2. 默认语言为 `zh`。
3. 用户切换语言后持久保存，下次打开应用沿用选择。
4. 覆盖主要前端静态文案，包括页面标题、按钮、弹窗、toast、placeholder、aria-label、枚举标签、相对时间。
5. 保持现有 React、TypeScript、Vite 和 Tailwind 技术栈，不引入重量级 i18n 依赖。

## 非目标

1. 不翻译用户输入内容，例如项目名、房间名、任务标题、任务描述和聊天消息。
2. 不翻译 agent 或 provider 品牌名，例如 OpenClaw、Codex、Claude Code、OpenCode。
3. 不新增浏览器语言自动探测，避免默认语言偏离“默认中文”的要求。
4. 不改变后端错误结构；后端返回的动态错误原文可保持原样。

## 推荐方案

采用轻量自研 i18n 层：

- 新增前端 i18n 模块，定义 `Locale = 'zh' | 'en'`。
- 通过 React Context 提供当前语言、语言切换函数和翻译函数。
- 使用集中式字典维护中文和英文文案。
- 使用 `localStorage` 保存语言选择，建议 key 为 `openclaw-room-locale`。
- 在应用入口包裹 `I18nProvider`，并在 `AppShell` 中提供语言切换控件。

不引入 `i18next` 或 `react-i18next`。当前项目只有双语需求，自研方案更轻，依赖面更小，也更符合本地优先工具的维护成本。

## 架构设计

### i18n 模块

新增 `packages/frontend/src/lib/i18n.tsx`，职责包括：

- 定义 `Locale`、`isLocale()` 和默认语言。
- 从 `localStorage` 读取初始语言，无有效值时返回 `zh`。
- 提供 `I18nProvider`。
- 提供 `useI18n()`，返回：
  - `locale`
  - `setLocale`
  - `t(key, params?)`
  - `formatRelativeTime(timestamp)`
- 在语言变化时同步：
  - `localStorage`
  - `document.documentElement.lang`

新增或同文件维护 `messages` 字典：

```ts
type Locale = 'zh' | 'en';
type MessageKey = keyof typeof zhMessages;
type Messages = Record<MessageKey, string>;
```

中文字典作为类型来源，英文字典必须满足同一组 key，避免漏翻译。

### 翻译函数

`t(key, params?)` 用于简单字符串替换：

```ts
t('project.stats.rooms', { count: 3 })
```

参数替换只支持 `{name}` 这类简单占位，不做复杂 ICU 规则。英文复数第一版可通过专门 key 或简单 helper 处理，避免过度设计。

### 相对时间

当前 `relativeTime()` 返回中文。新增 locale-aware 格式化：

- `zh`：
  - 刚刚
  - `{count}分钟前`
  - `{count}小时前`
  - `{count}天前`
- `en`：
  - just now
  - `{count} min ago`
  - `{count} hr ago`
  - `{count} days ago`

为减少大范围重构，可保留原工具函数，新增 `formatRelativeTime()` 从 i18n context 暴露给组件使用。

## UI 入口

语言切换控件放在侧边栏底部，靠近主题切换控件：

- 两个选项：`中文` / `EN`
- 当前语言使用 `aria-pressed`
- 控件不进入路由，不触发数据刷新

后续如果系统设置弹窗已有稳定信息架构，可以再把语言作为系统级偏好同步放入设置弹窗；第一版以侧边栏入口为准。

## 数据流

1. 应用启动时读取 `openclaw-room-locale`。
2. 无有效值时使用 `zh`。
3. `I18nProvider` 将 `locale` 和 `t()` 传给子组件。
4. 用户点击语言切换控件后更新 `locale`。
5. effect 将语言写入 `localStorage`，并设置 `document.documentElement.lang`。
6. React 重新渲染所有使用 `useI18n()` 的组件。

## 错误处理

- 如果 `localStorage` 中的语言值不是 `zh` 或 `en`，忽略并回退到 `zh`。
- 如果某个 key 在开发中漏掉，TypeScript 应尽量通过字典类型发现。
- 运行时不显示 key；必要时 `t()` 回退到中文文案。
- API 动态错误保持原始 message，避免误翻译技术细节。

## 测试与验证

最低验证：

1. 运行 `npm run build`，确认 TypeScript 和 Vite 构建通过。
2. 手动切换 `zh` / `en`，刷新页面后确认语言选择保持。
3. 抽查以下视图无明显中英混杂和布局溢出：
   - 项目总览
   - 项目详情
   - 房间页面
   - 创建项目弹窗
   - 创建任务弹窗
   - 系统设置弹窗
4. 抽查 toast、placeholder、aria-label、相对时间在两种语言下可用。

如新增测试 runner，可补充 i18n 字典 key 完整性测试和 `isLocale()` 单元测试。

## 实施边界

本次实现应集中在 `packages/frontend`。不修改后端 schema、不新增 API、不迁移数据库。

现有工作区包含与本任务无关的未提交改动，实施时不得回滚或覆盖这些改动。

## 验收标准

1. 用户首次打开应用时界面为中文。
2. 用户可在界面上切换到英文。
3. 切换语言后主要 UI 文案立即更新。
4. 刷新页面后仍保持用户选择的语言。
5. `zh` / `en` 是唯一有效语言类型。
6. `npm run build` 通过，或如无法执行则明确说明原因。

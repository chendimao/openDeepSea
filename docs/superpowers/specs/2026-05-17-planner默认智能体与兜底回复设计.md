# Planner 默认智能体与兜底回复设计

## 目标

`planner` 是系统默认内置智能体。所有群聊必须默认包含 `planner`，用户不能从群聊移除它，也不能删除全局 `planner`。聊天协作默认策略改为“兜底回复”，兜底智能体默认为 `planner`。移除“兜底调度”作为可配置能力，并清理历史旧数据。

## 范围

- 后端设置默认值、数据迁移、API 校验和群聊智能体约束。
- 前端设置页、项目路由弹窗、群聊顶部路由提示和类型定义。
- 测试覆盖默认值、迁移清理、群聊自动加入、不可移除和前端可选项。

不保留 `fallback_route` 历史兼容。旧数据统一迁移为 `fallback_reply`，兜底智能体统一补为 `planner`。

## 后端设计

- `MessageRoutingMode` 只保留 `mentions_only | fallback_reply`。
- 系统默认设置改为：
  - `message_routing_mode: 'fallback_reply'`
  - `fallback_agent_id: 'planner'`
- 数据库启动迁移清理：
  - `projects.message_routing_mode = 'fallback_reply'` where old value is `fallback_route`
  - `settings.message_routing_mode = 'fallback_reply'` where old value is `fallback_route`
  - routing mode 为 `fallback_reply` 但 `fallback_agent_id` 为空时补为 `planner`
- `roomRepo.create()` 创建群聊后自动加入全局内置 `planner`。
- `roomAgentRepo.listByRoom()` 或专用 `ensureDefaultPlanner(roomId)` 对旧群聊补齐 `planner`，避免旧房间缺失默认智能体。
- `roomAgentRepo.remove()` 或删除路由在目标为内置 `planner` 时返回 409，不允许移除。
- 设置 API 不再接受 `fallback_route`。
- 如果保存兜底回复且未传 `fallback_agent_id`，后端默认使用 `planner`，但仍校验该 ID 对应有效智能体。

## 前端设计

- 类型 `MessageRoutingMode` 移除 `fallback_route`。
- 设置页和项目消息路由弹窗只展示：
  - 只响应 @
  - 兜底回复
- 系统设置聊天默认值初始为兜底回复 + planner。
- 兜底智能体输入统一改为下拉单选：
  - 系统/项目级使用全局智能体列表，默认选 `planner`。
  - 群聊级使用当前群聊智能体列表；由于后端保障 planner 存在，默认可选。
  - 不再出现手动输入 `agent_id` 的降级输入框。
- 群聊顶部路由提示移除“兜底调度”文案。
- 添加/邀请智能体弹窗中 `planner` 已存在时不可重复邀请；如果列表展示内置模板，应体现它已经内置在当前群聊。

## 验证

- 后端定向测试：
  - 启动迁移会清理 `fallback_route`。
  - 新群聊自动包含 `planner`。
  - 旧群聊读取智能体列表会补齐 `planner`。
  - 移除群聊 `planner` 返回 409。
  - 设置接口拒绝 `fallback_route`。
- 前端构建验证：`npm run build`。
- 若测试脚本可用，运行后端相关 `node --test` 文件。

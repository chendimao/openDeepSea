# ACP 系统上下文检索能力设计

## 背景

当前 ACP 在普通群聊回复中主要依赖 prompt 中的群聊摘要、记忆和最近消息来理解系统状态。这个方式适合补充背景，但不适合作为事实来源。比如用户问“当前群聊有几个任务”时，ACP 可能只从摘要中看到某条“创建任务：test”消息，而没有查询 `tasks` 表，导致回答与数据库真实状态不一致。

OpenClaw Room 已经具备项目、群聊、任务、文件、消息、智能体等结构化数据源，但 ACP 缺少一个受控的系统查询入口。目标是让 ACP 可以按需检索系统整体情况，并基于结构化结果进行分析，而不是依赖提前塞入 prompt 的摘要。

## 目标

1. 让 ACP 能准确了解 OpenClaw Room 的系统状态，包括项目、群聊、任务、智能体、文件、消息、记忆和工作流。
2. 将精确事实查询与语义检索分离，避免用搜索结果回答计数类问题。
3. 提供只读、受控、可审计的系统上下文工具，禁止 ACP 直接执行原始 SQL 或随意读取数据库。
4. 支持从最小可用能力逐步演进到完整系统知识层。
5. 返回可追溯结果，包含来源、作用域、生成时间、计数和引用对象。

## 非目标

1. 不让 ACP 直接访问 SQLite 原始表或执行任意 SQL。
2. 不把完整系统状态、完整消息历史或完整 trace 直接塞进 prompt。
3. 不在第一阶段实现 embedding 或复杂向量检索。
4. 不把语义检索结果当作唯一事实来源。
5. 不索引或暴露 API Key、完整 stderr、raw trace、ACP session id 等敏感字段。

## 核心原则

### 事实查询和语义检索分离

精确问题走确定性查询。例如：

- 当前群聊有几个任务
- 当前项目有几个群聊
- 某个群聊有哪些智能体
- 当前群聊有哪些文件

模糊问题走系统上下文搜索。例如：

- 哪个群聊在做测试菜单相关内容
- 找一下模型可用性测试相关上下文
- 这个项目最近主要在做什么

### ACP 只能使用领域级只读能力

ACP 可以调用 `getRoomOverview`、`searchSystemContext`、`listRoomTasks` 等领域级工具，但不能直接访问 `SELECT * FROM settings` 这类底层能力。

### 结果必须可追溯

每次查询返回：

- `source`：数据来自哪个系统能力
- `scope`：查询作用域
- `generated_at`：生成时间
- `counts`：聚合计数
- `results`：结构化对象
- `citations`：可追溯引用

## 分阶段设计

### 阶段 0：事实查询层

目标是先让系统能查准，不引入索引和模型。

新增系统上下文 service，聚合现有 repo：

- `projectRepo`
- `roomRepo`
- `roomAgentRepo`
- `taskRepo`
- `fileRepo`
- `messageRepo`
- `memoryRepo`
- `workflowRepo`

核心能力：

```ts
getSystemOverview(): SystemOverview
getProjectOverview(projectId: string): ProjectOverview
getRoomOverview(roomId: string): RoomOverview
listRoomTasks(roomId: string): TaskSummary[]
listRoomAgents(roomId: string): RoomAgentSummary[]
listProjectFiles(projectId: string, filters?: FileFilters): FileSummary[]
listRoomFiles(roomId: string, filters?: FileFilters): FileSummary[]
```

`getSystemOverview` 返回项目数量、群聊数量、任务数量、文件数量、最近更新时间和项目摘要。

`getProjectOverview` 返回项目名称、描述、路径摘要、群聊数量、任务统计、文件统计、活跃群聊和最近活动。

`getRoomOverview` 返回群聊名称、描述、所属项目、任务列表摘要、智能体列表、文件列表摘要、最近消息摘要和工作流状态。

验收标准：

- “当前群聊有几个任务”与 `taskRepo.listByRoom(roomId)` 一致。
- “当前有哪些项目”与 `projectRepo.list()` 一致。
- 查询结果不包含敏感字段。
- 结果包含 `source`、`scope`、`generated_at` 和 `counts`。

### 阶段 1：受控 ACP 工具入口

目标是让 ACP 主动查询系统状态，而不是只读 prompt 摘要。

第一步可以提供 CLI 入口：

```bash
npm run openclaw:context -- system-overview
npm run openclaw:context -- project-overview <projectId>
npm run openclaw:context -- room-overview <roomId>
npm run openclaw:context -- list-room-tasks <roomId>
npm run openclaw:context -- list-room-agents <roomId>
npm run openclaw:context -- list-files --project <projectId>
```

后续再把同一套 service 接入 ACP adapter，暴露为只读工具：

```ts
openclaw_context.get_system_overview
openclaw_context.get_project_overview
openclaw_context.get_room_overview
openclaw_context.list_room_tasks
openclaw_context.list_room_agents
openclaw_context.list_files
```

权限边界：

- 工具只能读。
- 工具不能执行原始 SQL。
- 默认 scope 绑定当前 room/project。
- 跨项目查询必须显式请求。
- agent 权限由 room agent runtime boundary 决定。

验收标准：

- ACP 回答系统事实问题时可以先调用 context tool。
- agent run trace 能看到调用了哪个 context tool。
- prompt 中不需要再塞完整系统摘要。
- 失败时返回结构化错误，不让 ACP 猜测。

### 阶段 2：统一系统上下文索引

目标是支持模糊检索和跨实体关联。

新增统一索引文档表：

```ts
system_context_documents
- id
- type
- project_id
- room_id
- source_id
- title
- body
- metadata_json
- updated_at
- deleted_at
```

文档类型：

- `project`
- `room`
- `task`
- `agent`
- `file`
- `message`
- `memory`
- `workflow`

检索接口：

```ts
searchSystemContext(input: {
  query: string;
  projectId?: string;
  roomId?: string;
  types?: SystemContextDocumentType[];
  limit?: number;
}): SystemContextSearchResponse
```

索引内容：

- 项目：名称、描述、统计摘要。
- 群聊：名称、描述、所属项目、目标摘要。
- 任务：标题、描述、状态、来源消息、相关智能体。
- 文件：文件名、类型、来源群聊、来源任务、内容摘要。
- 消息：重要用户消息、任务事件、系统事件、agent 产物摘要。
- 智能体：名称、职责、所在群聊、运行边界摘要。
- 记忆：项目和群聊 memory entries。
- 工作流：工作流名称、状态、阶段和输出摘要。

验收标准：

- 能搜索到“测试菜单”相关任务、消息和文件。
- 能搜索到“模型可用性测试”相关上下文。
- 每条结果包含 `type`、`source_id`、`snippet`、`score` 和 `reason`。
- 搜索作用域不会越过指定 project 或 room。

### 阶段 3：混合检索与重排

目标是提高召回率和语义理解能力。

检索链路：

1. 关键词或 FTS5 初筛。
2. 结构化过滤。
3. BM25 排序。
4. 可选 LLM rerank 或 embedding rerank。

返回模式：

```ts
type SearchMode = 'keyword' | 'semantic' | 'hybrid';
```

建议先使用 SQLite FTS5，再根据中文效果决定是否增加 trigram 或 embedding。模型只负责重排和解释，不负责生成事实。

验收标准：

- 查询“模型可用性测试”能命中标题为“系统设置中，添加模型ai后，需要可以测试模型ai是否可用”的任务。
- 查询“测试菜单”能同时返回任务、消息、文件等不同类型结果。
- LLM rerank 失败时自动回退 keyword 结果。
- 结果解释只引用候选结果，不编造不存在的对象。

### 阶段 4：系统上下文包

目标是让 ACP 获得适合当前意图的上下文组合，而不是只拿搜索列表。

新增：

```ts
buildContextPack(input: {
  intent: 'chat' | 'planning' | 'implementation' | 'review' | 'debug';
  projectId?: string;
  roomId?: string;
  taskId?: string;
  query: string;
  maxChars?: number;
}): ContextPack
```

返回：

```ts
{
  facts: FactItem[];
  relatedProjects: ProjectSummary[];
  relatedRooms: RoomSummary[];
  relatedTasks: TaskSummary[];
  relatedAgents: AgentSummary[];
  relatedFiles: FileSummary[];
  recentMessages: MessageSummary[];
  suggestedNextQueries: string[];
  citations: ContextCitation[];
}
```

用途：

- Planner 判断用户请求属于哪个项目、群聊或任务。
- Executor 接任务前获取相关文件、任务和历史消息。
- Reviewer 获取任务历史、执行记录和相关文件。
- Chat 回答系统状态问题时引用结构化事实。

验收标准：

- ACP 可以回答“这个项目现在主要在做什么”，并列出引用来源。
- ACP 创建新任务前可以识别相似任务。
- ACP 能说明“根据哪些任务、消息和文件判断”。
- 上下文包长度受控，不超过调用方指定限制。

### 阶段 5：增量更新、审计与权限

目标是生产级稳定运行。

能力：

- repo 写入后触发索引更新。
- 任务、文件、消息软删除后同步标记索引。
- 支持全量 rebuild。
- context tool 调用审计。
- agent 级可访问 scope。
- 敏感字段脱敏。
- 检索质量评估集。

权限策略：

```ts
planner:
  可查所有项目概况和当前项目详情。

room agent:
  默认只查当前 room/project。

executor:
  可查当前任务相关 room、files、messages。

reviewer:
  可查任务历史、相关文件、workflow 输出。

普通 chat:
  只查当前 room 可见信息。
```

验收标准：

- 索引与数据库一致性可检测。
- context tool 调用有日志。
- 敏感字段不会出现在 ACP 输出中。
- 可以一键 rebuild index。
- 权限测试覆盖跨项目、跨群聊和敏感字段。

## 返回协议

所有系统上下文能力使用统一返回包装：

```ts
interface SystemContextResponse<T> {
  source: string;
  scope: {
    project_id?: string;
    room_id?: string;
    task_id?: string;
  };
  generated_at: number;
  counts?: Record<string, number>;
  results: T;
  citations?: Array<{
    type: string;
    id: string;
    title?: string;
  }>;
  warnings?: string[];
}
```

搜索响应：

```ts
interface SystemContextSearchResponse {
  query: string;
  mode: 'keyword' | 'semantic' | 'hybrid';
  total: number;
  degraded: boolean;
  degraded_reason: string | null;
  results: Array<{
    id: string;
    type: string;
    project_id: string;
    room_id: string | null;
    title: string;
    snippet: string;
    score: number;
    reason: string;
    metadata: Record<string, unknown>;
  }>;
}
```

## 安全与脱敏

禁止返回或索引：

- OpenAI API Key 和其他凭证。
- ACP session id。
- 完整 stderr。
- raw trace。
- 大段工具输出。
- 本机绝对路径的敏感细节。
- 环境变量值。

允许返回：

- 项目名称、描述、路径 basename 或相对描述。
- 群聊名称、描述。
- 任务标题、状态、描述摘要。
- 文件名、类型、来源。
- 智能体名称、职责、运行边界摘要。
- 消息摘要和可见系统事件。

## 与社区方案的关系

OpenClaw 和 Hermes 的共同模式是：

1. 小型常驻记忆进入 prompt。
2. 大量历史和系统状态通过工具按需检索。
3. SQLite、FTS5、BM25 或 hybrid search 作为索引层。
4. 工具访问受控，检索结果可追溯。

本设计采用同样方向，但检索对象扩展为 OpenClaw Room 的业务实体，包括项目、群聊、任务、文件、消息、智能体、记忆和工作流。

## 推荐落地顺序

1. 实现阶段 0 和阶段 1，让 ACP 先能查准系统事实。
2. 实现阶段 2，用统一文档索引支持跨实体搜索。
3. 实现阶段 3，加入 FTS5、BM25 和可选模型重排。
4. 实现阶段 4，构造面向意图的上下文包。
5. 实现阶段 5，补齐增量更新、审计、权限和评估。

## 开放问题

1. 默认哪些 ACP 智能体可以查跨项目概况。
2. 是否允许普通群聊 agent 查询全局项目列表。
3. 消息索引是否只索引用户消息、系统事件和 agent 文档，还是包含所有 agent 回复摘要。
4. 文件内容是否只索引 agent documents，还是也读取上传文件内容。
5. 第一版工具入口优先 CLI，还是直接接入 ACP adapter。

推荐默认选择：

1. Planner 可以查全局概况，其他 room agent 默认只查当前项目和群聊。
2. 普通群聊 agent 不查全局项目列表，除非用户明确询问且权限允许。
3. 第一版消息索引只包含用户消息、系统事件、任务事件和 agent documents。
4. 第一版文件内容只索引 agent documents；上传文件只索引文件名、类型和来源。
5. 第一版先做 service + CLI，再接 ACP adapter。

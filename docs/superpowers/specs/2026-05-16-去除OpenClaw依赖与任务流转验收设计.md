# 去除 OpenClaw 依赖与任务流转验收设计

## 背景

当前项目已经具备项目、聊天室、ACP agent、任务、工作流、记忆和模型配置页面。阶段 B 的 LangGraph Workflow Runtime 已经可以作为编排入口，但系统仍保留 OpenClaw Gateway、OpenClaw agent 导入、legacy runtime 和部分用户可见品牌文案。

本次目标是先移除 OpenClaw 功能依赖和用户可见品牌，再验证 ACP、自动记忆、任务互通、LangGraph 自动分配流转和聊天页视口布局。

## 范围

### 本次包含

1. 移除 OpenClaw 功能依赖：
   - 后端不再依赖 `openclaw` npm 包。
   - 删除或停用 `packages/backend/src/openclaw/*`。
   - 后端启动不再连接 Gateway。
   - 移除 Gateway 事件绑定和 OpenClaw agent 消息转发。
   - 移除 `/gateway/agents` 和 health 响应中的 Gateway 字段。
   - `dispatcher` 不再支持 `openclaw` backend。

2. 移除用户可见 OpenClaw 品牌：
   - AppShell 不再展示 Gateway 状态入口。
   - Add Agent 不再读取本机 OpenClaw agents，只展示内置 ACP 模板和手动配置。
   - README、页面标题、用户可见中英文文案不再把产品描述为 OpenClaw 集成。

3. 自动记忆改造：
   - 普通 ACP 聊天回复完成后的 LLM 蒸馏改为走现有 OpenAI-compatible 模型配置。
   - 模型未配置时，跳过 LLM 蒸馏但保留确定性记忆。
   - 任务创建时继续写入 `task_summary` 初始记忆。
   - LangGraph 工作流完成验收后继续写入任务总结记忆，并在模型可用时做任务对话蒸馏。

4. 任务互通与启动：
   - `/task ...` 和“新建任务”共用 `createTaskWithConversation`。
   - 两种入口都必须写入任务表，并在任务列表中实时出现。
   - 两种入口都必须在聊天记录中写入用户意图消息和系统任务创建消息。
   - `auto_recommended` 创建后自动启动 LangGraph 工作流。
   - `ask_user` 创建后等待用户启动。
   - 群聊确定性启动命令支持 `/start-task <任务ID>` 和 `开始任务 #<任务ID>`。
   - 任务列表中的启动按钮和群聊命令走同一个 workflow start API。

5. 工作流聊天回写：
   - 启动、计划待审批、任务分配、执行开始/完成、审查、验证、验收、记忆写入、失败和取消都应写入聊天系统消息。
   - 手动新建任务的流转也必须能在聊天记录中追踪。

6. 聊天页视口布局：
   - 聊天工作台必须保持在可视区域内。
   - 消息列表是聊天区主要滚动容器。
   - Composer 固定在聊天面板底部，不被父级 `overflow: hidden` 截断。
   - 右侧任务/记忆/详情面板各自滚动，不挤压聊天底部。

7. 真实浏览器验收：
   - 在真实浏览器中发送 ACP 消息并看到成功回复。
   - 用 `/task` 创建任务，并确认任务列表出现。
   - 用“新建任务”创建任务，并确认聊天记录出现。
   - 验证 `auto_recommended` 自动进入 LangGraph 流转。
   - 验证 `ask_user` 可通过 `/start-task <任务ID>` 或 `开始任务 #<任务ID>` 启动。
   - 验证任务记忆写入。
   - 验证聊天页底部内容可见，滚动条和输入框正常。

### 本次不包含

1. 不重命名 workspace 包名 `@openclaw-room/*`。
2. 不迁移 `OPENCLAW_ROOM_DB` 环境变量和已有 SQLite 文件名。
3. 不迁移 `openclaw_flow_id`、`openclaw_child_task_id` 等历史兼容字段。
4. 不强制迁移旧 localStorage key。
5. 不实现自然语言模糊识别“开始刚才那个任务”。

## 后端设计

### OpenClaw 功能移除

`packages/backend/src/server.ts` 不再导入 Gateway client，也不再后台连接 Gateway。`dispatcher.ts` 删除 OpenClaw streaming 分支，agent 可执行性只来自 ACP 配置或模型直连 fallback。未启用 ACP 的 agent 不应被当作可执行 agent 调用；如果路由选中不可执行 agent，应返回明确系统消息或跳过并记录原因。

`routes.ts` 删除 `/gateway/agents`。`/health` 只返回后端自身状态和必要版本信息，不返回 Gateway 状态。`AddAgentDialog` 所需 agent 来源改为 `/api/agent-templates` 和手动输入。

### 自动记忆

记忆蒸馏不再依赖 Gateway。`memory/distill.ts` 改为使用现有 OpenAI-compatible 模型配置，和模型配置页面保持一致。模型配置缺失时返回空候选，不影响主流程。任务创建记忆和工作流完成任务总结记忆仍是确定性写入，不依赖 LLM。

### 任务创建与启动

`createTaskWithConversation` 保持任务互通主入口。新增启动策略：

- `auto_recommended`：任务创建成功后自动调用 `workflowOrchestrator.start(task.id)`。
- `ask_user`：只创建任务和聊天消息，等待用户启动。

自动启动失败不能回滚任务创建，应写入聊天系统消息说明失败原因，并让用户可以重试。

群聊命令解析在发送消息前处理：

- `/task <title>`：创建任务。
- `/start-task <taskId>`：启动对应任务工作流。
- `开始任务 #<taskId>`：启动对应任务工作流。

启动命令应先创建用户聊天消息，再调用同一 workflow start API，并通过 WebSocket 刷新工作流、任务和聊天消息。

### LangGraph 聊天回写

LangGraph runtime 的节点或 tools 在关键状态变化时调用 `recordTaskEvent`。事件内容应简短、可读，并带上 `task_id`、`workflow_run_id`、`workflow_step_id` metadata，方便前端未来聚合展示。

## 前端设计

### Agent 添加

`AddAgentDialog` 删除 OpenClaw agents 区域和 Gateway 查询。默认展示内置 ACP 模板，保留手动输入 agent 名称、角色、ACP backend 和运行配置。空状态和错误文案只围绕 ACP 配置。

### AppShell

侧边栏删除 Gateway 状态按钮和详情弹窗。产品标题和说明改为不含 OpenClaw 的名称。本次不改内部路由结构。

### 任务与聊天

`RoomPage` 的 composer 继续处理 `/task`，并新增启动命令处理。任务列表中的启动按钮如果尚不存在，应在任务详情或任务卡片上提供清晰入口。启动成功后聊天记录应出现系统事件，任务列表和工作流状态通过现有 query invalidation 和 WebSocket 刷新。

### 布局

布局修复遵循一个原则：外层负责占满视口，聊天消息列表负责滚动，composer 不参与页面滚动。

目标结构：

- `html/body/#root` 使用可用视口高度。
- 应用主区域和 `workspace-root` 使用 `height: 100%` 或 `100dvh` 链路。
- `workspace-grid`、`workbench-panel`、`chat-workspace`、`ChatColumn` 全部保证 `min-height: 0`。
- `Conversation` 拥有 `overflow-y: auto`。
- composer 位于聊天列底部，`flex-shrink: 0`。

## 验证策略

### 自动化验证

1. 后端测试：

```bash
PATH="$(dirname $(mise which node)):$PATH" npm run test -w @openclaw-room/backend
```

2. 根构建：

```bash
PATH="$(dirname $(mise which node)):$PATH" npm run build
```

3. 针对新增行为补充测试：
   - dispatcher 不再调用 OpenClaw backend。
   - memory distill 使用模型 invoker 或模型配置。
   - `auto_recommended` 创建后自动启动 workflow。
   - `/start-task` 和 `开始任务 #id` 启动同一任务。
   - LangGraph 关键事件写入聊天消息。

### 真实浏览器验收

真实浏览器验收必须在本地 dev server 上执行，不能只依赖 API 脚本。

验收步骤：

1. 打开模型配置页面，配置 OpenAI-compatible base URL、API key 和模型。
2. 创建或确认 ACP agent 可用。
3. 在聊天页发送普通 ACP 消息，等待 agent 回复和 agent run 完成。
4. 用 `/task <title>` 创建任务，确认聊天记录和任务列表同时出现。
5. 用“新建任务”创建 `ask_user` 任务，确认聊天记录出现但不自动启动。
6. 发送 `/start-task <任务ID>` 或 `开始任务 #<任务ID>`，确认工作流启动。
7. 创建 `auto_recommended` 任务，确认自动启动 LangGraph。
8. 跟踪任务分配、执行、审查、验证、验收和记忆事件是否写入聊天。
9. 打开记忆面板，确认任务创建记忆和任务完成记忆存在。
10. 在桌面和较小高度视口检查聊天页，确认 composer、底部消息和滚动条可见。

## 风险与处理

1. OpenClaw 移除会影响旧数据中的 `default_runtime=openclaw` agent。
   - 处理：这些 agent 显示为未配置 ACP，不能执行。用户可在 ACP 配置面板补充 backend。

2. 模型配置缺失会导致 LLM 蒸馏不可用。
   - 处理：跳过蒸馏并保留确定性任务记忆，不阻塞 ACP 回复或工作流。

3. LangGraph 自动启动可能失败。
   - 处理：任务创建不回滚，聊天记录写失败系统消息，任务保持可手动重试。

4. 浏览器真实 ACP 调用耗时较长。
   - 处理：测试等待 agent run 状态完成，并记录失败日志；不以固定短超时判断失败。

## 成功标准

1. 代码中没有运行时 OpenClaw Gateway 功能依赖。
2. 用户可见 UI 和 README 不再把产品描述为 OpenClaw 集成。
3. `/task` 和“新建任务”任务互通，任务列表和聊天记录一致。
4. `auto_recommended` 自动启动 LangGraph，`ask_user` 可通过确定性命令或任务 UI 启动。
5. 任务流转关键阶段都能在聊天记录中看到。
6. 自动记忆在模型可用时完成蒸馏；模型不可用时确定性记忆仍写入。
7. 聊天页底部不再被截断，真实浏览器验证通过。

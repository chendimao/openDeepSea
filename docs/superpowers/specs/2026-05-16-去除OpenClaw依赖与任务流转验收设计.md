# 去除 OpenClaw 依赖与任务流转验收设计

## 背景

当前项目已经具备项目、聊天室、ACP agent、任务、工作流、记忆和模型配置页面。阶段 B 的 LangGraph Workflow Runtime 已经可以作为编排入口，但系统仍保留 OpenClaw Gateway、OpenClaw agent 导入、legacy runtime 和部分用户可见品牌文案。

本次目标是先移除 OpenClaw 功能依赖和用户可见品牌，再验证 ACP、自动记忆、任务互通、LangGraph 自动分配流转和聊天页视口布局。

## 范围

### 本次包含

1. 移除 OpenClaw 功能依赖：
   - 后端不再依赖 `openclaw` npm 包。
   - 删除或停用 `packages/backend/src/openclaw/*`。
   - 后端启动不再连接 Gateway。
   - 移除 Gateway 事件绑定和 OpenClaw agent 消息转发。
   - 移除 `/gateway/agents` 和 health 响应中的 Gateway 字段。
   - `dispatcher` 不再支持 `openclaw` backend。

2. 移除用户可见 OpenClaw 品牌：
   - AppShell 不再展示 Gateway 状态入口。
   - Add Agent 不再读取本机 OpenClaw agents，只展示内置 ACP 模板和手动配置。
   - README、页面标题、用户可见中英文文案不再把产品描述为 OpenClaw 集成。

3. 自动记忆改造：
   - 普通 ACP 聊天回复完成后的 LLM 蒸馏改为走现有 OpenAI-compatible 模型配置。
   - 模型未配置时，跳过 LLM 蒸馏但保留确定性记忆。
   - 任务创建时继续写入 `task_summary` 初始记忆。
   - LangGraph 工作流完成验收后继续写入任务总结记忆，并在模型可用时做任务对话蒸馏。

4. 任务互通与启动：
   - `/task ...` 和“新建任务”共用 `createTaskWithConversation`。
   - 两种入口都必须写入任务表，并在任务列表中实时出现。
   - 两种入口都必须在聊天记录中写入用户意图消息和系统任务创建消息。
   - `auto_recommended` 创建后自动启动 LangGraph 工作流。
   - `ask_user` 创建后等待用户启动。
   - 群聊确定性启动命令支持 `/start-task <任务ID>` 和 `开始任务 #<任务ID>`。
   - 任务列表中的启动按钮和群聊命令走同一个带聊天回写的 workflow start API。

5. 工作流聊天回写：
   - 启动、计划待审批、任务分配、执行开始/完成、审查、验证、验收、记忆写入、失败和取消都应写入聊天系统消息。
   - 手动新建任务的流转也必须能在聊天记录中追踪。

6. 聊天页视口布局：
   - 聊天工作台必须保持在可视区域内。
   - 消息列表是聊天区主要滚动容器。
   - Composer 固定在聊天面板底部，不被父级 `overflow: hidden` 截断。
   - 右侧任务/记忆/详情面板各自滚动，不挤压聊天底部。

7. 真实浏览器验收：
   - 在真实浏览器中发送 ACP 消息并看到成功回复。
   - 用 `/task` 创建任务，并确认任务列表出现。
   - 用“新建任务”创建任务，并确认聊天记录出现。
   - 验证 `auto_recommended` 自动进入 LangGraph 流转。
   - 验证 `ask_user` 可通过 `/start-task <任务ID>` 或 `开始任务 #<任务ID>` 启动。
   - 验证任务记忆写入。
   - 验证聊天页底部内容可见，滚动条和输入框正常。

### 本次不包含

1. 不重命名 workspace 包名 `@openclaw-room/*`。
2. 不迁移 `OPENCLAW_ROOM_DB` 环境变量和已有 SQLite 文件名。
3. 不迁移 `openclaw_flow_id`、`openclaw_child_task_id` 等历史兼容字段。
4. 不强制迁移旧 localStorage key。
5. 不实现自然语言模糊识别“开始刚才那个任务”。
6. 不删除历史数据中的 `default_runtime='openclaw'`、`agent_runs.backend='openclaw'`、历史 `session_key` 和前端兼容类型；它们只用于读取旧记录，不再触发运行时 Gateway。

## 后端设计

### OpenClaw 功能移除

`packages/backend/src/server.ts` 不再导入 Gateway client，也不再后台连接 Gateway。`dispatcher.ts` 删除 OpenClaw streaming 分支，agent 可执行性只来自 ACP 配置或模型直连 fallback。未启用 ACP 的 agent 不应被当作可执行 agent 调用；如果路由选中不可执行 agent，应返回明确系统消息或跳过并记录原因。

`routes.ts` 删除 `/gateway/agents`。`/health` 只返回后端自身状态和必要版本信息，不返回 Gateway 状态。`AddAgentDialog` 所需 agent 来源改为 `/api/agent-templates` 和手动输入。

历史兼容字段和枚举只保留为数据兼容层：

- `room_agents.default_runtime='openclaw'`：旧数据可显示，但运行时视为 `none`；新增 agent 默认值改为 `none` 或由 ACP 模板显式写入 `acp`。
- `agent_runs.backend='openclaw'`：旧运行记录可显示，不允许创建新的 OpenClaw run。
- `openclaw_flow_id`、`openclaw_child_task_id`、历史 `session_key`：保留读取和序列化，不作为新逻辑输入。
- 前端类型中如果暂时保留 `'openclaw'`，只能用于旧记录展示和兼容判断，不再出现新增配置入口。

### 自动记忆

记忆蒸馏不再依赖 Gateway。`memory/distill.ts` 改为使用现有 OpenAI-compatible 模型配置，和模型配置页面保持一致。模型配置来源为 `settingsRepo.getLangChainPlannerSettings()` 中的 `openai_base_url`、`openai_api_key`、`langchain_planner_model`；实现上应复用或抽取 `chat-model.ts` / `langchain-planner.ts` 中的 `ChatOpenAI` 构造逻辑，避免重复解析配置。

两条蒸馏路径都受 `auto_distill_enabled` 控制：

- 普通 ACP 聊天回复完成后，如果房间有效、非工作流、run 完成且 `auto_distill_enabled=true`，调用对话蒸馏。
- LangGraph `memory` 节点在写入确定性任务总结记忆后，如果 `auto_distill_enabled=true` 且模型配置可用，调用 `distillFromTask` 或等价模型蒸馏。

模型配置缺失时返回空候选，不 import、不连接 Gateway，也不影响主流程。任务创建记忆和工作流完成任务总结记忆仍是确定性写入，不依赖 LLM。

### 任务创建与启动

`createTaskWithConversation` 保持任务互通主入口。自动启动逻辑放在后端会话创建路径中，而不是由前端二次调用；这样 `/task` 和“新建任务”行为一致，也避免刷新或请求重试导致漏启动。

- `auto_recommended`：任务创建成功后自动调用带聊天回写的启动服务。
- `ask_user`：只创建任务和聊天消息，等待用户启动。

启动服务必须是异步边界清晰的服务：

- HTTP 请求只负责创建任务、创建 workflow run、写入启动事件并排队/触发后台推进；响应不等待 LangGraph 跑到执行、审查或验收结束。
- 成功创建 workflow 后返回 `201` 或 `202`，后续状态通过 WebSocket 和轮询更新。
- 自动启动失败不能回滚任务创建，也不能让任务创建 API 返回失败；应写入聊天系统消息说明失败原因，并让用户可以重试。
- 启动服务在事务中检查 active workflow；已有 active workflow 时不再创建第二个 run。
- `/task` 由聊天消息创建任务时，使用 `source_message_id` 做去重；如果同一 source message 已创建任务，返回既有任务，避免请求重试重复建任务。
- 没有 source message 的手动新建任务暂不引入跨请求 idempotency key；但自动启动必须依赖 active workflow 检查保证不重复启动。

普通 `POST /rooms/:roomId/tasks` 是低级兼容 API。本轮处理方式为：前端不再使用它创建用户可见任务；后端保留接口但文档标注为内部/兼容入口，不提供聊天回写和自动启动。所有 UI 和群聊入口都必须使用 `/rooms/:roomId/tasks/conversation`。

群聊命令解析在发送消息前处理：

- `/task <title>`：创建任务。
- `/start-task <taskId>`：启动对应任务工作流。
- `开始任务 #<taskId>`：启动对应任务工作流。

新增上层启动服务和 API：`POST /rooms/:roomId/tasks/:taskId/workflows/start-with-conversation`。

- 先校验 task 存在，且 `task.room_id === roomId`；project 从 task/room 推导，不从客户端信任。
- 校验失败时不写用户启动意图消息。
- 请求 body 可包含 `content`、`sender_id`、`sender_name`、`source: 'chat_command' | 'task_button' | 'auto_start'`。
- 创建用户启动意图消息，内容保留用户输入或按钮触发文案；`auto_start` 可只写系统消息，不伪造用户消息。
- 调用 `workflowOrchestrator.start(task.id)`。
- 成功后返回 workflow，并依靠已有 WebSocket 广播刷新任务、工作流和聊天。
- 如果已有 active workflow，写入系统消息说明“任务已有运行中的工作流”，返回 409。
- 如果任务已经完成，写入系统消息说明不能重复启动，返回 409。
- 如果工作流启动失败，写入系统消息记录失败原因，返回对应错误码。

任务列表按钮和群聊命令都走该上层 API。底层 `POST /tasks/:id/workflows` 可以保留给内部/测试使用，但 UI 不直接调用它。

计划审批也需要带聊天回写的上层 API：`POST /rooms/:roomId/workflows/:workflowId/approve-plan-with-conversation`。

- 校验 workflow 属于 `roomId`，失败时不写消息。
- 写入用户审批意图消息或按钮触发系统说明。
- 调用现有 `workflowOrchestrator.approvePlan(workflowId, 'user')`。
- 成功后写入计划已批准/继续执行系统消息，并通过 WebSocket 刷新。
- 已非 `awaiting_approval` 时返回 409，并写入可读系统消息。

`auto_recommended` 本轮不自动批准计划。真实浏览器验收必须覆盖“计划待审批 -> 用户批准 -> 继续流转”。如果某个测试计划不需要审批，也要另外创建一个需要审批的任务来验证审批路径。

### LangGraph 聊天回写

LangGraph runtime 的节点或 tools 在关键状态变化时调用 `recordTaskEvent`。事件内容应简短、可读，并带上 `task_id`、`workflow_run_id`、`workflow_step_id` metadata，方便前端未来聚合展示。

事件矩阵：

| 节点/状态 | event_type | 内容要求 |
| --- | --- | --- |
| workflow start | `workflow_started` | 工作流已启动，包含任务标题 |
| context | `workflow_stage_changed` | 已读取项目、房间、任务和可用 agent 上下文 |
| planning started/completed | `workflow_stage_changed` / `workflow_plan_ready` | 开始规划；计划已生成并等待审批或继续 |
| approval pause | `workflow_plan_ready` | 计划待用户审批 |
| approval accepted | `workflow_stage_changed` | 用户已批准计划，继续分配和执行 |
| dispatch | `workflow_assignment_created` | 写明分配到的 agent 或无可用 agent |
| execute started/completed | `workflow_stage_changed` | 执行开始/完成，包含 step/agent 信息 |
| review | `workflow_stage_changed` | 审查开始/完成，包含是否需要修复 |
| verify | `workflow_stage_changed` | 验证开始/完成，包含命令数量和结果 |
| acceptance | `workflow_completed` 或 `workflow_blocked` | 验收通过则完成；不通过则阻塞并说明原因 |
| memory | `workflow_memory_written` | 已写入任务总结记忆；如蒸馏跳过需说明原因 |
| cancel | `workflow_cancelled` | 工作流已取消 |
| fail/block | `workflow_failed` 或 `workflow_blocked` | 失败原因或阻塞原因 |

需要新增 `TaskEventType` 枚举值 `workflow_failed` 和 `workflow_memory_written`。如果实现中选择不新增 `workflow_failed`，必须在计划中明确用 `workflow_blocked` 表示失败，并确保文案和测试能区分 blocked 与 failed。

## 前端设计

### Agent 添加

`AddAgentDialog` 删除 OpenClaw agents 区域和 Gateway 查询。默认展示内置 ACP 模板，保留手动输入 agent 名称、角色、ACP backend 和运行配置。空状态和错误文案只围绕 ACP 配置。

### AppShell

侧边栏删除 Gateway 状态按钮和详情弹窗。产品标题和说明改为不含 OpenClaw 的名称。本次不改内部路由结构。

### 任务与聊天

`RoomPage` 的 composer 继续处理 `/task`，并新增启动命令处理。任务列表中的启动按钮如果尚不存在，应在任务详情或任务卡片上提供清晰入口。启动成功后聊天记录应出现系统事件，任务列表和工作流状态通过现有 query invalidation 和 WebSocket 刷新。

前端命令解析只负责识别命令和调用上层 API，不在前端拼接 workflow 状态。`/start-task` 支持纯 ID；`开始任务 #<任务ID>` 支持 `#` 前后空格。命令解析失败时发送普通聊天消息，不启动任务。

### 布局

布局修复遵循一个原则：外层负责占满视口，聊天消息列表负责滚动，composer 不参与页面滚动。

目标结构：

- `html/body/#root` 使用可用视口高度。
- 应用主区域和 `workspace-root` 使用 `height: 100%` 或 `100dvh` 链路。
- `workspace-grid`、`workbench-panel`、`chat-workspace`、`ChatColumn` 全部保证 `min-height: 0`。
- `Conversation` 或其真实内部 scroll element 拥有 `overflow-y: auto`，并与 `use-stick-to-bottom` 使用同一个 scroll element。
- composer 位于聊天列底部，`flex-shrink: 0`。
- 移动和小高度视口下，避免 `.workspace-grid` 成为聊天区主滚动容器；页面根滚动保持为 0，聊天消息区和右侧/下方任务区各自滚动。

为真实浏览器测试增加稳定定位点：

- 聊天页根：`data-testid="room-page"`。
- 消息滚动容器：`data-testid="chat-scroll-container"`。
- 输入区 composer：`data-testid="chat-composer"`。
- 任务面板：`data-testid="task-panel"`。
- 记忆面板：`data-testid="memory-panel"`。

`use-stick-to-bottom` 和滚动按钮必须绑定到 `chat-scroll-container` 对应的真实 scroll element。

## 验证策略

### 自动化验证

1. 后端测试：

```bash
PATH="$(dirname $(mise which node)):$PATH" npm run test -w @openclaw-room/backend
```

2. 根构建：

```bash
PATH="$(dirname $(mise which node)):$PATH" npm run build
```

3. 针对新增行为补充测试：
   - dispatcher 不再调用 OpenClaw backend。
   - memory distill 使用模型 invoker 或模型配置。
   - `auto_recommended` 创建后自动启动 workflow。
   - `/start-task` 和 `开始任务 #id` 启动同一任务。
   - 自动启动响应不等待 workflow 完整结束；重复启动不会创建第二个 active workflow。
   - 启动 API 使用 `/rooms/:roomId/...` 上下文，跨房间 taskId 被拒绝且不写聊天消息。
   - 审批 API 使用 `/rooms/:roomId/...` 上下文，计划批准会写聊天消息并继续流转。
   - LangGraph 关键事件写入聊天消息。
   - `POST /rooms/:roomId/tasks/conversation` 是 UI 任务创建主入口；低级 `POST /rooms/:roomId/tasks` 不自动写聊天。
   - 旧 `default_runtime='openclaw'` agent 不会触发 Gateway 或创建新的 OpenClaw run。
   - `workflow_memory_written` 和失败/阻塞事件可在消息列表中查询。

### 真实浏览器验收

真实浏览器验收必须在本地 dev server 上执行，不能只依赖 API 脚本。

前置条件：

- 使用 `PATH="$(dirname $(mise which node)):$PATH" npm run dev` 启动本地后端和前端。
- 设置 `LANGGRAPH_WORKFLOW_ENABLED=1`。
- 不启动 OpenClaw Gateway；测试过程中不得出现 Gateway 连接请求。
- 本机至少安装并可调用一个 ACP CLI，优先使用 Codex；如同时验证 OpenCode，则单独记录。
- 测试项目路径使用真实 git 仓库路径，避免 Codex 在非 git 目录失败。
- 模型配置页面使用 OpenAI-compatible base URL、API key 和模型名。

验收步骤：

1. 打开模型配置页面，配置 OpenAI-compatible base URL、API key 和模型。
2. 创建或确认 ACP agent 可用。
3. 在聊天页发送普通 ACP 消息，等待 agent 回复和 agent run 完成。
4. 用 `/task <title>` 创建任务，确认聊天记录和任务列表同时出现。
5. 用“新建任务”创建 `ask_user` 任务，确认聊天记录出现但不自动启动。
6. 发送 `/start-task <任务ID>` 或 `开始任务 #<任务ID>`，确认工作流启动。
7. 如果工作流进入计划待审批，点击计划批准按钮或调用带聊天回写的批准操作，确认聊天记录出现批准事件并继续流转。
8. 创建 `auto_recommended` 任务，确认自动启动 LangGraph；如进入待审批，同样执行批准步骤。
9. 跟踪任务分配、执行、审查、验证、验收和记忆事件是否写入聊天。
10. 打开记忆面板，确认任务创建记忆和任务完成记忆存在。
11. 在桌面和较小高度视口检查聊天页，确认 composer、底部消息和滚动条可见。

浏览器断言：

- `window.scrollY === 0`。
- composer 的 `getBoundingClientRect().bottom <= window.innerHeight`。
- 消息滚动容器 `scrollHeight > clientHeight` 时可以滚到底，最新消息可见。
- 右侧任务/记忆面板内容超长时使用自身滚动，不推动 composer 离开视口。
- 至少保存一张桌面截图和一张小高度视口截图用于人工核对。

状态判定：

- ACP 消息：出现 agent 回复，相关 `agent_run.status` 为 `completed`。
- `/task`：聊天出现用户意图和系统创建消息，任务列表出现同一 ID/标题。
- `ask_user`：创建后没有 active workflow，启动命令后出现 workflow。
- `auto_recommended`：创建后无需二次点击即可出现 running/awaiting/completed workflow。
- `awaiting_approval`：批准操作后聊天出现批准事件，workflow 继续进入 assignment/execute 或后续状态。
- 记忆：记忆面板或 API 中存在 `task_summary`，source 为 `task` 或 `workflow`。
- 失败时收集浏览器 console、network 请求失败、后端日志和相关 agent run error。

## 风险与处理

1. OpenClaw 移除会影响旧数据中的 `default_runtime=openclaw` agent。
   - 处理：这些 agent 显示为未配置 ACP，不能执行。用户可在 ACP 配置面板补充 backend。

2. 模型配置缺失会导致 LLM 蒸馏不可用。
   - 处理：跳过蒸馏并保留确定性任务记忆，不阻塞 ACP 回复或工作流。

3. LangGraph 自动启动可能失败。
   - 处理：任务创建不回滚，聊天记录写失败系统消息，任务保持可手动重试。

4. 浏览器真实 ACP 调用耗时较长。
   - 处理：测试等待 agent run 状态完成，并记录失败日志；不以固定短超时判断失败。

## 成功标准

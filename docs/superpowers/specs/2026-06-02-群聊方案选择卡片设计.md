# 群聊方案选择卡片设计

## 背景

群聊中的头脑风暴消息经常会给出多个方案，例如推荐方案、备选轻量方案和不推荐方案。当前这些方案以普通 Markdown 文本展示，用户需要手动阅读、复制或继续追问，系统也无法可靠记录用户选择。

以消息 `J6wu5SDkYDYDnPBd` 为例，规划师在分析 `@` 文件/目录搜索问题后给出三个方案：

- 推荐方案：把 `@` 文件引用升级成“资源 + 工作区入口”的统一引用器。
- 备选轻量方案：只修复 `@` 空查询不显示目录的问题。
- 不推荐方案：禁止目录出现在 `@` 搜索结果。

这些方案不应该被简单处理成“点击后立即执行”。头脑风暴往往需要多轮边界确认，用户选择方案后只是表达偏好，是否能执行应由后续消息路由和智能体判断。

## 目标

在群聊消息中将明确的方案列表展示成可选择卡片，让用户用一次点击表达“我选择这个方案”，并继续走正常群聊消息链路。

核心目标：

- 让用户更容易比较多个方案。
- 让系统能记录用户选择的来源消息和选项。
- 选择后不直接执行任务或 workflow。
- 保留多轮澄清能力，由后续 planner/intent 判断是否需要继续确认边界或进入执行。

## 非目标

- 不在点击方案卡片后直接创建任务。
- 不在点击方案卡片后直接启动 workflow。
- 不要求所有 Markdown 列表都转成方案卡片。
- 不把方案选择卡片做成 workflow 决策表单；它仍属于群聊对话。

## 方案分类

方案卡片需要带一个轻量成熟度状态，用于帮助后续智能体判断用户选择的含义。

### exploratory

探索性方案。它主要用于讨论方向、排除路径或挑战假设，不能直接执行。

示例：不推荐方案“禁止目录出现在 `@` 搜索结果”。

点击后的语义：用户想继续讨论或确认这条路径，而不是要求实现。

### boundary_needed

边界待确认方案。方向明确，但仍缺关键产品边界、范围或验收标准。

示例：推荐方案“统一资源 + 工作区入口”，还需要确认触发规则、目录注入深度和错误提示策略。

点击后的语义：用户倾向此方案，planner 应继续补齐边界。

### actionable

可执行方案。目标、改动范围和验证方式已经足够清晰。

示例：备选轻量方案“只修复 `@` 空查询不显示目录的问题”。

点击后的语义：用户选择了较明确的实现方向，但仍由消息路由判断是否创建任务或进入 workflow。

## 卡片内容

每张方案卡片展示以下字段：

- 方案标题，例如“推荐方案”“轻量方案”“不推荐方案”。
- 一句话目标，描述该方案解决什么。
- 关键收益，最多三条。
- 风险或遗漏，最多三条。
- 成熟度标签：需继续讨论、需确认边界、可直接实现。
- 操作按钮：选择此方案。

卡片不应占用过多纵向空间。默认展示摘要，长内容可以折叠或截断。

## 点击行为

用户点击“选择此方案”后，前端只发送一条普通群聊消息，不直接执行副作用。

建议消息正文：

```text
我选择：推荐方案「统一资源 + 工作区入口」
```

建议 metadata：

```json
{
  "selected_option_id": "unified_workspace_reference",
  "selected_option_title": "统一资源 + 工作区入口",
  "selected_option_maturity": "boundary_needed",
  "source_message_id": "J6wu5SDkYDYDnPBd",
  "source_type": "brainstorming_option"
}
```

这条消息继续进入现有 `sendMessage -> dispatchUserMessage -> intent/router` 链路。后续是否继续澄清、创建任务或进入 workflow，由 planner 和路由逻辑基于消息内容与方案成熟度判断。

## 数据来源

短期可以从 agent 消息正文中识别“头脑风暴方案”段落。识别范围应保守，优先匹配明确标题：

- 推荐方案
- 备选方案
- 备选轻量方案
- 不推荐方案
- 方案 A / 方案 B / 方案 C

更稳妥的长期方向是在消息 metadata 中写入结构化 `brainstorming_options`，避免依赖 Markdown 文本解析。

建议结构：

```ts
interface BrainstormingOption {
  id: string;
  title: string;
  summary: string;
  benefits: string[];
  risks: string[];
  maturity: 'exploratory' | 'boundary_needed' | 'actionable';
  recommended?: boolean;
}
```

## 渲染规则

消息气泡在解析到 `brainstorming_options` 后，在正文下方展示方案卡片区。若没有结构化 metadata，可对正文做保守解析，解析成功后展示卡片；解析失败则保持普通 Markdown 展示。

为了避免重复信息，卡片展示后正文仍保留原文。后续可以增加“折叠原文中的头脑风暴方案段落”的优化，但不作为第一版要求。

## 后续判断逻辑

选择消息进入后续路由时，建议根据成熟度辅助判断：

- `exploratory`：默认继续群聊讨论。
- `boundary_needed`：默认让 planner 继续确认边界。
- `actionable`：可进入任务草案或 workflow 建议，但仍不由前端直接执行。

成熟度只是提示，不是硬规则。planner 仍应阅读上下文，判断是否缺少验收标准、风险边界或用户确认。

## 用户体验

卡片应符合现有群聊气泡风格：

- 使用紧凑卡片布局，不做大面积营销式卡片。
- 推荐方案使用轻量强调，但不要压制其他选项。
- 不推荐方案可以降权展示，但仍允许选择。
- 按钮文案统一为“选择此方案”。
- 已选择的方案在当前消息中显示已选状态，避免重复点击造成困惑。

## 验收标准

- 含推荐、备选、不推荐方案的头脑风暴消息能展示为多张选择卡片。
- 点击卡片只发送选择消息，不创建任务、不启动 workflow。
- 选择消息包含来源消息和选项 metadata。
- `boundary_needed` 方案被选择后，planner 能继续澄清边界。
- `actionable` 方案被选择后，后续链路可以建议创建任务或 workflow。
- 普通 Markdown 消息不受影响。

## 测试建议

- 单元测试：解析推荐方案、备选轻量方案、不推荐方案。
- 单元测试：不同成熟度的选项 metadata 正确生成。
- 前端渲染测试：方案卡片标题、标签、按钮可见。
- 集成测试：点击卡片后调用 `sendMessage`，但不调用创建任务或 workflow API。
- 回归测试：没有方案段落的 agent 消息仍按普通 Markdown 渲染。

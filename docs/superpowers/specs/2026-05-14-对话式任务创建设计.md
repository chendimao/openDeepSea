# 对话式任务创建设计

## 背景

当前 OpenClaw Room 中，聊天和任务已经具备各自独立能力：

- 聊天消息通过 `POST /rooms/:roomId/messages` 写入 `messages`，再按路由规则分发给 Agent。
- 手动任务通过 `POST /rooms/:roomId/tasks` 写入 `tasks`，并广播任务创建事件。
- `/task` 快捷命令由前端拦截后直接创建任务，没有先写入聊天记录。
- workflow 运行会写入 `workflow_runs`、`workflow_steps`、`task_artifacts` 和 `agent_runs`，但任务创建、计划确认、状态流转没有统一投影到聊天流。

因此用户在房间里会看到两条叙事线：一条是聊天，一条是任务看板。目标是让任务仍保持结构化可追踪，同时让所有任务相关动作在聊天中形成连续上下文。

## 设计结论

采用“对话优先，任务作为对话中的结构化产物”的形式。

聊天是房间的唯一叙事流，任务是可追踪状态对象，workflow 是任务的执行状态机。三者不合并为一张表，但通过来源和事件关联形成统一体验：

- 普通聊天只产生消息，不创建任务。
- 只有当 Agent 把用户问题拆解成计划，并且用户确认或策略允许自动确认时，才创建正式任务。
- 手动创建任务也要在聊天记录中形成对话事件，避免任务看板出现无法追溯来源的卡片。
- 任务运行、计划确认、子任务生成、执行、评审、验收等关键流转都要在聊天中形成简短记录。

## 范围

### 本期包含

- 统一聊天创建任务、手动创建任务、`/task` 快捷命令的行为语义。
- 为任务与消息建立来源关联。
- 为任务生命周期写入聊天事件。
- 支持 Agent 先提出计划，再由用户确认后创建任务。
- 支持手动创建任务时生成用户意图消息和系统确认消息。
- 支持 workflow 关键阶段在聊天中形成记录。

### 本期不包含

- 不让每条普通聊天自动变成任务。
- 不把 `messages`、`tasks`、`workflow_runs` 合并成单一表。
- 不在第一版实现复杂任务意图分类模型。
- 不要求 Agent 的所有中间 token 都转成任务事件；只记录关键节点。
- 不改变现有任务看板作为任务管理主视图的定位。

## 用户体验

### 普通聊天

用户发送普通问题，例如：

> 这个页面感觉有点乱，帮我看看。

系统只写入用户消息，并按当前路由规则交给 Agent。只要 Agent 没有产出结构化计划并进入确认流程，就不创建任务。

### 聊天中形成任务

用户提出可执行诉求后，Agent 先回复计划提案：

> 我建议把这个问题拆成 3 步：检查布局层级、调整关键操作区、补一次视觉回归检查。是否创建任务并开始执行？

此时系统写入一条带 `plan_proposed` metadata 的 Agent 消息或系统事件，但不创建正式任务。

用户确认后，系统创建主任务和必要子任务，并写入聊天事件：

> 已创建任务：优化页面布局体验。包含 3 个子任务，等待启动闭环。

### 手动创建任务

用户点击“新建任务”填写表单后，后端不应只创建任务，还应写入聊天记录：

1. 用户意图消息：`创建任务：优化登录页错误提示`
2. 系统确认消息：`已创建任务 #abc123，优先级 normal，未指派`

如果用户在表单里填写了描述、优先级、指派 Agent 和交互策略，系统确认消息应展示核心字段，但保持简短。

### `/task` 快捷命令

`/task 修复登录错误` 保留为快捷入口，但语义调整为“通过聊天创建任务”：

- 写入用户消息，内容可保留原始命令或规范化为“创建任务：修复登录错误”。
- 创建任务。
- 写入系统确认消息。

不再由前端直接绕过消息流创建任务。

### 任务运行流转

workflow 或任务状态变化时，聊天中写入轻量任务事件：

- `任务进入分析阶段`
- `计划已生成，等待确认`
- `已根据计划创建 4 个子任务`
- `子任务「调整布局结构」由 Codex 开始执行`
- `代码审查发现 2 个问题，等待修复`
- `任务完成，验收通过`

这些事件用于让用户回看上下文，不替代 workflow timeline 的详细展示。

## 数据模型

### `tasks` 扩展

建议增加：

- `source_message_id`：创建任务的来源消息，可为空。
- `created_from`：任务来源，枚举值建议为 `manual`、`chat_plan`、`slash_command`、`workflow_assignment`。

说明：

- 手动创建任务的 `source_message_id` 指向系统生成的用户意图消息。
- 聊天计划创建任务的 `source_message_id` 指向用户原始诉求消息或计划确认消息，具体实现时统一选一种。
- workflow 根据计划创建子任务时，`created_from` 使用 `workflow_assignment`，`parent_task_id` 指向主任务。

### `messages.metadata` 扩展

继续复用现有 `metadata` JSON 字段，增加任务事件信息：

```json
{
  "task_id": "task-id",
  "workflow_run_id": "workflow-id",
  "workflow_step_id": "step-id",
  "event_type": "task_created",
  "origin": "manual"
}
```

`event_type` 建议枚举：

- `plan_proposed`
- `task_created`
- `task_updated`
- `task_status_changed`
- `workflow_started`
- `workflow_stage_changed`
- `workflow_plan_ready`
- `workflow_assignment_created`
- `workflow_blocked`
- `workflow_completed`
- `workflow_cancelled`

### `message_type`

第一版优先复用 `system` 和 `text`，通过 `metadata.event_type` 判断是否按任务事件卡片渲染。

如果后续任务事件 UI 变复杂，再扩展 `message_type = task_event`。这样可以减少第一版数据库迁移和兼容成本。

## 后端服务边界

新增一个统一的任务对话服务，避免多个路由各写各的：

```ts
createTaskWithConversation({
  roomId,
  actor,
  taskInput,
  origin,
  sourceMessageId,
  userFacingContent
})
```

职责：

- 校验房间和项目。
- 必要时创建用户意图消息。
- 创建任务。
- 写入系统确认消息。
- 广播 `message:new` 和 `task:created`。
- 返回创建后的任务和相关消息。

新增一个任务事件记录服务：

```ts
recordTaskEvent({
  roomId,
  taskId,
  workflowRunId,
  workflowStepId,
  eventType,
  content
})
```

职责：

- 写入系统消息。
- 附带任务和 workflow metadata。
- 广播 `message:new`。

workflow orchestrator 只调用该服务记录关键节点，不直接拼接聊天消息细节。

## 前端交互

### 聊天区

聊天区仍按时间线渲染消息，但对带 `metadata.event_type` 的系统消息使用更紧凑的任务事件样式：

- 显示任务标题或短 ID。
- 显示事件文本。
- 可点击跳转或打开任务详情。
- 不使用大卡片，避免打断聊天流。

### 任务创建弹窗

`CreateTaskDialog` 提交时改调统一任务对话 API，而不是直接调普通 `createTask`。

提交成功后：

- 聊天区出现用户意图和系统确认。
- 任务看板出现任务卡。
- Composer 和任务弹窗状态正常复位。

### Composer

`/task` 命令不再前端直接调用任务 API，而是发送到后端的对话式任务创建入口。

后续可以把 `/task` 视为快捷命令，不作为主要用户引导；主要引导应是自然聊天和手动创建按钮。

## 任务形成规则

第一版使用明确规则，不做复杂意图识别：

- 用户普通发送消息：只聊天，不创建任务。
- 用户手动点击新建任务：立即创建任务，并写聊天事件。
- 用户输入 `/task xxx`：立即创建任务，并写聊天事件。
- Agent 输出计划提案：只写计划消息，不创建任务。
- 用户确认计划：创建任务，并写聊天事件。
- 房间策略为自动推荐时：Agent 计划满足结构化要求后可以自动创建任务，但必须写明“系统根据自动策略创建任务”。

## 错误处理

- 任务创建失败时，如果用户意图消息已经写入，应补一条系统失败消息，说明失败原因。
- 系统确认消息写入失败时，任务仍然保留，但前端应通过任务事件广播看到任务；后端记录警告，后续可补偿。
- workflow 事件写入失败不应阻塞 workflow 状态机，但应记录服务端日志。
- 重复提交时，第一版不做复杂去重；依赖按钮 pending 状态和后端基础校验。

## 验证方式

- 手动创建任务后，聊天区出现用户意图消息和系统确认消息，任务看板出现任务。
- `/task` 创建任务后，聊天区不再空缺上下文，任务看板同步出现任务。
- 普通聊天不会创建任务。
- Agent 提出计划后，确认前不会创建任务。
- 用户确认计划后，创建主任务和子任务，并写入聊天事件。
- workflow 运行到计划、分配、执行、评审、验收等关键阶段时，聊天区出现对应轻量记录。
- `npm run build` 作为实现后的最低验证门禁。

## 迁移策略

第一版新增字段应保持可空，避免破坏既有数据：

- 旧任务的 `source_message_id` 为空。
- 旧任务的 `created_from` 可回填为 `manual` 或保持为空后在读取层兜底。
- 旧消息没有 `event_type` 时按普通消息渲染。

## 后续演进

- 增加任务事件筛选，只看某个任务相关聊天。
- 支持从任意聊天消息“转为任务”。
- 支持计划提案的结构化 diff，用户可以编辑后再确认。
- 支持自动策略的置信度阈值，低置信度只建议，不自动创建。
- 支持任务完成后自动生成对话摘要和项目记忆。

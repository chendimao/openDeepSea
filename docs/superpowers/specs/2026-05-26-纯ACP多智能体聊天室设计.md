# 纯 ACP 多智能体聊天室设计

## 背景

当前房间消息链路混合了三种语义：

1. 普通 ACP 对话消息
2. 项目内置 task / workflow / collaboration 系统事件
3. planner 用于驱动任务化流程的结构化元数据

这会导致用户在聊天室里看到大量项目内置流程卡片，而不是 ACP 原始回复。现有系统虽然已经具备 ACP 流式输出、智能体运行状态、活动日志与多智能体调度基础，但主交互路径仍然被任务和工作流概念主导。

本设计将房间重构为“公开多智能体对话”主界面：去掉自动任务化与工作流化，保留 ACP 原始消息、完整 thinking、工具调用过程和规划师调度能力。

## 目标

1. 整个项目不再自动创建任务和工作流，只保留纯 ACP 对话主路径。
2. 房间直接显示 ACP 收到并返回的原始消息，不再渲染项目内置 task / workflow / collaboration 卡片。
3. 每个执行智能体、审查智能体、规划师的原始回复都逐条公开显示在房间里。
4. 完整保留 ACP 返回的 thinking 文本，但前端默认折叠，不默认展开。
5. 展示工具调用过程、工具结果和命令执行过程，尽可能接近 ACP 实际输出。
6. planner 能读取其他智能体公开消息，分析后决定下一步。
7. planner 调度模式可配置，默认行为为“给出下一步建议后暂停，等待用户确认”。

## 非目标

1. 本轮不要求物理删除数据库中的 task / workflow / collaboration 历史表。
2. 本轮不要求清理所有旧页面和旧接口，只要求主路径不再依赖它们。
3. 本轮不重写 ACP adapter 协议本身，只在现有流式解析基础上扩展结构化 trace。
4. 本轮不把 planner 的自然语言回复替换成系统卡片或隐藏式调度。

## 关键决策

### 1. 房间是唯一主容器

用户消息、planner 分析、执行智能体回复、审查智能体回复、trace 展示都在同一条房间时间线中发生。系统不再从用户消息自动创建 task，也不再启动 workflow run 或 collaboration run 作为主交互容器。

### 2. planner 是调度者，不是任务生成器

planner 不再产出 `task_readiness` 或 `collaboration_decision` 供系统升级为任务流。planner 的职责改为：

1. 读取用户消息与房间公开上下文
2. 给出自然语言分析与下一步建议
3. 在需要时产出结构化 `planner_decision`
4. 根据配置决定“暂停等待用户确认”或“自动继续下一轮派发”

### 3. 所有智能体消息公开可见

执行智能体、审查智能体、规划师的原始回复都直接显示在房间中，不作为 planner 的隐藏内部输入。planner 后续分析时读取这些公开消息。

### 4. thinking 全量保留但默认折叠

ACP 返回的完整 thinking 文本需要原样保存并可展开查看，但默认收起，仅显示折叠头或预览。

### 5. 默认控制模式为“建议后暂停”

planner 的控制模式可配置：

- `pause_after_suggestion`
- `auto_continue`

默认值为 `pause_after_suggestion`。也就是说 planner 默认只给下一步建议，并等待用户确认后再继续派发。

## 用户体验

### 房间时间线

房间中的每条 agent 消息统一采用普通消息卡片，而不是系统流程卡片。消息结构由三层组成：

1. 主回复正文
2. Thinking 折叠区
3. Tool / Command Trace 折叠区

planner 消息额外附带一个轻量决策区，用于展示：

- 当前模式
- 下一步建议
- 是否等待用户确认

### 前端交互按钮

planner 消息下保留最小操作集合：

1. `按建议继续`
2. `让我补充说明`
3. `展开 Thinking`
4. `展开工具轨迹`

移除或废弃以下主路径入口：

1. 开始协作
2. 生成任务
3. 提升为工作流
4. workflow retry
5. task readiness 相关操作

### system message 收敛

主路径尽量不再使用系统消息表达产品内置流程，只保留基础错误与运行状态：

1. ACP 连接失败
2. planner 自动继续达到上限
3. 某智能体不可用
4. trace 解析失败等基础异常

## 后端设计

### 主路径改造

房间主路径调整为：

```text
用户消息
  -> dispatchUserMessage
  -> planner 分析
  -> planner_decision
  -> 用户确认后继续 或 自动继续
  -> 一个或多个 agent_run
  -> 公开消息写入房间
  -> planner 读取公开结果后进入下一轮
```

不再经过：

```text
用户消息
  -> task_readiness
  -> collaboration_decision
  -> task/workflow/collaboration
```

### planner 输出协议

planner 回复由两部分组成：

1. 面向用户的自然语言分析与建议
2. 面向系统的结构化 `planner_decision`

`planner_decision` 替代 `task_readiness / collaboration_decision`，建议结构如下：

```json
{
  "planner_decision": {
    "mode": "pause_after_suggestion",
    "status": "suggested",
    "summary": "建议先验证模型配置与连接测试链路",
    "next_steps": [
      {
        "agent_id": "frontend-executor",
        "goal": "检查设置页是否已有测试模型入口"
      },
      {
        "agent_id": "backend-executor",
        "goal": "检查模型测试接口是否存在并确认返回结构"
      }
    ],
    "awaiting_user_confirmation": true
  }
}
```

### AgentRun 职责收敛

`agent_runs` 继续保留，但不再承载 task / workflow / collaboration 主语义。其职责收敛为：

1. 记录某个房间内某个智能体的一次 ACP 执行
2. 保存 stdout / stderr / activity / trace patch
3. 向前端流式广播运行状态

主路径中不再依赖 `task_id`、`workflow_run_id`、`collaboration_run_id` 做行为判断。

### trace 结构化存储

当前系统已能把 ACP stdout 区分为 `answer` 与 `activity`。为满足完整展示，需要把 trace 从文本拼接升级为结构化存储。

建议在消息 metadata 中新增：

```json
{
  "trace": {
    "thinking": [
      { "text": "完整 thinking 原文" }
    ],
    "tool_calls": [
      {
        "name": "search_files",
        "input": "{\"pattern\":\"model settings\"}",
        "output": "..."
      }
    ],
    "commands": [
      {
        "command": "rg -n \"model\" packages/frontend/src",
        "output": "..."
      }
    ]
  }
}
```

如果 adapter 能拿到更细粒度的事件类型，也可以继续细分为 `tool_result`、`reasoning_summary`、`command_start`、`command_end` 等记录。

### WebSocket 流式协议

现有 `message:stream` 主要服务 answer 文本。建议沿用现有通道模型，扩展 channel：

- `answer`
- `thinking`
- `tool`
- `command`

前端据此把不同流式片段写入消息内容区或 trace 折叠区。

### 新控制接口

在保留现有发送消息接口的基础上，新增房间控制接口：

1. `POST /rooms/:roomId/planner/continue`
   - 根据 planner 最新一条 `planner_decision` 继续下一轮
2. `POST /rooms/:roomId/planner/dispatch`
   - 显式执行某条 planner 决策中的 `next_steps`

这两个接口取代原来的 `promote to workflow` 与 `start collaboration`。

### 配置项

建议新增以下配置：

1. `planner_execution_mode`
   - `pause_after_suggestion`
   - `auto_continue`
2. `planner_max_auto_turns`
   - 防止 planner loop 无边界运行
3. `show_full_thinking_by_default`
   - 默认 `false`

默认配置：

```text
planner_execution_mode = pause_after_suggestion
show_full_thinking_by_default = false
```

## 前端设计

### Room 页面

`RoomPage` 不再根据以下 metadata 渲染特殊流程卡片：

1. `event_type`
2. `task_readiness`
3. `collaboration_decision`
4. `workflow_*`

主时间线统一渲染为普通用户消息或 agent 消息。

### Agent 消息卡片

agent 消息卡片结构：

1. Header
   - agent 名称
   - 时间
   - ACP backend 标记
2. Body
   - 正文内容
3. Planner 决策块
   - 仅 planner 消息显示
4. Thinking 折叠块
   - 默认折叠
   - 展开后显示完整原文
5. Tool / Command Trace 折叠块
   - 默认折叠
   - 展开后显示调用、参数、结果和命令输出

### Planner 交互

当 planner 消息存在 `planner_decision` 时：

1. 若 `awaiting_user_confirmation = true`
   - 显示“按建议继续”
   - 显示“让我补充说明”
2. 若 `mode = auto_continue`
   - 显示当前自动执行状态
   - 显示自动轮数与停止原因

## 交互规则

### 单轮规则

单轮建议流程：

1. 用户发消息
2. planner 分析
3. planner 给出建议并暂停，或自动继续
4. 被派发的执行智能体/审查智能体公开回复
5. planner 读取这些公开回复
6. planner 决定是否继续下一轮

### 必须暂停的场景

planner 在以下情况必须暂停：

1. 需要用户做产品决策
2. 需要用户确认是否继续改代码
3. 信息不足，无法安全继续
4. 达到 `planner_max_auto_turns`
5. 检测到重复循环或无有效进展

### 自动继续前提

只有满足以下条件时 planner 才能自动继续：

1. 房间或系统配置为 `auto_continue`
2. 当前 decision 不要求用户确认
3. 未达到最大自动轮数
4. 未检测到循环、失败或阻塞

## 迁移策略

### 第一阶段

1. 停掉自动 `task_readiness / collaboration_decision / workflow` 生成
2. Room 前端移除相关卡片和按钮
3. 主路径只显示普通消息

### 第二阶段

1. 为 agent message 增加结构化 `trace`
2. 前端支持 thinking 与工具轨迹折叠区
3. planner 改产出 `planner_decision`

### 第三阶段

1. 增加 `planner/continue` 与 `planner/dispatch` 接口
2. 新增默认暂停与自动继续配置
3. 加入 `planner_max_auto_turns` 与循环保护

### 第四阶段

1. 清理遗留 routes / tests / 页面中的 task/workflow 主路径依赖
2. 旧 task/workflow 页降级为历史只读或后续彻底移除

## 测试策略

### 后端

1. 用户消息不再产生 `task_readiness`
2. planner 消息不再产生 `collaboration_decision`
3. planner 可以生成 `planner_decision`
4. `planner/continue` 能正确触发下一轮派发
5. `auto_continue` 能在达到上限时自动停止
6. trace 能保存完整 thinking 与工具调用内容

### 前端

1. Room 页面不再渲染 task/workflow/collaboration 卡片
2. planner 消息能展示 `planner_decision`
3. thinking 默认折叠，展开后显示完整原文
4. 工具轨迹默认折叠，展开后显示完整记录
5. “按建议继续”按钮能触发下一轮派发

### 验证

1. `npm run build`
2. 受影响后端测试
3. 受影响前端组件测试
4. 房间内多智能体对话 smoke test

## 风险与处理

### 风险一：planner loop 失控

处理：

1. 必须引入 `planner_max_auto_turns`
2. 检测重复 dispatch / 重复建议 / 无进展状态
3. 默认使用 `pause_after_suggestion`

### 风险二：thinking 与工具输出过大

处理：

1. trace 分块存储
2. 前端默认折叠
3. 超长内容按块展示，但不裁切完整原文

### 风险三：旧测试与旧 UI 大量依赖 workflow 语义

处理：

1. 先切主路径
2. 再清理旧接口和旧页面
3. 重写围绕 `task_readiness / collaboration_decision / workflow_*` 的测试

## 验收标准

1. 房间里不再出现项目内置 task / workflow / collaboration 卡片。
2. 用户消息不会自动创建任务或工作流。
3. planner、执行智能体、审查智能体的原始回复都会逐条公开显示。
4. ACP 返回的完整 thinking 可以展开查看，默认折叠。
5. 工具调用过程和命令执行过程可以在消息中查看。
6. planner 可以基于公开消息做后续分析并给出下一步建议。
7. 默认模式为“建议后暂停”，用户可以点击继续下一轮。

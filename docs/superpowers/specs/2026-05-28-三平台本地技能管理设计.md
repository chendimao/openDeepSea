# 三平台本地技能管理设计

## 背景

当前 Skills 页面管理的是 OpenDeepSea 内部技能：技能被导入到 `~/.opendeepsea/skills`，再通过 runtime scope、trigger、binding 参与 planner、model chat、workflow、memory、review 等内部模型路径的受控提示词注入。

这套模型不等同于用户本机 Claude Code、OpenCode、Codex 的原生 skills。新需求是把技能菜单重构为本地平台技能管理器：扫描三类 CLI 的本地 skills，支持从线上安装到指定平台，并让用户清楚知道某个 skill 被安装在哪里、以什么方式安装。

## 目标

- 管理本机 Codex、Claude Code、OpenCode 三个平台的原生 skills。
- 支持在线搜索公开 `skills.sh` 技能，并安装到一个或多个目标平台。
- 默认采用复制安装，允许高级用户选择 symlink 安装整个 skill 目录。
- 展示每个平台的本地目录状态、已安装技能、安装方式、来源、版本与冲突状态。
- 保留现有 OpenDeepSea 内部 skills 能力，但不把 runtime scope、trigger、binding 等内部概念混入三平台原生管理主界面。

## 非目标

- 第一版不实现完整的中央仓库、跨平台部署记录和回滚系统。
- 第一版不自动迁移现有 `~/.opendeepsea/skills` 到三平台目录。
- 第一版不在三平台原生 skill 与 OpenDeepSea runtime skill 之间做语义同步。
- 第一版不执行平台 CLI 运行验证，只做文件级安装、扫描和格式校验。

## 平台路径

默认平台写入路径如下：

- Codex：`$CODEX_HOME/skills`，未设置时使用 `~/.codex/skills`。
- Claude Code：`~/.claude/skills`。
- OpenCode：优先使用 OpenCode 原生目录 `~/.config/opencode/skills`。

OpenCode 可能兼容读取 `.claude/skills`、`.agents/skills` 等目录，但本管理器默认只写入 OpenCode 原生目录，避免一个平台安装动作隐式影响其他平台。

## 安装策略

安装方式分为两种：

- `copy`：默认方式。把线上或本地 skill 目录复制到目标平台目录下，适合普通用户，隔离性最好。
- `symlink`：高级方式。把目标平台目录下的 skill 目录软链接到源目录，便于统一维护，但需要清楚路径依赖和平台兼容风险。

symlink 必须链接整个 skill 目录，不只链接 `SKILL.md`。界面中应将 symlink 标为高级选项，并在安装确认里提示“移动源目录会导致目标平台 skill 失效”。

## 页面设计

Skills 页面改为三平台本地技能管理器。

主要结构：

- 页面头部：标题、说明、刷新按钮、平台目录健康摘要。
- 平台切换：`Codex`、`Claude Code`、`OpenCode` 三个 tab。
- 在线安装区：搜索 `skills.sh`，展示结果，点击安装后选择目标平台，可多选。
- 当前平台库存：列出当前 tab 平台已安装 skills。
- 详情面板：展示选中 skill 的名称、描述、路径、来源、版本、安装方式、是否 symlink、最后修改时间、校验问题。
- 高级安装选项：折叠展示 `copy` / `symlink` 选择，默认 `copy`。

当前的 runtime scope、trigger mode、系统绑定、prompt preview、运行历史不出现在三平台管理主界面。它们可以后续迁移到 “OpenDeepSea 内部技能” 子入口。

## 后端设计

新增独立模块 `platform-skills`，避免污染现有 `skills` 模块的内部 runtime 语义。

建议 API：

- `GET /api/platform-skills/platforms`
  - 返回三平台目录、目录是否存在、可写状态、已安装数量。
- `GET /api/platform-skills/:platform`
  - 扫描指定平台目录，返回 skills 列表。
- `GET /api/platform-skills/:platform/:skillName`
  - 返回某个 skill 详情，包括 `SKILL.md` frontmatter、文件状态和 symlink 信息。
- `GET /api/platform-skills/marketplace?q=...`
  - 复用现有 `SkillsShClient` 搜索公开 `skills.sh`。
- `POST /api/platform-skills/install`
  - 输入 `installLabel`、`targets`、`installMode`，从 `skills.sh` 安装到一个或多个平台。
- `POST /api/platform-skills/import-local`
  - 输入本地目录、目标平台、安装方式，把本地 skill 安装到平台目录。
- `DELETE /api/platform-skills/:platform/:skillName`
  - 删除平台目录内的 skill；删除前必须确认目标路径位于该平台 skills 根目录内。

## 数据模型

第一版可以不新增持久化表，优先以文件系统为事实来源。扫描结果返回结构建议：

```ts
type PlatformSkillProvider = 'codex' | 'claudecode' | 'opencode';
type PlatformSkillInstallMode = 'copy' | 'symlink' | 'unknown';

interface PlatformSkill {
  provider: PlatformSkillProvider;
  name: string;
  description: string | null;
  path: string;
  manifestPath: string | null;
  installMode: PlatformSkillInstallMode;
  sourceLabel: string | null;
  version: string | null;
  lastModifiedAt: number | null;
  valid: boolean;
  issues: string[];
}
```

后续如果需要统一更新、安装来源追踪和回滚，可以再新增 deployment 表，把平台安装视为中央技能库的部署结果。

## 冲突与校验

- 安装前检查目标平台是否已有同名目录。
- 同名冲突默认阻止覆盖，提供明确的“覆盖安装”二次确认能力作为后续增强。
- 本地导入必须包含 `SKILL.md`。
- 复制安装跳过 `.git`、`node_modules`、过大文件和 symlink，沿用现有安全拷贝策略。
- 删除时只允许删除目标平台 skills 根目录内的路径，禁止跟随 symlink 删除源目录。
- symlink 删除只删除链接本身。

## 复用与迁移

可复用现有能力：

- `SkillsShClient`：搜索和下载公开 `skills.sh`。
- `installer-runner`：把 skills.sh package materialize 成本地目录。
- 现有本地安全复制策略。
- 前端 `SkillMarketplacePanel` 的搜索交互思路。

需要新建或重写：

- 平台路径解析与扫描服务。
- 平台安装服务。
- 三平台管理页组件。
- 平台 skill 类型与 API client 方法。

## 验证方式

- 后端单元测试覆盖平台路径解析、目录扫描、复制安装、symlink 安装、同名冲突、删除边界。
- 路由测试覆盖 local token、marketplace 搜索、安装到多平台、删除平台 skill。
- 前端构建验证 TypeScript 类型。
- 手动 smoke：在临时 `HOME` 或临时平台根目录中安装一个 mock skill，确认三个 tab 均能展示正确状态。

## 风险

- 三个平台对 `SKILL.md` frontmatter 支持不完全一致，第一版只做通用字段展示，不强行转换平台特有语义。
- Codex 对 symlink 发现行为可能受版本影响，因此复制是默认方式，symlink 只作为高级选项。
- OpenCode 的兼容目录较多，第一版只写入原生目录，避免隐式跨平台污染。
- 删除与覆盖属于高风险文件操作，必须保持路径边界校验和 UI 二次确认。

## 已确认决策

- 采用平台原生管理器方案，而不是继续扩展 OpenDeepSea 内部 `skills` 表。
- 默认复制安装。
- 允许高级用户选择 symlink，但只 symlink 整个 skill 目录。
- 第一版不把三平台管理和 OpenDeepSea runtime skill binding 混在同一个主界面。

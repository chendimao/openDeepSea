# 项目列表删除项目设计

## 背景

当前项目列表没有删除项目入口。后端已存在 `DELETE /projects/:id` 和 `projectRepo.delete()`，前端 API 也已有 `api.deleteProject(id)`，但 Dashboard 页面没有暴露删除交互。用户确认删除语义为：只从 OpenDeepSea 的项目列表和本地数据库中移除项目，不删除真实磁盘项目目录。

## 目标

1. 在项目列表卡片提供低误触的删除入口。
2. 删除前展示明确确认，说明不会删除本地项目文件夹。
3. 删除成功后刷新项目列表并给出反馈。
4. 后端删除前阻止运行中的 agent 或 workflow 被级联移除。
5. 清理项目级内部数据，避免孤儿设置。

## 非目标

1. 不删除 `project.path` 指向的真实代码目录。
2. 不实现项目归档、恢复、回收站或批量删除。
3. 不改变项目创建、项目详情、群聊和任务的核心流程。
4. 不引入新的数据库迁移体系。

## 推荐方案

采用项目卡片右上角更多菜单作为删除入口。

- 项目卡片整体仍然点击进入项目。
- 卡片右上角增加独立的更多按钮，阻止事件冒泡，避免触发卡片跳转。
- 菜单内提供危险色的“删除项目”动作。
- 点击后打开二次确认弹窗。
- 弹窗展示项目名称、项目路径和影响范围。
- 确认按钮调用 `api.deleteProject(id)`。

选择该方案的原因：

- 列表页可以直接清理项目，操作可发现。
- 删除动作不直接暴露为单击按钮，误触风险低。
- 移动端和桌面端都可使用，不依赖 hover。
- 符合现有卡片列表的管理习惯。

## 用户体验

删除确认弹窗应表达三类信息：

1. 将从 OpenDeepSea 中移除该项目。
2. 项目内的群聊、消息、任务、运行记录、文件记录和记忆等内部数据会被移除。
3. 不会删除本地项目文件夹或其中的真实代码文件。

建议文案：

- 标题：`删除项目`
- 描述：`将从 OpenDeepSea 中移除「{name}」。相关群聊、消息、任务、文件记录和记忆会一并移除，但不会删除本地项目文件夹。`
- 路径提示：展示 `project.path`
- 取消按钮：`取消`
- 确认按钮：`删除项目`

删除成功：

- 关闭弹窗。
- 刷新 `['projects']` 查询。
- 显示 toast：`项目已删除`。

删除失败：

- 若后端返回 `409`，显示提示：项目仍有运行中的智能体或工作流，请先停止或等待完成。
- 其他错误沿用当前 API 错误信息 toast。

## 后端行为

现有 `DELETE /projects/:id` 需要增强为受保护删除：

1. 项目不存在时返回 `404`。
2. 删除前查询项目下是否存在运行中的 agent run。
3. 删除前查询项目下是否存在 active workflow。
4. 若存在活动项，返回 `409`，附带可读错误和数量摘要。
5. 无活动项时执行事务：
   - 删除 `settings` 中 `scope='project' AND scope_id=projectId` 的记录。
   - 删除 `settings` 中属于该项目房间的 `scope='room'` 记录。
   - 删除 `projects` 行，依赖外键级联移除关联表。

不删除真实项目目录，也不删除 `project.path` 下任何文件。

项目上传文件的磁盘缓存属于 OpenDeepSea 内部数据。本次实现可以优先只删除数据库记录；若实现文件系统清理，只允许清理 `packages/backend/data/uploads/files/<projectId>` 这类 OpenDeepSea 管理目录，不能触碰 `project.path`。

## 数据与级联

`projects` 已被多张表通过外键引用，并启用了 `ON DELETE CASCADE`。删除项目行将清理：

- `rooms`
- `files`
- `message_file_refs`
- `workflow_runs`
- `memory_entries`
- 以及通过房间、任务、运行记录等外键间接关联的数据

需要额外处理：

- `settings` 表没有外键，需要显式清理 project scope 和 room scope。
- 运行中的 agent 或 workflow 不应被静默级联删除。

## 前端改动范围

主要改动：

- `packages/frontend/src/pages/DashboardPage.tsx`
- 可能新增 `DeleteProjectDialog` 或在页面内提取局部组件。
- `packages/frontend/src/lib/i18n.tsx` 添加中英文文案。

现有 API：

- `api.deleteProject(id)` 已存在，可直接复用。

交互细节：

- 更多按钮使用 lucide 图标，例如 `Ellipsis` 或 `MoreHorizontal`。
- 删除动作使用现有 `Button` 的 `danger` 视觉或菜单危险样式。
- 按钮点击必须 `preventDefault()` 和 `stopPropagation()`，避免触发项目卡片导航。
- mutation pending 时禁用确认按钮，显示删除中状态。

## 测试与验证

最低验证：

1. `npm run build`
2. 手动验证：
   - 创建或选择一个测试项目。
   - 从项目列表打开更多菜单。
   - 取消删除后项目仍存在。
   - 确认删除后项目从列表消失。
   - 刷新页面后项目仍不存在。
   - 确认真实项目目录仍存在。

建议自动化测试：

- 后端路由测试：删除不存在项目返回 `404`。
- 后端路由测试：有 active run 或 active workflow 时返回 `409`。
- 后端路由测试：无活动项时删除项目并清理 settings。

## 风险

1. 当前删除路由直接删除项目，若不增加活动项保护，可能导致运行中的后台任务继续向已删除 room 写入。
2. `settings` 不受外键约束，若不显式清理会留下孤儿配置。
3. 项目卡片是 `Link`，新增菜单按钮必须阻止冒泡，否则会误导航。
4. 删除确认文案必须清楚地区分 OpenDeepSea 内部记录和真实磁盘目录。

## 验收标准

1. 用户能在项目列表卡片右上角找到删除入口。
2. 删除前必须二次确认。
3. 确认文案明确说明不会删除本地项目文件夹。
4. 删除后项目从列表消失，刷新后仍不存在。
5. 项目真实目录未被删除。
6. 有运行中 agent 或 workflow 的项目删除失败并提示原因。
7. 项目级和房间级 scoped settings 不留下孤儿记录。
8. 构建通过。

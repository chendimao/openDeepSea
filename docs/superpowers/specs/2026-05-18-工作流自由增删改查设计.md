# 工作流自由增删改查设计

## 背景

当前系统已经支持工作流定义表、内置默认工作流、创建草稿、发布、房间可见列表、工作流快照和基础可视化编排器。最近提交还新增了 `/workflow` 总览入口，但该页面目前主要是只读列表；房间设置中的编排入口会创建并发布一个群聊级工作流，随后设为当前群聊默认。

现有问题是用户无法在一个稳定入口中自由管理工作流：只能围绕默认工作流做有限创建，不能完整编辑、复制、归档、删除草稿，也不能管理系统级和项目级工作流。用户明确希望工作流支持自由增删改查，不再只有一个不可编辑的默认工作流。

## 目标

1. 将 `/workflow` 升级为工作流管理中心。
2. 支持系统级、项目级、群聊级用户工作流。
3. 支持新建、编辑草稿、复制、发布、归档、删除草稿、设为默认。
4. 已发布工作流不可直接修改；编辑已发布工作流时派生新草稿。
5. 内置工作流保持只读保护，只允许查看和复制。
6. 历史运行继续通过 `workflow_definition_snapshot` 保持可追溯。

## 非目标

1. 不做版本历史详情页。
2. 不做回滚。
3. 不放开任意图结构 runtime。
4. 不支持多个同类型节点。
5. 不在第一版实现已归档恢复按钮，除非实现成本很低。

## 生命周期

工作流定义按状态和保护类型处理：

| 类型 | 行为 |
| --- | --- |
| 内置工作流 | `builtin_key` 非空，只读，不可编辑、删除、归档，可复制为用户工作流 |
| 草稿 | `status=draft`，可编辑、删除、发布、复制 |
| 已发布 | `status=published`，不可直接编辑；编辑时创建新草稿副本；可复制、归档、设为默认 |
| 已归档 | `status=archived`，不进入默认选择推荐列表；仍可查看和复制 |

发布版本按不可变语义处理。已发布工作流的编辑入口不修改原记录，而是创建一个基于当前定义的新草稿。原发布版本继续服务已有默认设置引用和历史运行快照。

## 作用域

支持三类作用域：

| 作用域 | 可见范围 | 默认设置 |
| --- | --- | --- |
| `system` | 所有项目和群聊 | 可设为系统默认 |
| `project` | 对应项目及其群聊 | 可设为项目默认 |
| `room` | 对应群聊 | 可设为群聊默认 |

用户可以创建 `system` 级工作流，表示全局可见工作流。内置工作流仍通过 `builtin_key` 区分，并保持不可编辑保护。用户创建的 `system` 工作流不写 `builtin_key`。

`system` 级用户工作流的 `scope_id` 统一使用 `default`。

## 后端设计

现有 `workflow_definitions` 表已经包含所需字段：`scope`、`scope_id`、`status`、`builtin_key`、`version`、`definition_json`、`created_at`、`updated_at`。第一版不新增表。

需要补齐 `workflowDefinitionRepo` 能力：

1. `createDraft(input)`：允许创建 `system`、`project`、`room` 草稿。`system` 只接受或归一化为 `scope_id=default`。
2. `updateDraft(id, patch)`：只允许更新非内置草稿。
3. `publish(id)`：发布非内置草稿。已发布后作为不可变版本。
4. `duplicate(id, target)`：复制任意可见工作流为新草稿，可指定目标作用域。
5. `createEditDraft(id)`：已发布工作流编辑时派生新草稿；内置工作流也通过复制语义生成草稿。
6. `archive(id)`：归档非内置已发布工作流。
7. `deleteDraft(id)`：只删除非内置草稿。
8. `list(filters)`：支持按作用域、状态、项目、群聊、是否包含归档过滤。

建议新增或扩展 API：

| 方法 | 路径 | 用途 |
| --- | --- | --- |
| `GET` | `/workflow-definitions` | 管理中心列表，支持过滤参数 |
| `POST` | `/workflow-definitions` | 创建草稿 |
| `PATCH` | `/workflow-definitions/:id` | 编辑草稿 |
| `POST` | `/workflow-definitions/:id/publish` | 发布草稿 |
| `POST` | `/workflow-definitions/:id/duplicate` | 复制为新草稿 |
| `POST` | `/workflow-definitions/:id/edit-draft` | 从已发布或内置工作流派生编辑草稿 |
| `POST` | `/workflow-definitions/:id/archive` | 归档已发布工作流 |
| `DELETE` | `/workflow-definitions/:id` | 删除草稿 |

保留 `GET /rooms/:roomId/workflow-definitions` 作为群聊默认选择列表。该列表默认只返回当前群聊可见的已发布、未归档工作流。

## 默认设置规则

默认设置保存时必须校验工作流可见性和发布状态：

1. 系统默认只能选择 `published system` 工作流。
2. 项目默认可选择 `published system` 或当前项目的 `published project` 工作流。
3. 群聊默认可选择 `published system`、当前项目的 `published project`、当前群聊的 `published room` 工作流。
4. 已归档工作流不出现在新选择列表中。
5. 如果已有设置引用了已归档工作流，详情中仍显示该工作流及“已归档”状态，提醒用户重新选择。

草稿通常不应被默认设置引用。如果删除草稿时发现仍被设置引用，后端应拒绝删除并返回明确错误。

## 前端设计

`/workflow` 升级为管理中心：

1. 顶部提供搜索、状态筛选、范围筛选和 `新建工作流`。
2. 提供 `全部`、`系统`、`项目`、`群聊`、`草稿`、`已归档` 视图。
3. 卡片或列表展示名称、描述、作用域、状态、版本、节点数、连线数、更新时间、是否内置。
4. 卡片操作按状态显示：
   - 内置：查看、复制。
   - 草稿：编辑、发布、复制、删除。
   - 已发布：查看、编辑新草稿、复制、归档、设为默认。
   - 已归档：查看、复制。

`WorkflowBuilderDialog` 从“房间发布并应用”改为通用 builder：

1. 支持选择作用域：系统、项目、群聊。
2. 支持保存草稿。
3. 支持发布。
4. 发布后可选择是否设为对应作用域默认工作流。
5. 编辑已发布工作流时先走派生草稿流程，再打开草稿编辑。
6. 继续使用受控节点类型、受控连线和现有 runtime 校验。

房间设置保留快捷入口：

1. 继续提供默认工作流下拉选择。
2. `编排` 按钮打开 builder，默认作用域为当前群聊。
3. 不在房间设置中承载完整 CRUD，避免与管理中心分叉。

系统和项目设置增加默认工作流选择：

1. 系统设置列出可用的系统级已发布工作流。
2. 项目设置列出系统级和当前项目级已发布工作流。
3. 群聊设置列出系统级、项目级和群聊级已发布工作流。

## 错误处理

后端硬拦：

1. 内置工作流不可编辑、删除、归档。
2. 非草稿不可直接 `PATCH`。
3. 草稿可删除；已发布只能归档。
4. `builtin_key` 只能由内置初始化逻辑写入。
5. 发布前继续调用 `validateDefinition()`。
6. 默认设置保存时拒绝不可见、非发布、已归档工作流。

前端提示：

1. 编辑已发布工作流时提示“将创建新草稿，原发布版本不变”。
2. 删除草稿前二次确认。
3. 归档已发布工作流前二次确认，并说明不影响历史运行。
4. 发布失败时展示后端校验错误。
5. 内置工作流不显示不可执行操作。

## 验证策略

后端 repo 测试：

1. 创建 system、project、room 草稿。
2. 复制内置、已发布和草稿工作流。
3. 已发布工作流派生编辑草稿，原记录保持不变。
4. 发布、归档、删除草稿。
5. 内置保护。
6. 默认设置可见性和发布状态校验。

后端 route 测试：

1. 新增 API 成功路径。
2. 非法作用域、非法状态、内置保护、不可见默认设置等失败路径。
3. 列表过滤和群聊可见列表。

前端验证：

1. `npm run build`。
2. 手动 smoke：新建草稿、发布、设默认、编辑已发布生成草稿、归档。
3. 若 UI 改动范围较大，补充组件级测试。

## 验收标准

1. `/workflow` 可以管理系统级、项目级、群聊级工作流。
2. 用户可以创建、编辑草稿、复制、发布、归档、删除草稿。
3. 已发布工作流编辑时生成新草稿，不修改原发布记录。
4. 内置工作流不可编辑、删除、归档，只能复制。
5. 默认设置只能保存当前作用域可见且已发布的工作流。
6. 已归档工作流不再出现在默认选择推荐列表中。
7. 历史 workflow run 的 definition snapshot 不受后续编辑影响。
8. 后端新增路径有测试覆盖，前端构建通过。

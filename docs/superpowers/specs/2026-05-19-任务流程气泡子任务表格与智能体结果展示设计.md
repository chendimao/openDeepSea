# 任务流程气泡子任务表格与智能体结果展示设计

## 背景

当前群聊里的任务启动流程，已经能把用户消息提升为任务，也能把工作流跑起来。但任务的“计划文档”、子任务拆分、智能体执行结果和 UI 展示仍然耦在一起，容易出现两类问题：

1. 任务的自然语言计划和真正执行所需的结构化数据不一致。
2. 用户只能看到聊天气泡里的自然语言输出，看不到明确的子任务状态、并行/串行关系和各智能体执行结果。

本设计要解决的是：在用户点击“开始 workflow”后，由产品经理先把已有 Markdown 计划归一为固定 JSON，再由工作流协调器按 JSON 调度智能体；同时在气泡里提供结构化进度展示。

## 目标

1. 用户点击开始 workflow 时，系统将 Markdown 计划转换为固定 JSON。
2. JSON 作为唯一执行源，驱动子任务拆分、依赖关系、并行/串行调度和智能体分配。
3. 气泡上方展示子任务表格与动态进度。
4. 气泡下方按智能体分 Tabs 展示执行结果。
5. 每个智能体按照 workflow 流程逐步执行，不再靠自然语言临时猜测。

## 非目标

1. 不在 UI 层重新实现调度逻辑。
2. 不让 Markdown 计划文档直接参与执行判断。
3. 不做复杂 DAG 编辑器。
4. 不做多版本计划管理。
5. 不在第一版支持跨房间分布式编排。

## 核心原则

1. Markdown 只做人可读稿。
2. JSON 只做机器可执行稿。
3. 调度逻辑只认 JSON。
4. UI 只展示状态，不决定状态。
5. 并行/串行必须由依赖关系推导，不靠文本标签猜。

## 交互流程

1. 用户在群聊里确认任务。
2. 点击“开始 workflow”。
3. 产品经理读取已有 Markdown 计划并归一为固定 JSON。
4. 协调器根据 JSON 生成子任务，计算依赖和执行顺序。
5. 系统把任务表格、进度和 agent 结果实时推送到气泡中。
6. 每个智能体完成后，状态回写并驱动下一步。

## 固定 JSON 结构

建议定义一个稳定的 `workflow_plan` 对象，作为执行层输入。

```ts
interface WorkflowPlanJson {
  workflow_name: string;
  source_message_id: string;
  goal: string;
  summary: string;
  tasks: WorkflowPlanTaskJson[];
}

interface WorkflowPlanTaskJson {
  id: string;
  title: string;
  description: string;
  role: 'planner' | 'executor' | 'reviewer' | 'acceptor';
  agent_id: string | null;
  mode: 'parallel' | 'serial';
  depends_on: string[];
  status: 'pending' | 'running' | 'completed' | 'blocked' | 'failed';
  progress: number;
  result_refs: string[];
}
```

### 结构约束

1. `id` 必须稳定，可回溯。
2. `agent_id` 允许为空，表示尚未最终分配。
3. `mode` 是展示与调度的派生结果，不单独手写。
4. `depends_on` 是串行控制的唯一依据。
5. `progress` 由任务状态和结果回写驱动。
6. `result_refs` 存放消息 ID、artifact ID、step ID 等引用。

## 调度规则

### 任务拆分

产品经理只负责把 Markdown 计划拆成 JSON 任务，不直接执行子任务。

### 并行 / 串行

1. `depends_on` 为空且同一阶段可执行的任务，标记为并行。
2. 有前置依赖的任务标记为串行。
3. 若某任务依赖尚未完成，则保持 `blocked`。
4. 同一智能体若被多个任务命中，默认按任务依赖顺序串行执行。

### 智能体分配

1. 协调器根据 `role`、能力、工作区权限和已有房间智能体选择执行者。
2. 允许一个智能体负责多个任务，但不允许跳过依赖。
3. reviewer / acceptor 必须在对应执行结果完成后再启动。

## 气泡展示

### 上方：任务总览

显示：

- 已完成 / 总数
- 当前运行中任务数
- 阻塞任务数
- 总进度条
- 当前阶段标签

### 中间：子任务表格

建议列：

- 任务名
- 角色
- 执行智能体
- 并行 / 串行
- 依赖
- 状态
- 进度

表格要求：

1. 自动刷新。
2. 当前运行行高亮。
3. 阻塞行显示原因。
4. 完成行可折叠。

### 下方：按智能体分 Tabs

Tabs 以智能体为一级分组，不以任务为一级分组。

每个 Tab 建议包含：

- 摘要
- 原文
- 产物
- 审查意见
- 相关任务列表

如果智能体数量过多，增加：

- 搜索
- 折叠 inactive 智能体
- “全部结果”汇总页

## 动态更新链路

1. workflow step 创建时，表格新增/更新对应行。
2. step 状态变化时，更新任务状态和进度。
3. agent run 完成时，将结果写入对应 Tab。
4. 任务完成后，自动推进依赖中的后续任务。
5. 进度条始终由子任务完成度汇总计算，不手工维护。

## 错误处理

1. Markdown 转 JSON 失败时，先展示错误，不直接启动执行。
2. 某个智能体不可用时，任务标记为 `blocked`。
3. 某个子任务失败时，保持其他独立分支可继续。
4. 依赖链断裂时，必须显式显示阻塞原因。
5. JSON schema 不合法时，禁止调度。

## 与现有系统的关系

1. 现有 workflow runtime 继续作为执行引擎。
2. 现有 `planner` 角色负责生成或整理计划 JSON。
3. 现有 `dispatch` / `execute` / `review` / `acceptance` 节点负责落地执行。
4. 前端现有工作流时间线、消息气泡、任务面板可复用部分组件，但需要新增任务表格和智能体 Tabs 视图。

## 验证方式

### 后端

1. Markdown 计划能稳定转为固定 JSON。
2. JSON 中的依赖关系能正确算出串行与并行。
3. 任务分配能落到正确智能体。
4. 进度能随 step 状态变化刷新。

### 前端

1. 气泡顶部表格能显示任务状态和进度。
2. Bottom Tabs 能按智能体切换结果。
3. 任务完成后 UI 能实时更新。
4. 阻塞、失败、完成态展示清晰。

### 回归

1. 用户确认任务后，任务标题和描述不再被产品经理过程性文本污染。
2. “开始 workflow” 不会再次把自然语言计划误当成执行依据。
3. 已有 workflow 事件链不被破坏。

## 验收标准

1. 用户点击开始 workflow 后，系统先生成固定 JSON。
2. JSON 能描述子任务、依赖、并行 / 串行和智能体分配。
3. 气泡顶部可见任务表格和动态进度。
4. 气泡底部能按智能体查看执行结果。
5. 每个子任务的执行顺序由 JSON 和 workflow state 决定。
6. UI 不再依赖自然语言判断执行流程。

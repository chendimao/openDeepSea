# 群聊协作与正式工作流分流设计

## 背景

当前聊天室存在两套执行机制：

- 群聊协作：普通消息经 `fallback_route` 交给 planner，planner 回复“建议协作：@A @B”，系统解析 @ 名单后并发启动多个智能体。
- 正式工作流：基于任务启动 LangGraph workflow，按 `context -> planning -> approval -> dispatch -> execute -> review -> verify -> acceptance -> memory` 流转。

两套机制本身可以共存，但当前边界不清。用户提出“修复这个问题”这类执行型请求时，系统仍停留在普通群聊协作；reviewer、tester、acceptor 会和 executor 同时开始，无法基于实现结果审查或验收。

## 目标

1. planner 在分析用户请求后，明确让用户选择“轻量群聊协作”或“正式 workflow”。
2. 轻量群聊协作不再简单并发所有智能体，而是具备最小阶段流转。
3. 正式 workflow 继续复用现有 LangGraph 任务工作流。
4. 用户能从界面和消息中清楚知道当前处于哪种模式。

## 分阶段交付

### 第一阶段：最小可用闭环

第一阶段优先修复用户可感知的核心问题：执行型群聊请求不能再让所有智能体同时开始，也不能让用户误以为普通群聊就是正式 workflow。

范围：

- planner 先产出协作决策卡片。
- 用户可以选择“轻量群聊协作”或“正式 workflow”。
- 选择“正式 workflow”后，从原消息创建 task 并启动现有 LangGraph workflow。
- 选择“轻量群聊协作”后，按 `execute -> review/test -> acceptance` 顺序运行。
- fullstack 场景允许 frontend/backend executor 并行，但 review/test 必须等待 executor 完成。
- 通过系统消息或轻量 UI 明确展示当前模式和阶段。

第一阶段可以暂不实现完整的 collaboration 持久化表。如果为了降低改动量，可以先用消息 metadata 和 `agent_runs` 关联字段记录轻量协作状态。

### 第二阶段：增强与可追踪化

第二阶段把轻量群聊协作升级为可追踪、可重试、可审计的独立执行实体。

范围：

- 新增 `collaboration_runs` 和 `collaboration_steps` 表。
- 增加轻量协作阶段时间线 UI。
- 支持阶段级重试、取消、失败恢复。
- 支持 planner summary 汇总。
- 支持更精细的协作策略，例如 triage-only、executor-only、review-only。
- 为轻量协作写入必要记忆，但不混同正式 workflow 的任务完成记忆。

第二阶段完成后，系统将形成三种清晰层级：普通回复、轻量协作、正式 workflow。

## 非目标

- 不替换现有 LangGraph workflow。
- 不把所有普通聊天都强制升级为任务。
- 不在第一阶段实现复杂 DAG、跨房间协作或长期多分支 worktree 编排。
- 不允许 planner 直接绕过用户选择启动正式 workflow，除非房间交互策略明确设置为自动模式。

## 模式边界

### 群聊协作

适用于轻量、即时、探索性或小范围处理：

- “分析一下”
- “帮我看看可能原因”
- “让前端和后端各给个判断”
- 用户明确选择“轻量群聊协作”

轻量群聊协作仍可运行代码相关智能体，但它不等同正式 workflow，不负责完整任务记忆、审批、任务拆分和长期追踪。

### 正式 workflow

适用于明确交付型请求：

- “修复这个 bug”
- “实现这个功能”
- “提交改动”
- “需要验收/测试/回归”
- 用户选择“正式 workflow”

正式 workflow 必须创建 task，再启动现有 LangGraph workflow。

## Planner 决策

planner 的职责从“直接 @ 其他智能体”调整为“先产出协作决策”。

建议结构：

```ts
interface CollaborationDecision {
  intent: 'question' | 'analysis' | 'implementation';
  recommendedMode: 'chat_collaboration' | 'formal_workflow';
  problemArea: 'frontend' | 'backend' | 'fullstack' | 'unknown';
  summary: string;
  rationale: string;
  proposedAgents: {
    executors: string[];
    reviewers: string[];
    testers: string[];
    acceptors: string[];
  };
  stages: CollaborationStagePlan[];
  needsUserChoice: boolean;
}

interface CollaborationStagePlan {
  stage: 'execute' | 'review' | 'acceptance' | 'summary';
  agentIds: string[];
  parallel: boolean;
  goal: string;
}
```

决策原则：

- 问答型请求默认保持普通回复或轻量协作。
- 明确实现/修复/提交类请求推荐正式 workflow。
- 前后端边界不清时，先推荐轻量协作的 triage，或正式 workflow 的 planning。
- planner 不再通过“建议协作：@A @B”直接触发二次并发 dispatch。

## 用户交互

planner 分析后，聊天区展示决策卡片：

- 请求类型：问答 / 分析 / 实现
- 推荐模式：轻量群聊协作 / 正式 workflow
- 问题归属：前端 / 后端 / 全栈 / 未确定
- 推荐参与者
- 推荐原因
- 操作按钮：
  - `轻量群聊协作`
  - `正式 workflow`

如果房间交互策略将来支持自动模式，可以默认选择推荐模式，但仍应在聊天中记录系统事件。

## 轻量群聊协作流转

轻量群聊协作使用阶段化 runner，而不是一次性 `Promise.all` 所有智能体。

推荐流转：

```text
planner decision
-> execute
-> review/test
-> acceptance
-> summary
```

规则：

- `execute` 阶段可按问题归属决定是否并行。
- `review/test` 必须等待 execute 完成。
- `acceptance` 必须等待 review/test 完成。
- `summary` 由 planner 汇总，可选但推荐保留。

问题归属映射：

```text
frontend:
  frontend-executor -> reviewer/tester -> acceptor -> planner summary

backend:
  backend-executor -> reviewer/tester -> acceptor -> planner summary

fullstack:
  frontend-executor + backend-executor 并行
  -> reviewer/tester
  -> acceptor
  -> planner summary

unknown:
  planner triage 后让用户选择，或只运行分析型协作
```

## 数据模型

第一阶段可以选择轻量实现，第二阶段推荐新增轻量协作表，避免把它伪装成正式 workflow。

### 第一阶段最小记录

第一阶段可使用 message metadata 记录 planner decision，并在 `agent_runs` 中增加可选字段：

```text
collaboration_run_id
collaboration_step_id
collaboration_stage
```

如果第一阶段暂不新增表，也必须保证：

- 用户能看到当前选择的是轻量协作还是正式 workflow。
- 后端能保证 executor 与 reviewer/tester 的顺序。
- agent_run 至少能标记所处轻量协作阶段。

### 第二阶段持久化表

第二阶段新增：

```sql
collaboration_runs
- id
- room_id
- source_message_id
- decision_message_id
- status
- mode
- problem_area
- recommended_mode
- selected_mode
- summary
- error
- created_at
- updated_at
- completed_at

collaboration_steps
- id
- collaboration_run_id
- stage
- status
- room_agent_id
- agent_run_id
- prompt
- result_message_id
- error
- sort_order
- started_at
- completed_at
```

## 后端接口

新增接口建议：

```text
POST /rooms/:roomId/messages/:messageId/collaboration-decision
```

生成 planner 决策，不启动执行智能体。

```text
POST /rooms/:roomId/collaborations
```

根据用户选择启动轻量群聊协作。

```text
POST /rooms/:roomId/messages/:messageId/promote-to-workflow
```

从原消息创建 task，并启动正式 workflow。

正式 workflow 内部继续使用现有：

```text
startWorkflowWithConversation()
createTaskWithConversation()
```

## 前端展示

前端需要新增：

- Collaboration decision card。
- Collaboration stage timeline。
- 清晰标识：
  - `轻量协作`
  - `正式 workflow`
  - `普通回复`
- 对正式 workflow 启动后，刷新 task board、workflow detail、room workflows。

## 错误处理

- planner 决策解析失败：降级为普通 planner 文本回复，不自动启动其他 agent。
- 用户选择轻量协作但缺少 executor：创建系统消息提示缺少可用智能体。
- 某阶段 agent 失败：协作 run 标记 blocked，允许重试该阶段。
- 用户选择正式 workflow 但 task 创建失败：保留原消息，展示 JSON 错误和系统事件。

## 兼容性

- `/task ...` 继续直接创建任务。
- `/start-task ...` 和 `开始任务 #id` 继续直接启动正式 workflow。
- `mentions_only` 下用户显式 @ 智能体仍可直接触发普通 agent 回复。
- `fallback_reply` 可维持原行为。
- `fallback_route` 调整为默认进入 planner decision，而不是直接二次并发 dispatch。

## 验收标准

1. 执行型普通消息不会直接并发启动所有 agent。
2. planner 产出可见决策卡片，并提供两种模式选择。
3. 选择轻量群聊协作时，executor 先运行，review/test 后运行，acceptance 最后运行。
4. fullstack 场景中前后端 executor 可以并行，但 reviewer/tester 不提前启动。
5. 选择正式 workflow 时，数据库存在 task、workflow_run、workflow_steps。
6. 原有 slash command 和显式 @ 不回归。
7. 用户能从 UI 上区分当前执行属于普通消息、轻量协作还是正式 workflow。

# 群聊 `@` 从「提及智能体」改为「文件引用模糊搜索」设计

- 日期：2026-06-01
- 范围：前端群聊输入框 + 后端消息派发 + 设置（排除目录 / 路由配置）
- 目标房间：`http://localhost:5173/projects/ZwgdJRslFpih/rooms/vcHtzxxovkbl`

## 1. 背景与现状

当前群聊 `@` 的核心作用是**消息路由**：

- 前端：`RichMessageComposer` 输入 `@` → `RichMessageComposer.triggers.ts` 同步搜索房间内 `RoomAgent` → 选中插入 chip → 发送时收集 `mentions: room_agent.id[]`。
- 后端：`mentions.ts#resolveMentionedAgentRoomIds` 合并「显式 mentions」与「正文 `@xxx` 文本解析」→ `dispatcher.ts#dispatchUserMessage` 据此决定哪个智能体回复（`mentions_only` / `fallback_reply`）。

现有可复用能力：

- 项目文件库：`GET /projects/:projectId/files`（`fileRepo.listByProject`，支持 `q` 模糊搜索），返回带 `id` 的 `ProjectFile`。
- 工作区磁盘文件：`GET /projects/:projectId/workspace/search`（`workspace-files.ts#searchWorkspaceFiles`，已硬编码忽略 `node_modules/.git/dist/build/.next/.turbo/coverage`），仅返回路径。
- 附件链路：`fileIds`（项目库文件）/ `files`（本地上传 `File[]`）→ `api.sendMessage` → `POST /rooms/:roomId/messages`。
- 路由设置默认值已是 `message_routing_mode = fallback_reply`、`fallback_agent_id = planner`（`repos/settings.ts` DEFAULT_SETTINGS）。

## 2. 目标行为

1. 群聊输入 `@` → 弹出**文件模糊搜索**下拉，替代原智能体列表。
2. 候选两路合并：**项目文件库**（`ProjectFile`，有 `id`）+ **工作区磁盘文件**（仅路径）。
3. 选中 → 输入框插入 `@文件名` chip（复用现有 chip 机制）。
4. 发送时分流：项目库文件 → `fileIds`；磁盘文件 → 新增 `fileRefs`（路径引用）。
5. 不再发送 `mentions`；消息统一落到兜底智能体（planner）回复。
6. 排除目录可在**系统设置 + 项目设置**两层配置，**追加**到默认黑名单。

## 3. 详细设计

### 3.1 前端：`@` 触发器改为文件搜索

文件：`packages/frontend/src/components/RichMessageComposer.triggers.ts`

- `@` 触发器 `onSearch(query)` 从「内存搜 agents」改为**异步搜文件**：
  - 并行请求：`api.listProjectFiles(projectId, { q })` + `api.searchWorkspaceFiles(projectId, query)`。
  - 防抖（如 200ms）、结果去重、合并排序（名称匹配优先）、限量（如 8 条）。
- 候选项区分两类来源，写入 chip data 供发送时分流。

⚠️ **关键风险（计划阶段第一步必须先验证）**：现有 trigger dropdown 的 `onSearch` 当前是**同步**返回 `TriggerSuggestion[]`。文件搜索是网络请求，需先确认/改造 trigger 系统支持 `Promise<TriggerSuggestion[]>` + 防抖 + loading 态。若改造成本高，需回到设计调整。

### 3.2 前端：chip 数据结构与发送分流

文件：`packages/frontend/src/components/prompt-area/types.ts`、`RichMessageComposer.tsx`

- chip（`ChipSegment`，trigger 仍为 `@`）携带：
  - `kind: 'project' | 'workspace'`
  - `value`：项目库为 `fileId`，工作区为相对 `path`
  - `displayText`：`@文件名`
  - `data`：文件元数据
- `handleSubmit`：
  - 移除 `mentions` 提取逻辑（`getChipsByTrigger(@).map(value)` 与 `findTypedMentionRoomAgentIds`）。
  - `kind === 'project'` 的 chip → 并入 `fileIds`。
  - `kind === 'workspace'` 的 chip → 收集为 `fileRefs: { path }[]`。

### 3.3 前端：API 与类型

文件：`packages/frontend/src/lib/api.ts`

- `sendMessage` 入参与 `RichMessageComposerProps.onSend` 入参新增 `fileRefs?: { path: string }[]`。
- JSON 与 FormData 两条发送路径都序列化 `fileRefs`（FormData 用 `JSON.stringify`）。
- 停止填充 `mentions`（字段保留以兼容类型，但不再赋值）。
- 新增 `searchWorkspaceFiles(projectId, query)` 封装（若尚未存在），对应 `GET /workspace/search`。

### 3.4 后端：消息入口接收 `fileRefs`

文件：`packages/backend/src/routes.ts`（`POST /rooms/:roomId/messages` 的 JSON 与 multipart 两个处理函数）

- 解析 `fileRefs`（字符串路径数组）。
- 用 `workspace-files.ts` 现有路径安全校验，确保每个 path 落在项目 workspace 内、不越界（拒绝 `..`、绝对路径逃逸）。
- 持久化到 message metadata，如 `metadata.file_refs: string[]`（仅存合法路径）。

### 3.5 后端：派发时读盘注入

文件：`packages/backend/src/dispatcher.ts`

- 构建 prompt / 附件时，读取 message metadata 中的 `file_refs`：
  - 逐个用 `readWorkspaceFilePreview`（带大小上限）读取内容。
  - 文本文件：作为附件上下文注入 prompt（标注文件名/路径）。
  - 图片文件：解析为磁盘绝对路径，并入 `imagePaths`。
- `fileIds`（项目库文件）继续走现有附件链路，不变。

### 3.6 后端：群聊路径停用正文提及解析

文件：`packages/backend/src/routes.ts#dispatchUserMessageFromRoute`、`dispatcher.ts`

- 群聊派发不再用正文 `@xxx` 文本解析智能体（避免 `@文件名` 误命中同名 agent）。
- 实现：调用 `resolveMentionedAgentRoomIds` 时不传/忽略正文解析分支，`explicitRoomAgentIds` 为空 → `resolveInitialTargets` 自然落到 `fallback_reply` → planner。
- `mentions.ts` 模块保留（其他路径或测试可能引用），仅在群聊派发链路停用正文解析。

### 3.7 设置：可配置排除目录

文件：`packages/backend/src/types.ts`、`repos/settings.ts`、`workspace-files.ts`、`routes.ts`；前端 `SettingsDialogs.tsx`、`lib/api.ts`、`lib/types.ts`

- 新增设置字段 `workspace_excluded_dirs: string[]`，可在**系统层 + 项目层**配置，默认 `[]`。
- 有效忽略目录 = 默认黑名单 ∪ 系统层配置 ∪ 项目层配置（**追加 / 并集**）。
- `searchWorkspaceFiles` / `listWorkspaceDirectory` 接收「合并后的忽略列表」参数；对应 workspace 路由处理时解析有效设置并传入。
- 前端：`SystemSettingsDialog` 与 `ProjectSettingsDialog` 各加一个目录标签编辑器（增删 tag），通过既有 `updateSystemSettings` / `updateProjectSettings` 提交。

### 3.8 设置：路由配置数据变更（不动 UI）

- 将当前项目 `ZwgdJRslFpih` / 房间 `vcHtzxxovkbl` 的设置值确保为 `message_routing_mode = fallback_reply`、`fallback_agent_id = planner`。
- 通过既有设置 API/路由对话框完成，不改路由枚举与路由 UI。

## 4. 数据流（改造后）

```
输入 @ → 异步搜索(项目库 + 工作区) → 选中插入 @文件名 chip
   ↓ 发送
onSend({ content, fileIds[], fileRefs[] })   // 不再有 mentions
   ↓
POST /rooms/:roomId/messages  → 校验 fileRefs 路径安全 → 存 metadata.file_refs
   ↓
dispatchUserMessage  → 显式 mentions 为空 → fallback_reply → planner
   ↓
读 fileIds(附件链路) + 读盘 file_refs(注入 prompt / imagePaths)
   ↓
planner 智能体回复
```

## 5. 不做（YAGNI）

- 不改路由模式枚举与路由 UI（`mentions_only` 保留但群聊无触发途径）。
- 不删 `mentions.ts` 模块，仅在群聊派发链路停用正文解析。
- 工作区磁盘文件不落库为 `ProjectFile`（采用路径引用方案 B）。

## 6. 风险与验证

- **R1 异步触发器**：trigger dropdown 是否支持异步 `onSearch`。计划阶段第一步验证；不支持则需先做最小改造。
- **R2 路径安全**：`fileRefs` 必须严格限制在项目 workspace 内，复用 `workspace-files.ts` 校验，补单测覆盖越界路径。
- **R3 大文件**：`file_refs` 读盘需大小上限（复用 `readWorkspaceFilePreview` 的 512KB 限制），避免 prompt 爆量。
- **R4 误命中智能体**：确认群聊派发不再正文解析提及，补测试：正文含 `@planner` 文本不应改变路由（统一兜底）。
- **验证方式**：前端定向交互验证（输入 `@` 搜索 → 选中 → 发送）+ 后端单测（路径安全、排除目录并集、派发兜底）+ 端到端在目标房间冒烟。

# 组合式主题体系设计

## 背景

当前前端主题是平铺模型：`light`、`dark`、`console`、`minimal`。其中 `light` 是现有苹果玻璃风亮色视觉，`minimal` 是新增极简亮色视觉，`dark` 是单一暗色主题，`console` 只有入口和类型意义，缺少独立视觉体系。

用户希望去掉控制台主题，将当前浅色重命名为“苹果风”，将极简重命名为“极简风”，并为这两个亮色主题分别重新适配暗色主题。当前 `light` 的视觉必须保持，不应误改。

## 目标

将主题系统改为“风格 × 明暗”组合模型：

- 风格：`苹果风`、`极简风`
- 明暗：`亮色`、`暗色`

最终可切换的组合为：

- 苹果风 + 亮色
- 苹果风 + 暗色
- 极简风 + 亮色
- 极简风 + 暗色

## 非目标

- 不改变页面布局、DOM 结构、路由、数据流或业务交互。
- 不重做当前 `light` 视觉；当前 `light` 必须作为“苹果风 + 亮色”原样继承。
- 不保留 `控制台` 主题入口。
- 不引入新的设计系统库或主题运行时依赖。
- 不修改任务、消息、Agent、工作流等业务模型。

## 主题模型

建议使用组合式类型：

```ts
export type ThemeStyle = 'apple' | 'minimal';
export type ThemeTone = 'light' | 'dark';
export type ThemeMode = `${ThemeStyle}-${ThemeTone}`;
```

辅助函数建议包括：

- `parseThemeMode(value)`：读取 localStorage 并兼容旧值。
- `getThemeStyle(theme)`：从组合值读取风格。
- `getThemeTone(theme)`：从组合值读取明暗。
- `createThemeMode(style, tone)`：组合风格与明暗。

## 旧值迁移

localStorage key 继续使用 `openclaw-room-theme`，但值迁移为组合式主题。

迁移规则：

- `light` -> `apple-light`
- `dark` -> `apple-dark`
- `minimal` -> `minimal-light`
- `console` -> `minimal-light`
- `apple-light`、`apple-dark`、`minimal-light`、`minimal-dark` 保持原值
- 未知值或空值 -> `apple-light`

迁移后应把组合值写回 localStorage，避免反复走兼容分支。

## UI 入口

侧栏底部主题区改为两个 segmented controls：

- 风格控件：`苹果风` / `极简风`
- 明暗控件：`亮色` / `暗色`

交互规则：

- 切换风格时保留当前明暗。
- 切换明暗时保留当前风格。
- `aria-pressed` 应分别反映当前风格和当前明暗。
- 图标可复用现有 lucide 图标，但不应依赖图标表达唯一含义，文字必须可见。

## CSS 主题结构

### 苹果风亮色

`apple-light` 等价于当前 `light`。

实现要求：

- `:root` 继续保留当前苹果亮色 token，防止初始渲染闪动。
- `[data-theme="apple-light"]` 可以不重复定义 token，或只定义少量显式别名。
- 不调整当前 `:root` 的颜色、背景、阴影、玻璃拟态规则。

### 苹果风暗色

`apple-dark` 基于当前 `dark`，但应保持苹果风的玻璃拟态气质：

- 深色背景
- 半透明 surface
- 柔和边缘
- 适度 blur / shadow
- 保留玻璃式卡片与按钮层次

它不是极简暗色，不能硬边化。

### 极简风亮色

`minimal-light` 继承当前 `minimal` 的浅色硬边细线风格：

- 近白背景
- 纯白面板
- 细线边框
- 低圆角
- 无玻璃模糊
- 无大面积渐变和大阴影

### 极简风暗色

`minimal-dark` 是极简风的暗色版本：

- 深灰黑背景
- 深色纯色面板
- 低对比但清晰的 1px 细线
- 低圆角
- 无玻璃模糊
- 无大阴影和发光
- active 状态仍使用细色条或边框强调

## Scoped 覆盖策略

当前极简覆盖规则从：

```css
[data-theme="minimal"] .selector { ... }
```

迁移为：

```css
[data-theme="minimal-light"] .selector,
[data-theme="minimal-dark"] .selector { ... }
```

颜色 token 分别由 `[data-theme="minimal-light"]` 和 `[data-theme="minimal-dark"]` 提供。

苹果风覆盖规则应尽量少。`apple-light` 主要依赖 `:root`；`apple-dark` 主要通过 token 与现有玻璃类自然生效，只有必要时补充 scoped 修正。

## 验证方式

最低验证：

- `npm run build`

浏览器验证：

- 旧 localStorage 值 `light` 迁移为 `apple-light`
- 旧 localStorage 值 `dark` 迁移为 `apple-dark`
- 旧 localStorage 值 `minimal` 迁移为 `minimal-light`
- 旧 localStorage 值 `console` 迁移为 `minimal-light`
- 风格切换保留当前明暗
- 明暗切换保留当前风格
- 四个组合均能切换
- 苹果风亮色与当前 `light` 视觉一致
- 极简风亮色保持当前极简视觉
- 苹果暗色与极简暗色有明显风格差异

## 风险

最大风险是误改当前 `light` 视觉。实现时应把当前 `:root` 视为苹果亮色基线，只做命名和组合入口迁移，不重写其 token。

第二个风险是旧主题值迁移错误导致用户已有偏好丢失。应集中在主题 helper 中处理迁移，不把兼容逻辑散落到组件里。

第三个风险是极简暗色和苹果暗色视觉趋同。实现时应确保苹果暗色保留玻璃层次，极简暗色保持硬边无玻璃。

## 批准结论

用户已批准采用组合式主题模型。当前 `light` 视觉必须保持，作为“苹果风 + 亮色”的基准。

# 全局智能体库与对话式创建设计

## 背景

当前系统的 agent 能力主要绑定在聊天室维度：用户可以在聊天室中添加 `room_agents`，并配置 ACP 后端、会话、权限和工作流角色。这个模型适合临时邀请，但不适合维护长期复用的智能体身份。

本次目标是完善“智能体”页面，提供完整的全局智能体增删改查，并支持通过对话式问答生成智能体草稿。智能体本身独立于项目和聊天室，聊天室只负责从全局智能体库拉入智能体。

## 目标

1. 左侧侧边栏入口从 `Agent` 改为 `智能体`。
2. 新增独立的全局智能体配置页面。
3. 全局智能体支持完整 CRUD。
4. 每个智能体可独立设置：
   - 用户称呼
   - 智能体名称
   - 智能体 ID
   - 性格
   - 行为规则
   - 主要工作
   - 默认 ACP 后端与默认权限
5. 聊天室可以从全局智能体库拉入智能体。
6. 聊天室移除智能体时，只移除聊天室成员关系，不删除全局智能体。
7. 删除全局智能体时，如果已被聊天室引用，则禁止删除并提示引用位置。
8. 支持对话式创建智能体：先收集信息并生成草稿，用户确认后才入库。

## 非目标

1. 不在本阶段实现智能体版本历史。
2. 不在本阶段支持同一智能体的多版本锁定。
3. 不在本阶段做跨设备或云端同步。
4. 不在本阶段重构 ACP provider 执行链。
5. 不把聊天室级 session、workflow role、权限全部提升为全局唯一配置。

## 数据模型

### 全局智能体表

新增 `agents` 表，作为全局智能体库。

字段建议：

- `id`：主键
- `agent_id`：稳定标识，用于 @ 提及、兜底路由和展示
- `name`：显示名称
- `description`：简介
- `preferred_user_name`：智能体称呼用户的方式
- `personality`：性格设定
- `rules`：行为规则
- `responsibilities`：主要工作
- `default_acp_backend`：默认 ACP 后端
- `default_acp_permission_mode`：默认 ACP 权限
- `created_at`
- `updated_at`

`agent_id` 应全局唯一，避免同一个聊天室拉入多个同名智能体后 @ 提及和路由歧义。

### 聊天室智能体关系

现有 `room_agents` 表继续承担聊天室成员关系，并新增 `global_agent_id` 字段引用 `agents.id`。

`room_agents` 保留聊天室级覆盖配置：

- `workflow_role`
- `acp_enabled`
- `acp_backend`
- `acp_session_id`
- `acp_session_label`
- `acp_permission_mode`
- `acp_writable_dirs`
- `capabilities`
- `default_runtime`
- `memory_max_context_chars`

显示和执行时，后端查询应将 `agents` 与 `room_agents` 合并为现有前端可消费的 `RoomAgent` 形态。全局字段作为默认身份，聊天室字段作为当前聊天室覆盖。

## 数据迁移

升级时需要迁移现有 `room_agents`：

1. 为每条没有 `global_agent_id` 的 `room_agents` 创建或复用全局 `agents` 记录。
2. 优先按 `agent_id` 查找已存在的全局智能体。
3. 不存在时，使用当前 `room_agents.agent_id`、`agent_name`、`agent_role` 创建全局智能体。
4. 将新建或复用的 `agents.id` 回填到 `room_agents.global_agent_id`。
5. 保留现有聊天室级 ACP/session/workflow 配置，避免破坏已有执行链。

## API 设计

### 全局智能体 CRUD

- `GET /agents`：列出全局智能体，包含引用数量。
- `POST /agents`：创建全局智能体。
- `GET /agents/:agentId`：读取单个全局智能体。
- `PATCH /agents/:agentId`：更新全局智能体。
- `DELETE /agents/:agentId`：删除全局智能体。

删除规则：

- 未被任何聊天室引用时允许删除。
- 已被引用时返回 `409`，响应中包含引用的聊天室信息。

### 对话式创建

第一阶段可在前端本地实现规则化问答和草稿生成，不强依赖外部模型。

- `POST /agents/draft-from-conversation` 可作为后续扩展接口。
- 初版可以在前端收集问答后生成结构化草稿，再调用 `POST /agents` 创建。

必问问题：

1. “我应该怎么称呼你？”
2. “这个智能体叫什么？”
3. “它是什么性格？”
4. “它主要做哪些工作？”
5. “它有哪些必须遵守的规则？”
6. “默认使用哪个 ACP 后端？”

生成草稿内容：

- `agent_id`：由名称自动生成，可编辑。
- `name`：来自用户回答。
- `preferred_user_name`：来自用户回答。
- `personality`：归纳性格回答。
- `responsibilities`：归纳主要工作回答。
- `rules`：归纳规则回答。
- `default_acp_backend`：来自用户选择或默认 `codex`。
- `default_acp_permission_mode`：默认 `bypass`，可编辑。

### 聊天室拉入智能体

- `POST /rooms/:roomId/agents`：从全局智能体库拉入智能体。
- 请求体使用 `global_agent_id`。
- 后端从全局智能体读取默认 ACP 配置，创建 `room_agents` 关系。
- 同一聊天室中同一个全局智能体不可重复拉入。

保留现有模板添加能力可以作为兼容层：模板先创建或复用全局智能体，再拉入聊天室。

## 前端页面设计

### 导航

左侧侧边栏：

- `Agent` 改为 `智能体`。
- 链接到 `/agents`。

全局智能体页面不依赖当前项目和聊天室。

### 页面结构

页面采用独立管理视图：

- 左侧：智能体列表
  - 搜索
  - 新建智能体
  - 对话式创建
  - 引用数量
- 中间：详情编辑
  - 基础信息
  - 性格
  - 规则
  - 主要工作
  - 默认 ACP 配置
  - 使用状态
  - 保存、删除
- 右侧或弹窗：对话式创建流程
  - 问答消息流
  - 当前步骤
  - 生成草稿
  - 编辑草稿
  - 确认创建

### 聊天室邀请入口

聊天室中的“邀请 Agent”改为“邀请智能体”或“拉入智能体”。

行为变化：

- 不再默认手写孤立 room agent。
- 打开后展示全局智能体库。
- 可搜索智能体。
- 选择后拉入当前聊天室。
- 已在当前聊天室的智能体置灰或显示“已加入”。

## 提示词与执行上下文

后端在构建 ACP prompt 时，应将全局智能体设定注入执行上下文。

建议结构：

```text
你的智能体身份：
- 名称：{name}
- 用户称呼：{preferred_user_name}
- 性格：{personality}
- 主要工作：{responsibilities}
- 必须遵守的规则：
{rules}
```

注入位置应靠近系统级约束，优先级低于当前会话用户明确要求和项目 `AGENTS.md`，但高于普通聊天历史。

## 错误处理

1. 创建智能体时 `agent_id` 重复：返回 `400` 或 `409`，前端提示用户修改。
2. 删除被引用的全局智能体：返回 `409`，展示引用聊天室。
3. 拉入不存在的智能体：返回 `404`。
4. 重复拉入同一智能体：返回 `409` 或直接返回已有 room agent。
5. 对话式创建信息不足：不允许生成最终创建请求，继续追问缺失字段。

## 测试与验证

最低验证：

1. `npm run build`
2. 后端仓储测试：
   - 创建全局智能体
   - 更新性格和规则
   - 被聊天室引用时禁止删除
   - 未引用时允许删除
   - 聊天室拉入全局智能体
3. 前端类型构建：
   - `/agents` 页面可编译
   - 聊天室邀请入口改为从全局智能体选择
4. 手动验收：
   - 新建智能体
   - 编辑性格和规则
   - 通过对话式流程生成草稿并创建
   - 在聊天室拉入智能体
   - 已引用智能体删除失败且提示引用位置

## 待确认事项

本设计已确认：

- 智能体是全局资源，不属于项目或聊天室。
- 聊天室可以拉入全局智能体。
- 删除被引用的全局智能体时禁止删除。
- 新建智能体支持对话式问答，先生成草稿，再确认创建。

实施计划阶段需要进一步明确：

- 对话式创建初版是否完全前端规则化，还是直接接入系统模型生成草稿。
- 全局智能体页面是否需要在第一版支持批量操作。
- 现有内置 ACP 模板是否在迁移后全部预置为全局智能体。

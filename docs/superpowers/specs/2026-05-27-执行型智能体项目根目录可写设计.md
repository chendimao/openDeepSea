# 执行型智能体项目根目录可写设计

## 背景

当前内置群聊智能体已经有运行边界配置：

- 全局智能体默认值保存在 `agents.default_workspace_policy`。
- 群聊智能体实例运行边界保存在 `room_agents.workspace_policy`。
- ACP 运行时通过 `resolveAgentRuntimeProfile` 将相对路径解析到当前项目根目录内。

现状中部分执行型智能体的默认可写目录过窄。例如后端执行者只能写 `packages/backend`，前端执行者只能写 `packages/frontend`，技术写作者只能写 `docs` 和 `.git`。这会让群聊中的执行型智能体在处理跨包改动、根配置、文档、脚本或提交时被默认边界卡住。

## 目标

1. 让内置群聊智能体中 `workflow_role` 为 `executor` 的角色默认可以写当前项目根目录。
2. 保留规划、审查、验收、分析和高风险建议角色的只读默认边界。
3. 让已有内置全局智能体和已有房间内置智能体随 runtime profile 版本升级迁移到新默认。
4. 不覆盖用户手动深度自定义过的非内置智能体配置。

## 非目标

- 不让所有智能体都获得写权限。
- 不改变非 `executor` 角色的只读职责边界。
- 不放宽项目根目录之外的路径校验。
- 不调整 ACP provider 的命令参数语义。
- 不改变用户通过 UI 手动设置的普通自定义智能体边界。

## 角色边界

以下内置模板改为默认 `workspace-write`，并设置 `workspace_policy.write` 为 `['.']`：

- `backend-executor`
- `frontend-executor`
- `computer-assistant`
- `devops-engineer`
- `technical-writer`
- `sales-assistant`

以下内置模板继续只读，`workspace_policy.write` 保持 `[]`：

- `planner`
- `ui-designer`
- `data-analyst`
- `product-manager`
- `qa-tester`
- `security-reviewer`
- `reviewer`
- `acceptor`
- `accounting-advisor`
- `legal-assistant`
- `medical-assistant`
- `marketing-strategist`

## 实现设计

### 模板默认值

更新 `packages/backend/src/agent-templates.ts` 中内置模板：

- 对所有 `workflow_role: 'executor'` 的模板设置 `acp_permission_mode: 'workspace-write'`。
- 对所有 `workflow_role: 'executor'` 的模板设置 `workspace_policy: { read: ['.'], write: ['.'] }`。
- 保留各角色原有 `tool_policy` 能力，不因为目录变更额外增加浏览器、图片或提交能力。

这样可写范围由 `workspace_policy.write` 控制，工具能力仍由 `tool_policy.allowed` 控制。

### 版本迁移

将内置 runtime profile version 从 `3` 升到 `4`：

- `packages/backend/src/repos/agents.ts`
- `packages/backend/src/repos/rooms.ts`

全局智能体列表读取时会执行内置智能体种子同步，房间智能体列表读取时会执行房间内置运行边界迁移。版本升级后，旧版本内置智能体会重新套用源码模板中的运行边界。

对于用户已手动深度自定义过、且不符合旧内置默认边界的配置，现有迁移逻辑不会强行覆盖。

### 路径安全

保持现有 `resolveProjectPath` 规则：

- `'.'` 会解析为当前项目根目录。
- 绝对路径必须仍位于项目根目录下。
- 包含 `..` 的路径继续被拒绝。
- 项目根目录之外的读写不会被允许。

## 数据流

### 新内置智能体加入房间

1. 用户从模板创建或拉入内置智能体。
2. 后端读取内置模板。
3. 如果模板角色是 `executor`，写入 `workspace_policy.write: ['.']`。
4. ACP runtime 解析后获得当前项目根目录可写边界。

### 已有内置智能体迁移

1. 用户打开或刷新群聊。
2. `roomAgentRepo.listByRoom` 触发内置 room agent runtime profile 迁移。
3. 旧版本内置执行型 room agent 更新到版本 `4`。
4. 后续运行时使用项目根目录写权限。

## 错误处理

- 如果项目路径不存在或 ACP provider 无法启动，沿用现有运行失败处理。
- 如果 `workspace_policy` 中出现非法路径，沿用现有 warning 和忽略非法路径逻辑。
- 如果执行型智能体缺少写工具能力，即使可写目录是根目录，也不能执行写文件工具；本次不自动增加工具能力。

## 测试策略

后端测试：

- `agent-templates.test.ts` 断言所有 executor 模板默认 `workspace-write` 且 `write: ['.']`。
- 同一测试断言非 executor 模板仍为只读或至少 `write: []`。
- `repos/agents.test.ts` 更新内置全局智能体默认边界断言。
- `repos/rooms` 相关测试更新房间内置智能体迁移断言。
- 如有旧版本迁移用例，补充 version `3` 到 `4` 的 executor 根目录可写验证。

最低验证：

- 运行相关后端单测。
- 运行 `npm run build`。

## 验收标准

1. 新建或重新同步的内置 executor 智能体默认可写项目根目录。
2. 非 executor 内置智能体仍不能默认写文件。
3. 运行时提示中的可写目录显示为当前项目根目录。
4. 项目根目录外路径仍不会被允许。
5. 构建和相关测试通过。
